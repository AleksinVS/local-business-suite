# ADR-0003: Архитектура динамического ИИ-дашборда

## Статус
Черновик (Draft)

## Контекст
Необходимо реализовать модульный дашборд на главной странице, который пользователь может настраивать под свои нужды. Настройку осуществляет ИИ-агент (малая LLM), формируя структуру данных и визуализацию на основе текстовых запросов пользователя. Система должна быть расширяемой для будущих модулей без изменения ядра.

## Предложенная архитектура

### 1. Трехуровневая модель данных
1.  **Слой агрегатов (Performance Layer):** Использование Materialized Views или пред-рассчитанных таблиц для популярных метрик (количество заявок, среднее время, остатки).
2.  **Слой сырых данных (Flexibility Layer):** Прямой доступ к Django-моделям через унифицированный интерфейс для специфических или новых запросов, которых нет в агрегатах.
3.  **Слой метаданных (Semantic Layer):** Реестр (Registry), описывающий доступные сущности, их поля (Dimensions) и возможные операции над ними (Metrics).

### 2. Гибкий генератор запросов
ИИ-агент не пишет SQL или Python-код напрямую. Вместо этого он генерирует абстрактный JSON-объект (Query Schema), описывающий:
- Источник (Source)
- Фильтры (Filters)
- Группировки (Dimensions)
- Агрегации (Metrics)
- Тип визуализации (Chart Type)

### 3. Компоненты системы
- **Metadata Registry:** Центральный каталог, где каждый модуль (workorders, inventory и т.д.) регистрирует свои возможности для аналитики.
- **AI Agent Controller:** Малая LLM, интерпретирующая запрос пользователя и сопоставляющая его с Metadata Registry.
- **Query Executor:** Бэкенд-сервис на Django, который:
    - Валидирует JSON-запрос от ИИ.
    - Выбирает оптимальный источник (агрегат или сырая таблица).
    - Применяет права доступа (Row-Level Security).
    - Возвращает данные для фронтенда.
- **Dynamic Frontend Renderer:** Набор UI-компонентов (Charts, Stats, Tables), которые инициализируются полученными данными.

### 4. Расширяемость (Перспективные модули)
- Новые модули просто добавляют запись в Metadata Registry.
- Если поле отсутствует в агрегатах, Executor автоматически переключается на тяжелый запрос к сырым данным с использованием механизмов кэширования.
- Пользователь может вводить "Виртуальные измерения" (Calculated Dimensions), которые вычисляются на уровне базы через CASE/EXTRACT.

## Последствия
- **Плюсы:** Высокая гибкость, минимальный риск галлюцинаций LLM за счет работы со схемой, а не кодом, высокая производительность для типовых запросов.
- **Риски:** Нагрузка на БД при "сырых" запросах (требуется Query Timeout), сложность первичного описания метаданных для всех модулей.

## Дальнейшие шаги
1. Разработать спецификацию JSON-схемы для Metadata Registry.
2. Создать прототип Query Executor для модуля `workorders`.
3. Описать системный промпт для малой LLM, обучающий её работе с данной схемой.
