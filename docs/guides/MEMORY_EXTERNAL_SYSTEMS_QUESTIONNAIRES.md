# Опросники: сбор знаний из внешних информационных систем

## Назначение

Эти опросники нужны перед подключением внешней информационной системы к памяти. Они написаны для бизнес-заказчика и владельца данных, без технических терминов там, где это возможно.

Первый опросник помогает понять, **какие данные забирать из внешней системы и как часто**. Второй помогает собрать начальные сведения о **типах сущностей и связей для графа знаний**.

Ответы не являются техническим заданием сами по себе. По ним архитектор памяти готовит source contract, план доступа, график синхронизации, правила хранения и задачи на реализацию коннектора.

## Опросник 1: подключение внешней информационной системы

### 1. Зачем подключаем систему

1. Как называется система?
2. Кто бизнес-владелец этой системы?
3. Кто технический владелец или администратор?
4. Для каких рабочих задач ИИ-боту нужны знания из этой системы?
5. Какие вопросы пользователи должны задавать боту после подключения?
6. Какие решения или действия должны стать быстрее после подключения?
7. Что будет считаться успешным результатом первого этапа?

### 2. Какие данные нужны

1. Какие разделы системы важны для памяти?
2. Какие записи точно нужно забирать?
3. Какие записи точно нельзя забирать?
4. Нужны ли только активные записи или также архивные?
5. Нужны ли комментарии, история изменений, вложения, файлы?
6. Нужны ли справочники: статусы, типы, категории, подразделения, сотрудники, оборудование?
7. Нужны ли связи между объектами: заявка связана с оборудованием, пациентом, отделением, договором?
8. Какие поля являются ключевыми для поиска и объяснения ответа пользователю?
9. Какие поля нужны только для фильтрации, но не должны показываться в ответах?

### 3. Насколько свежими должны быть знания

1. Как часто данные реально меняются?
2. Как быстро ИИ-бот должен узнавать об изменениях?
3. Что допустимо для первого этапа: раз в сутки, раз в несколько часов, каждый час, почти сразу?
4. Есть ли периоды, когда систему лучше не нагружать?
5. Можно ли выполнять тяжелую синхронизацию ночью?
6. Нужна ли отдельная быстрая синхронизация для критичных данных?
7. Что страшнее: устаревший ответ или дополнительная нагрузка на внешнюю систему?

### 4. Объем и нагрузка

1. Сколько примерно записей в нужных разделах?
2. Сколько новых или измененных записей появляется за день?
3. Есть ли очень большие записи или вложения?
4. Есть ли ограничения API или выгрузки?
5. Есть ли официальный способ получить только изменения, а не выгружать все заново?
6. Что делать, если система временно недоступна?
7. Кто должен получать уведомление о сбоях синхронизации?

### 5. Доступ и права

1. Кто имеет право читать эти данные в исходной системе?
2. Должен ли ИИ-бот соблюдать такие же ограничения доступа?
3. Есть ли группы пользователей, которым часть данных недоступна?
4. Есть ли данные только для руководителей, администраторов или отдельных подразделений?
5. Как определить, к какому подразделению или роли относится запись?
6. Кто имеет право разрешить подключение этих данных к памяти?
7. Кто будет владельцем знаний после загрузки в память?

### 6. Чувствительные данные

1. Есть ли персональные данные?
2. Есть ли медицинские, финансовые, договорные или кадровые данные?
3. Есть ли пароли, токены, ключи доступа, строки подключения?
4. Есть ли коммерческая тайна или внутренние служебные данные?
5. Какие поля нельзя показывать пользователям в ответах ИИ-бота?
6. Какие поля нельзя сохранять в память вообще?
7. Какие данные можно сохранять только после обезличивания?
8. Нужно ли согласование с ответственным за безопасность или персональные данные?

### 7. Хранение выгрузок

1. Нужно ли хранить исходную выгрузку из внешней системы после обработки?
2. Если да, зачем: отладка, аудит, повторная обработка, юридические требования?
3. Какой срок хранения допустим: не хранить, 7 дней, 30 дней, 90 дней, другой срок?
4. Нужно ли удалять выгрузки сразу после успешного извлечения знаний?
5. Кто имеет право смотреть выгруженные данные?
6. Нужно ли шифрование выгрузок?
7. Как поступать с выгрузками, где найдены секреты или запрещенные данные?

### 8. Проверка качества

1. Кто сможет проверить, что бот правильно понял данные?
2. Какие примеры вопросов и правильных ответов нужно использовать для проверки?
3. Какие ошибки являются критичными?
4. Нужно ли показывать ссылку на исходную запись в ответе?
5. Как пользователь должен сообщать, что знание неверное или устарело?
6. Как часто нужно пересматривать качество знаний из этой системы?

### 9. Первый безопасный этап

1. С какой небольшой части системы лучше начать?
2. Какие данные можно взять для теста без риска?
3. Кто подтвердит, что тестовая выборка безопасна?
4. Можно ли сначала подключить только справочники или обезличенные данные?
5. Когда можно перейти от теста к регулярной синхронизации?

## Опросник 2: начальные типы сущностей графа знаний

### 1. Рабочая область

1. Какой процесс или подразделение описываем первым?
2. Какие основные рабочие задачи выполняются в этой области?
3. Какие документы, заявки, журналы или системы отражают этот процесс?
4. Какие вопросы руководитель или специалист чаще всего задает по этой области?
5. Какие ответы должны быть не просто текстом, а связями между объектами?

### 2. Основные сущности

1. Какие главные объекты существуют в этой области?
2. Как эти объекты называют сотрудники в обычной речи?
3. Есть ли у объекта официальный номер, код или инвентарный идентификатор?
4. Какие объекты являются людьми, подразделениями, оборудованием, документами, событиями, задачами или местами?
5. Какие объекты нужно отличать друг от друга даже при похожих названиях?
6. Какие объекты не нужно делать отдельными сущностями, потому что это просто свойства других объектов?

Примеры формулировок:

- "Оборудование" - отдельная сущность.
- "Заявка на ремонт" - отдельная сущность.
- "Статус заявки" - скорее свойство заявки, если по статусу нет отдельной истории и связей.
- "Отделение" - отдельная сущность, если по нему строится доступ, аналитика или связи.

### 3. Свойства сущностей

Для каждого важного типа объекта:

1. Как он называется?
2. Какое у него понятное описание?
3. Какие поля помогают его узнать?
4. Какие поля обязательны?
5. Какие поля часто меняются?
6. Какие поля чувствительные и не должны попадать в обычные ответы?
7. Какие синонимы, сокращения или старые названия встречаются?
8. Какие значения считаются ошибочными или шумовыми?

### 4. Связи между сущностями

1. Какие объекты связаны между собой?
2. Как сотрудники описывают эту связь словами?
3. Связь постоянная или временная?
4. У связи есть дата начала, дата окончания, статус или причина?
5. Может ли один объект иметь много таких связей?
6. Нужно ли хранить историю изменений связи?
7. Какие связи особенно важны для поиска причин, ответственности или влияния?

Примеры:

- оборудование находится в отделении;
- заявка относится к оборудованию;
- заявка назначена исполнителю;
- документ утвержден сотрудником;
- договор связан с поставщиком;
- событие произошло в подразделении.

### 5. Правила узнавания и объединения

1. По каким признакам понятно, что два упоминания означают один и тот же объект?
2. Какие номера, коды или идентификаторы самые надежные?
3. Какие названия часто пишут по-разному?
4. Есть ли устаревшие названия, которые нужно сопоставлять с новыми?
5. Какие совпадения опасны и требуют проверки человеком?
6. Какие слова часто похожи на сущности, но на самом деле являются шумом?

### 6. Важные события

1. Какие события нужно запоминать?
2. Какие события меняют состояние объекта?
3. Какие события должны попадать в историю?
4. Какие события важны только короткое время?
5. Какие события должны запускать уведомление, задачу или проверку?

### 7. Доступ и чувствительность

1. Какие типы сущностей доступны всем сотрудникам?
2. Какие доступны только подразделению?
3. Какие доступны только руководителям или отдельным ролям?
4. Какие типы сущностей содержат персональные или медицинские данные?
5. Какие связи могут раскрывать чувствительную информацию?
6. Какие типы нельзя использовать в тестах с внешними cloud-сервисами?

### 8. Примеры для проверки

1. Дайте 5-10 реальных или обезличенных примеров объектов.
2. Дайте 5-10 примеров связей между объектами.
3. Дайте 5-10 вопросов, на которые граф должен отвечать.
4. Дайте примеры неправильных или спорных упоминаний.
5. Кто из профильных экспертов сможет подтвердить правильность типов и связей?

## Как использовать ответы

1. Архитектор памяти переводит ответы в draft source profile и graph schema proposal.
2. Технический владелец проверяет доступность API, ограничения, объемы и способ синхронизации.
3. Владелец данных подтверждает состав данных, права доступа и retention.
4. Ответственный за безопасность проверяет чувствительные поля и запреты.
5. Профильный эксперт подтверждает типы сущностей и связей.
6. Владелец графа знаний принимает итоговую схему или отправляет ее на доработку.

## Зафиксированные MVP-ответы

- Raw API responses можно хранить в short-lived quarantine, если это явно включено для конкретного источника.
- Права внешней системы на первом этапе маппятся в `scope_tokens` портала вручную при внедрении источника.
- Очередь коннектора нужна сразу как отдельный queue backend; текущий MVP использует standalone SQLite queue under `data/memory/queues/`.
