# ADR-0014: MLflow для оценки качества и трассировки системы памяти

## Статус

Accepted

## Дата

2026-05-26

## Контекст

В системе памяти уже есть собственная команда `memory_eval`. Она проверяет базовые правила проекта:

- секреты не должны попадать в память и поиск;
- персональные данные должны проходить безопасную обработку;
- права доступа через `scope_tokens` должны соблюдаться;
- ненадежные источники не должны попадать в обычный контекст агента;
- горячий путь поиска не должен требовать вызова LLM;
- архитектурные договоренности памяти должны оставаться проверяемыми локально.

Эти проверки нужны как быстрый и воспроизводимый технический контур. Они не должны зависеть от сети, внешней модели, платного API или случайной оценки LLM.

При этом для развития памяти нужны дополнительные возможности:

- измерять качество поиска по знаниям;
- сравнивать разные версии полнотекстового, векторного и графового поиска;
- оценивать, достаточно ли найденного контекста для ответа;
- проверять, что ответ агента обоснован найденными знаниями;
- хранить историю прогонов, параметры, метрики и артефакты;
- разбирать цепочку выполнения запроса: поиск, проверка прав, сбор контекста, ответ агента.

Инструменты вроде Ragas и DeepEval полезны для отдельных оценок качества, но сами по себе не дают полного контура учета экспериментов, трассировки и сравнения прогонов. MLflow закрывает этот более широкий слой и может подключать Ragas/DeepEval как отдельные оценщики.

Актуальные возможности MLflow GenAI включают:

- трассировку функций и шагов через `mlflow.trace`;
- поиск и анализ трасс через `mlflow.search_traces`;
- оценку RAG-сценариев через `mlflow.genai.evaluate`;
- встроенные RAG-оценщики `RetrievalRelevance`, `RetrievalGroundedness`, `RetrievalSufficiency`;
- подключение Ragas-оценщиков через `mlflow.genai.scorers.ragas`;
- локальный или self-hosted tracking server с SQL backend и хранилищем артефактов.

## Решение

Использовать MLflow как внешний контур оценки качества и трассировки системы памяти.

Собственный `memory_eval` остается обязательным базовым контуром и не заменяется MLflow.

Целевая схема:

```text
memory_eval
  -> быстрые обязательные проверки безопасности, прав и архитектурных правил

memory_quality_eval
  -> трассировка и оценка качества через MLflow
  -> опциональные MLflow scorers, Ragas scorers, DeepEval/custom scorers
  -> отчеты в MLflow и краткие JSON-итоги в data/memory/eval/
```

MLflow не должен участвовать в горячем пути ответа пользователю. Оценка качества запускается:

- вручную инженером;
- по расписанию в дешевое ненагруженное время;
- в CI для безопасных синтетических наборов;
- после существенных изменений поиска, индексации, промптов или модели агента.

## Границы ответственности

### `memory_eval`

`memory_eval` проверяет правила, которые специфичны для проекта:

- запрет выдачи секретов;
- безопасную обработку персональных данных;
- соблюдение `scope_tokens`;
- запрет прямой выдачи ненадежных источников;
- отсутствие обязательной LLM в горячем поиске;
- целостность архитектурных договоренностей памяти.

Эти проверки должны быть быстрыми, детерминированными и доступными без внешних сервисов.

### MLflow

MLflow отвечает за:

- трассировку выполнения проверочных запросов;
- хранение прогонов оценки;
- хранение параметров поиска, версии индексов, версии промптов и модели;
- хранение метрик качества;
- сравнение прогонов;
- просмотр ошибок и деградаций качества;
- подключение внешних оценщиков.

### Ragas и DeepEval

Ragas и DeepEval рассматриваются как подключаемые оценщики внутри MLflow, а не как основной контур хранения результатов.

Пример целевой схемы:

```text
MLflow run
  -> собственные числовые метрики поиска
  -> MLflow RAG scorers
  -> Ragas scorers
  -> DeepEval/custom agent checks
```

## Безопасность и приватность

В MLflow запрещено писать:

- секреты;
- исходные пароли, ключи, токены;
- необезличенные персональные данные;
- сырые письма, документы и API-ответы без отдельного разрешенного режима;
- данные из источников, которые не прошли правила доступа для текущего проверочного пользователя.

Допустимо писать:

- синтетические проверочные данные;
- обезличенный текст;
- `document_id`, `memory_id`, `source_code`, `source_ref`;
- оценки релевантности, ранги, счетчики и технические признаки;
- сокращенный безопасный контекст, если он прошел тот же фильтр, что и выдача агенту.

Если в трассе нужен текст найденного знания, он должен быть безопасным текстом знания из файла, прошедшим правила доступа и фильтры чувствительности. Для боевых данных трассировка текста по умолчанию выключена.

`MemoryAccessAudit` остается основным аудитом доступа в приложении. MLflow не заменяет аудит прав. При интеграции нужно связать записи через идентификаторы:

```text
MemoryAccessAudit.request_id
MLflow trace_id
MLflow run_id
```

## Целевая интеграция

### Новый командный контур

Добавить отдельную команду:

```bash
python manage.py memory_quality_eval --engine mlflow --suite smoke
```

Команда должна:

1. Читать активные `MemoryEvalCase`.
2. Запускать `memory.search` от имени проверочного пользователя.
3. Собирать найденные `document_id`, ссылки на источники, оценки ранжирования и безопасный контекст.
4. Опционально запускать ответ агента.
5. Записывать трассу и метрики в MLflow.
6. Записывать краткий JSON-итог в `data/memory/eval/`.

### Трассировка шагов

Минимальные шаги трассировки:

```text
quality_eval_case
  -> memory.search
  -> fulltext_candidates
  -> vector_candidates
  -> graph_candidates
  -> policy_filter
  -> trust_filter
  -> context_packing
  -> optional_agent_answer
  -> scorers
```

На первом этапе трассировка может быть включена только внутри `memory_quality_eval`. Постоянная трассировка боевых запросов требует отдельного решения по приватности и нагрузке.

### Метрики первого этапа

Собственные числовые метрики:

- `expected_document_recall_at_k`;
- `forbidden_document_hits`;
- `trusted_source_violations`;
- `scope_violations`;
- `latency_ms`;
- `context_item_count`;
- `context_token_estimate`;
- `graph_used`;
- `vector_used`;
- `fulltext_used`.

MLflow/RAG-метрики:

- retrieval relevance;
- retrieval groundedness;
- retrieval sufficiency.

Ragas-метрики можно подключить позже:

- faithfulness;
- context precision;
- context recall;
- response relevancy.

## Хранение MLflow

Для локального MVP:

```text
data/mlflow/mlflow.db
data/mlflow/artifacts/
```

MLflow Tracking URI:

```text
sqlite:///data/mlflow/mlflow.db
```

Для промышленной эксплуатации:

- PostgreSQL как backend store;
- отдельное хранилище артефактов;
- ограничение доступа к UI;
- регламент хранения трасс и отчетов;
- отдельные правила маскирования и удаления данных.

MLflow runtime-данные не коммитятся в основной репозиторий.

## Рассмотренные альтернативы

### Только собственный `memory_eval`

Отклонено как единственный подход. Собственные проверки нужны, но их недостаточно для анализа качества поиска, ответа агента и сравнения экспериментов.

### Использовать Ragas напрямую

Отклонено как основной контур. Ragas полезен как набор оценок качества, но не закрывает полноценное хранение прогонов, трассировку, сравнение параметров и артефактов. Ragas можно подключить через MLflow.

### Использовать DeepEval напрямую

Отклонено как основной контур. DeepEval удобен для тестов поведения агента и интеграции с `pytest`, но для этого проекта важнее единый журнал прогонов, трассировка и сравнение версий. DeepEval можно подключить как дополнительный оценщик.

### Использовать LangSmith

Отклонено для MVP. Инструмент силен для трассировки и оценки LLM-приложений, но является внешним сервисным контуром и требует отдельного решения по приватности и размещению данных.

### Использовать Phoenix/Arize или TruLens

Отклонено для MVP. Эти инструменты полезны для наблюдаемости и оценки RAG, но MLflow лучше совпадает с задачей учета экспериментов, локального запуска, артефактов и подключения разных оценщиков.

## Последствия

### Плюсы

- Появляется единое место для оценки качества памяти и агента.
- Можно сравнивать версии поиска, индексов, промптов и моделей.
- Можно анализировать цепочку выполнения запроса, а не только итоговый ответ.
- Ragas и DeepEval можно подключать без смены основного контура.
- Собственные проверки безопасности остаются быстрыми и обязательными.

### Минусы

- Добавляется новая зависимость и новый runtime-контур.
- Нужно внимательно контролировать, какие данные попадают в трассы.
- Для промышленной эксплуатации потребуется отдельная настройка хранилища и доступа.
- LLM-оценки могут быть дорогими, медленными и не полностью стабильными.

## Требуемые действия

- Создать проектный план интеграции MLflow с системой памяти.
- Добавить `memory_quality_eval` как отдельную команду, не заменяющую `memory_eval`.
- Добавить настройки MLflow tracking URI и режима записи безопасного контекста.
- Добавить адаптер, который преобразует результат `memory.search` в MLflow trace/documents.
- Добавить безопасные наборы `MemoryEvalCase` для качества поиска.
- Добавить отчеты в `data/memory/eval/`.
- Зафиксировать запрет записи секретов и сырых чувствительных данных в MLflow.
- Рассмотреть подключение Ragas/DeepEval после появления стабильного базового MLflow-контура.
