# ADR-0013: Текст знания только в файле знания

## Статус

Accepted

## Дата

2026-05-22

## Контекст

ADR-0011 зафиксировал целевую схему: принятые знания хранятся в файлах и версионируются через Git, а база хранит метаданные и служебные состояния.

Текущий MVP-срез еще не полностью соответствует этому правилу:

- `MemoryKnowledgeItem.text` хранит текст знания в `data/db/knowledge_meta.sqlite3`;
- файл знания в `data/knowledge_repo/` тоже хранит тот же текст;
- `read_knowledge_item_file()` может тихо вернуть текст из базы, если файла нет;
- полнотекстовый индекс хранит индексируемое тело в `data/indexes/fulltext/search.sqlite3`;
- `MemorySearchDocument` частично дублирует метаданные `MemoryKnowledgeItem`;
- часть аудита и проверок все еще использует старые имена `chunk_id`/`returned_chunk_ids`;
- `MemoryClaim` остается будущей заготовкой, но в текущей реализации содержит поле `text`.

Production-внедрения и тестовой эксплуатации еще не было. Поэтому не требуется переносить существующие данные. Можно удалять и переименовывать поля напрямую штатными миграциями.

## Решение

### 1. Файл знания - единственный источник текста знания

Текст принятого знания должен храниться только в файле:

```text
data/knowledge_repo/**/*.md
```

`MemoryKnowledgeItem` больше не должен содержать поле полного текста знания.

`MemoryKnowledgeItem` остается карточкой знания и хранит только метаданные:

- `memory_id`;
- `scope`;
- `owner_user`;
- `kind`;
- `text_hash`;
- `sensitivity`;
- `scope_tokens`;
- `status`;
- `source_session`;
- `source_message_ids`;
- `source_content_hash`;
- `source_refs`;
- `source_code`;
- `source_kind`;
- `knowledge_file_path`;
- `knowledge_file_hash`;
- `knowledge_file_commit`;
- `index_status`;
- `provenance`;
- `metadata`;
- `created_by`;
- `supersedes`;
- timestamps.

Если файл знания отсутствует или хэш не сходится, это ошибка целостности. Система не должна подменять файл текстом из базы.

### 2. Writer service пишет сначала файл, затем метаданные

Целевой порядок записи:

```text
очищенный текст в памяти процесса
  -> временный файл
  -> atomic rename
  -> Git commit
  -> MemoryKnowledgeItem без текстового тела
  -> MemorySearchDocument
  -> индекс
```

Очищенный текст может существовать в памяти процесса только во время обработки запроса. В базу он не записывается.

### 3. Reader service всегда читает текст из файла

Выдача знания пользователю и агенту должна идти только через чтение файла знания:

```text
MemoryKnowledgeItem.knowledge_file_path
  -> reader service
  -> проверка прав
  -> проверка хэша
  -> текст из файла
```

Fallback на `MemoryKnowledgeItem.text` запрещен.

### 4. Индекс является перестраиваемой производной структурой

Индекс не является источником истины.

Допустимо хранить в индексе производные поисковые данные:

- `document_id`;
- токены полнотекстового поиска;
- векторные представления;
- служебные ранжировочные признаки;
- производные признаки для ранней фильтрации, если они полностью пересобираются из файлов и метаданных.

Недопустимо хранить в индексе извлекаемый полный текст знания. Текст для выдачи всегда читается из файла знания.

Для SQLite FTS нужно перейти на режим, при котором таблица индекса не хранит retrievable body. Если FTS5 недоступен, fallback не должен возвращаться к хранению полного текста в отдельной таблице; допустим fallback-сканирование файлов или деградация с понятным статусом.

### 5. Убрать дублирование между `MemoryKnowledgeItem` и `MemorySearchDocument`

`MemoryKnowledgeItem` отвечает за состояние знания.

`MemorySearchDocument` отвечает за техническое участие объекта в поиске.

Для корпуса `knowledge` `MemorySearchDocument` должен хранить минимум:

- `document_id`;
- `corpus_type`;
- `object_kind`;
- ссылку на `MemoryKnowledgeItem`;
- `index_status`;
- `body_hash`;
- `indexed_at`;
- технические метаданные индекса.

Права, чувствительность, источник и ссылки на источник для знания читаются из `MemoryKnowledgeItem`, а не дублируются как второй источник истины.

Для корпуса `source_data` `MemorySearchDocument` ссылается на `MemorySourceObject`. Смысловые метаданные исходного объекта читаются из `MemorySourceObject` и `MemorySource`.

### 6. Переименовать старые `chunk`-поля

Так как внедрения еще не было, поля можно переименовывать без совместимой миграции данных.

Нужно заменить:

- `returned_chunk_ids` -> `returned_document_ids`;
- `expected_chunk_ids` -> `expected_document_ids`;
- `chunk_id` compatibility aliases в поисковом backend удалить после перевода тестов и контрактов.

### 7. `MemoryClaim` не входит в активный MVP-путь

`MemoryClaim` - это будущий слой проверяемых утверждений.

Он нужен не для обычного "запомни", а для более зрелых сценариев:

- выделить из разных источников отдельные проверяемые утверждения;
- сравнить противоречащие утверждения;
- показать аудит, почему одно утверждение принято, а другое отклонено;
- строить будущую управляемую позицию системы по спорным вопросам.

Пример:

```text
Источник A: "Согласование закупок делает Иванов."
Источник B: "Согласование закупок делает Петров."

Claim 1: согласующий = Иванов
Claim 2: согласующий = Петров
Статусы: accepted / rejected / contested / superseded
```

Для MVP `MemoryClaim` не должен создаваться обычным `memory.remember`, не должен участвовать в обычном `memory.search` и не должен хранить копию текста знания. Если модель остается как заготовка, она должна хранить только структурированные данные и ссылки на доказательства. Если она мешает упрощению, ее можно удалить из MVP и вернуть отдельным будущим ADR.

### 8. Графовый поиск остается отдельной задачей

Этот ADR не проектирует новый графовый индекс. Графовый поиск будет возвращен отдельным блоком вокруг `document_id`, утвержденных типов сущностей и кандидатства новых типов.

## Рассмотренные альтернативы

### Оставить текст в `MemoryKnowledgeItem`

Отклонено. Это создает два источника истины: база и файл. При расхождении непонятно, чему верить.

### Хранить полный текст в поисковом индексе для удобства

Отклонено. Индекс должен быть перестраиваемой производной структурой. Выдача текста из индекса нарушает правило "текст знания только в файле".

### Объединить `MemoryKnowledgeItem` и `MemorySearchDocument`

Отклонено. Поиск нужен и по знаниям, и по исходным объектам. `MemorySearchDocument` оставляется как единая техническая карточка поиска, но без смыслового дублирования.

### Реализовать `MemoryClaim` уже в MVP

Отклонено. Сейчас это усложнит систему и снова создаст дублирование. Claim-слой нужен позже, когда появятся противоречивые источники, аудит утверждений и отдельный процесс проверки.

## Последствия

### Плюсы

- У знания остается один источник истины.
- Становится проще объяснять хранение памяти.
- База метаданных становится легче.
- Индексы можно безопасно пересобрать.
- Уменьшается риск расхождения текста в базе, файле и индексе.

### Минусы

- Чтение знания всегда зависит от доступности файла.
- Нужны строгие проверки целостности файлов и хэшей.
- Нужно переделать writer, reader, index worker, summary builder и тесты.
- SQLite FTS fallback становится сложнее, потому что нельзя хранить полный текст в отдельной таблице.

## Требуемые действия

- Создать проектный план `MEMORY_FILE_ONLY_KNOWLEDGE_BODY_PLAN.md`.
- Создать active plan `memory-file-only-knowledge-body.md`.
- Создать workflow-блок `workflow/active/memory-file-only-knowledge-body/`.
- Удалить `MemoryKnowledgeItem.text`.
- Перевести writer/index/search/summary/edit/delete на чтение и запись файлов.
- Убрать текстовые fallback из reader service.
- Убрать смысловое дублирование из `MemorySearchDocument`.
- Переименовать `chunk`-поля аудита и eval в `document`.
- Изолировать или удалить `MemoryClaim` из MVP active path.
