# Русификация интерфейса портала и подготовка к локализации

## Статус

MVP implemented, awaiting owner acceptance.

Архитектурное решение: `docs/adr/ADR-0022-interface-russification-and-localization-roadmap.md`.

Workflow-блок: `workflow/active/interface-russification/`.

## Цель

Сделать пользовательский интерфейс портала полностью русскоязычным, не меняя бизнес-логику, модели данных и текущую архитектуру модулей.

Отдельно зафиксировать, что полноценная локализация будет следующим архитектурным этапом, а текущий срез задает русскую базовую версию интерфейса.

## Пользовательская ценность

- Пользователь видит единый язык во всех основных сценариях.
- Оператору памяти и настроек проще понять действие до его выполнения.
- В интерфейсе меньше неоднозначных терминов вроде `Issue`, `Retry`, `Delete stale`, `Tool Registry`.
- Будущие модули получают понятное правило: видимые UI-строки пишутся на русском.

## Область работ

Включено:

- базовый шаблон и меню;
- ИИ-чат и ИИ-центр;
- ревью памяти и поискового индекса;
- центр настроек;
- аналитика;
- доска заявок;
- лист ожидания;
- инвентарь;
- формы и сообщения Django;
- человекочитаемые labels статусов, действий и настроек;
- документация и карта проекта.

Не включено:

- перевод технических кодов, URL, JSON-ключей, tool id, имен переменных и идентификаторов БД;
- перевод пользовательских данных, уже введенных в БД;
- выбор языка в UI;
- создание `.po`/`.mo` каталогов;
- внедрение автоматической проверки непереведенных строк как обязательного CI-gate;
- перевод внешнего Django admin за пределами моделей проекта.

## Термины

Допустимо оставлять как технические термины:

- `API`, `AD`, `FTS`, `JSON`, `MCP`, `PII`, `CITO`;
- коды вроде `workorders.get`, `source_data`, `indexing_pending`;
- имена переменных окружения;
- значения контрактов, если они являются машинными ключами.

Предпочтительные русские подписи:

- `AI` -> `ИИ`;
- `AI Hub` -> `ИИ-центр`;
- `Tool Registry` -> `Реестр инструментов`;
- `Task Types` -> `Типы задач`;
- `Memory review` -> `Ревью памяти`;
- `Issue` -> `Проблема`;
- `Issues` -> `Проблемы`;
- `Severity` -> `Критичность`;
- `Search index` -> `Поисковый индекс`;
- `Diagnostics` -> `Диагностика`;
- `Settings Center` -> `Центр настроек`;
- `Proposal` -> `Заявка на изменение`;
- `Diff preview` -> `Предпросмотр изменений`;
- `Actor` -> `Пользователь`;
- `Action` -> `Действие`;
- `Decision` -> `Решение`;
- `Document` -> `Документ`;
- `Job` -> `Задание`.

## Этапы реализации

### Этап 1. Документация

Статус: completed.

Задачи:

- создать ADR по русскоязычному UI и будущей локализации;
- создать активный план;
- создать workflow-блок с task packets;
- добавить краткое руководство по правилам UI-строк.

### Этап 2. Шаблоны и JavaScript

Статус: completed.

Задачи:

- перевести видимые строки в `templates/`;
- перевести пользовательские сообщения в `static/src/js/`;
- не менять CSS-классы, идентификаторы, data-атрибуты и системные события.

### Этап 3. Django labels, forms, messages

Статус: completed.

Задачи:

- перевести labels в `TextChoices`;
- перевести `verbose_name`, `help_text`, form labels и validation messages;
- перевести человекочитаемые описания настроек и ИИ-инструментов;
- обновить сгенерированный контракт tools при необходимости.

### Этап 4. Проверки

Статус: completed.

Задачи:

- запустить `manage.py check`;
- запустить релевантные unit-тесты по UI-adjacent модулям;
- запустить e2e по основным поверхностям;
- проверить `rg`-скан на видимые английские строки и зафиксировать допустимые исключения.

Фактический результат MVP:

- русифицированы ключевые шаблоны, JS-сообщения, Django labels, формы, описания настроек, ИИ-инструментов и типов задач;
- обновлены дефолтные контракты `contracts/ai/tools.json` и `contracts/ai/task_types.json`;
- рабочие копии `data/contracts/ai/tools.json` и `data/contracts/ai/task_types.json` обновлены для текущего запущенного стенда;
- сгенерированы label-only миграции для русских `verbose_name`, `help_text` и `TextChoices`;
- скриншоты визуальной проверки сохранены в `.local/playwright/russification-screenshots/`;
- скан видимого текста ключевых страниц сохранен в `.local/playwright/russification-visible-text.txt`.

Допустимые исключения после MVP:

- технические аббревиатуры и машинные идентификаторы: `API`, `AD`, `FTS`, `JSON`, `MCP`, `PII`, `ID`, URL, CSS/JS id;
- пользовательские данные, импортированные названия и исторические записи из БД;
- внешние названия вроде `Django`, `OpenAI`, `LanceDB`, если они являются именами продуктов.

## Будущий этап локализации

Для полноценной локализации нужно отдельное решение:

- какие языки поддерживать;
- хранить ли язык на уровне пользователя или организации;
- использовать ли Django `gettext`;
- как переводить JS-строки;
- как проверять непереведенные строки;
- как локализовать runtime-контракты;
- что делать с пользовательскими данными и импортированными названиями.
