# ADR-0030: Приведение системы памяти к архитектуре hybrid-knowledge v0.5

## Статус

Accepted

Реализован 2026-07-04 блоком `memory-hybrid-knowledge-v05-alignment` (архив: `workflow/archive/2026/memory-hybrid-knowledge-v05-alignment/`, 8 пакетов, приемка выполнена).

Дополнен 2026-07-06 по итогам ревью целевой архитектуры, проведенного после исполнения: добавлены нормативные правила — нормализация переводов строк канона, семантика `needs-reconcile` без петли, reconciler-синглтон, владелец эмиссии рёбер, golden set для `memory_eval`, DDL-предохранитель материализатора датасетов, матрица восстановления. Правила, не покрытые исполненным блоком, реализуются блоком `memory-alignment-hardening` (см. «Требуемые действия», дополнение).

Дополнен 2026-07-07 паттерном «сверка на записи» из концепта §7.4: протокол противоречия для семантического прохода и детерминированный проверочный набор `memory_impact` — реализуется тем же блоком `memory-alignment-hardening`.

Дополнен 2026-07-13 вторым проходом кросс-ревью параллельного kb-проекта: потребитель сигнала `contradicts` (issue `knowledge_contradiction`), авто-снятие детерминированных reconciler-issues, per-file область пересчета очереди при `--force` — реализуется тем же блоком.

## Дата

2026-07-03

## Контекст

Целевая архитектура гибридной системы знаний зафиксирована в `docs/architecture/hybrid-knowledge-architecture-v0.5.md` (далее — «концепт v0.5»). Ревью текущей реализации `apps.memory` (2026-07-03) показало системное расхождение с концептом и необоснованное усложнение:

- **Канон инвертирован.** По концепту v0.5 (инварианты №1–2) канон знаний — markdown-файл с frontmatter в git; база — производная проекция. Фактически авторитетные метаданные живут в `MemoryKnowledgeItem`, frontmatter файла генерируется из базы, а чтение файла жестко валидируется против хэшей из базы и падает при расхождении. Ручная правка файла ломает его чтение — прямое нарушение §7.1 концепта («прямое редактирование разрешено и является частью ценности»).
- **Push-очередь на пути записи.** Путь `memory.remember -> MemoryWriteRequest -> MemoryIndexJob -> воркеры -> файл -> метаданные -> индекс` содержит шесть сущностей и три воркера там, где концепт предписывает прямую запись markdown и pull-reconciler по content-hash.
- **Граф построен как отдельный LLM-extraction контур** (`MemoryGraphEntity`, `MemoryGraphExtractionRun`, `MemoryGraphSchemaProposal`, `MemoryGraphReviewItem`) — анти-паттерн, явно отвергнутый в §6 концепта; при этом графовый поиск в runtime не работает.
- **Параллельный аудит-трейл** (`MemoryKnowledgeEvent`, собственная ревью-машинерия в базе) дублирует git-историю, которую концепт делает единственным журналом ревизий (§7.2).
- **Автоупорядочивание файловых источников** (11 из 31 моделей `apps.memory`) — самостоятельный домен управления файлами, отсутствующий в концепте и раздувающий приложение памяти.
- **Контуры без потребителей**: профили ранжирования, generic external connector, MLflow-трассировка — построены до появления реальной потребности.

Масштаб на момент ревью: 81 файл и около 18 700 строк в `apps/memory`, 31 модель, 26 management-команд. История workflow-блоков показывает повторяющиеся корректирующие циклы (`memory-mvp-remediation`, `memory-mvp-simplification`, `memory-snapshot-chunk-removal`, `memory-file-only-knowledge-body`): слои строились раньше потребности и затем вырезались.

Владелец принял предложения ревью со следующими оговорками:

- автоупорядочивание файлов выносится в отдельное приложение **и замораживается**;
- упрощение поиска до одного профиля принимается, но концепция ADR-0016 откладывается **управляемо, как долг на будущее**, а не отменяется;
- правило против рецидива фиксируется **как рекомендация**, а не жесткое правило;
- отмена или изменение предыдущих архитектурных решений оформляется через ADR — этот документ и выполняет эту роль (раздел «Влияние на предыдущие ADR»);
- полезные механики, не описанные в концепте v0.5 (кандидатство, разделение персональной и корпоративной памяти и др.), должны быть явно проанализированы и сохранены осознанно (раздел «Сохраняемые механики»).

## Решение

### 1. Файл знания становится каноном

- Frontmatter файла в `data/knowledge_repo/` — авторитетный источник метаданных знания: `scope`, владелец, `kind`, `sensitivity`, `scope_tokens`, `status`, `source_refs`, `supersedes`, `lifecycle`.
- `MemoryKnowledgeItem` понижается до пересобираемой индексной карточки; целевое состояние — слияние с `MemorySearchDocument` в одну производную проекцию. Проекции можно снести и детерминированно пересобрать из файлов (инвариант №3 концепта).
- Расхождение хэша файла и проекции перестает быть ошибкой чтения. Вместо падения страница помечается `lifecycle: needs-reconcile` и попадает в очередь семантической ре-консолидации (двухдорожечная пересборка, §7.1 концепта).
- Прямое редактирование файлов знаний владельцем канона разрешено и является штатным путем.
- **Чистота канона:** frontmatter несет только присущие знанию метаданные и workflow-флаги (`lifecycle`). Состояние производных слоев — `index_status`, версии индексов, собственные хэши файла (`text_hash`, `file_hash`) — в канон не пишется и живет в проекциях; текущие поля такого рода из frontmatter выводятся при миграции. Иначе каждая переиндексация «трогает» канон, порождая пустые коммиты и ложные `needs-reconcile` (инвариант №9 концепта).
- **Идентичность концепта — путь файла** (норма OKF); `knowledge_id` сохраняется в frontmatter как producer-defined стабильный ключ, что конформности не нарушает. Переименование/перенос файла — через redirect-стаб.
- **Нормализация переводов строк.** Файлы канона всегда хранятся с LF: в корне `data/knowledge_repo/` лежит `.gitattributes` с единственной строкой `* -text` (git не выполняет конверсий), writer перед записью заменяет CRLF на LF, и все content-hash считаются по содержимому, нормализованному к LF. Без этого git-autocrlf на Windows дает ложные hash mismatch и паразитные `needs-reconcile` — целевая среда включает обе платформы.
- **Удаление знания пользователем — мягкое:** `status: deleted` + исключение из всех проекций и выдачи; текст остается в git-истории. Настоящее стирание (отзыв источника, right-to-erasure) — только через документированный runbook `git filter-repo` (§9.4 концепта). Это ограничение фиксируется в пользовательской документации памяти.

### 2. Прямая запись и pull-reconciler вместо push-очередей

- `memory.remember` пишет файл знания и выполняет git commit синхронно (операция дешевая), после проверки прав и политики источника. Существующие проверки writer service (lock, временный файл, атомарная замена) сохраняются как реализация записи, но без промежуточной очереди статусов.
- Вводится команда/воркер `memory_reconcile` (pull-модель): хранит хэш последнего материализованного коммита, при старте, по таймеру и после операций записи сравнивает его с HEAD и с гейтом по content-hash переиндексирует измененные файлы. Идемпотентно, самовосстанавливается.
- Таблицы `MemoryWriteRequest`, `MemoryIndexJob`, `MemoryKnowledgeEvent`, `MemoryReflectionRun` схлопываются в одну таблицу-очередь (статус, retry-счетчик, dead-letter) для фоновых задач: ре-консолидация, ingestion, переиндексация. `MemoryIngestionIssue` остается очередью проблем. Таблица-очередь реализуется по паттерну database queue backend из ADR-0029 (`status`, `locked_by`/`locked_until`, `attempt_count`, `next_attempt_at`, `idempotency_key`; leasing через `select_for_update(skip_locked=True)` на PostgreSQL); при реализации рассмотреть объединение с `MemoryExternalConnectorJob` в единый механизм вместо двух параллельных очередей.
- **Дисциплина одного писателя сохраняется явно.** Прежняя очередь сериализовала записи одним воркером; при переходе на синхронную прямую запись эта скрытая сериализация исчезает. Прямая запись выполняется только под межпроцессной блокировкой. Целевая среда — и Linux, и Windows в зависимости от условий внедрения, поэтому кроссплатформенная корректность блокировки — требование архитектуры, а не частный случай. Текущая реализация `knowledge_repo_lock` использует `fcntl` и на Windows молча вырождается в no-op — это дефект относительно этого требования. Кроссплатформенная блокировка (например, `msvcrt.locking` на Windows и `fcntl` на POSIX за единым интерфейсом, либо эксклюзивное создание lock-файла как платформонезависимый вариант) — обязательное условие этапа 1, до отключения очереди записи. После принятия ADR-0029 в основном репозитории допустима реализация через PostgreSQL advisory lock (`pg_advisory_xact_lock`) — кроссплатформенная по построению; файловый вариант остается для SQLite-fork.
- **Учет грязности — в проекции, не в каноне.** Обнаруженный hash mismatch регистрируется записью в таблице-очереди (ключ: путь файла + content-hash ревизии) и не изменяет файл. Флаг `lifecycle: needs-reconcile` в frontmatter пишет только семантический проход ре-консолидации; его коммиты выполняются от выделенного git-автора `memory-reconciler`, и детерминированный проход пропускает коммиты этого автора, изменяющие только workflow-ключи frontmatter (`lifecycle`, `audit`). Это исключает петлю «пометка меняет файл -> файл считается измененным -> новая пометка».
- **Reconciler — синглтон.** Запуск `memory_reconcile` начинается с попытки взять PostgreSQL advisory lock с фиксированным ключом; если lock занят — команда сообщает об уже идущем reconcile и завершается успешно без работы. Конкурентные вызовы «после записи» из нескольких web-воркеров безвредны и не дублируют работу; в dev-режиме на SQLite вместо advisory lock используется файловая блокировка.
- **Валидатор reconciler не допускает молчаливого понижения классификации:** ручная правка не может снизить `sensitivity`/`scope_tokens` — понижение держится pending до явного ревью (гардрейл §7.1 концепта); автоматическое движение метки — только вверх.
- Файлы `_summary.md` заменяются на OKF `index.md`, генерируемые reconciler'ом из файлов знаний, а не из запросов к базе (сейчас summary строится из queryset — еще одна зависимость от базы-канона).

### 3. Рёбра графа из frontmatter, отмена LLM-extraction контура

- Отдельный контур извлечения графа удаляется: модели `MemoryGraphEntity`, `MemoryGraphExtractionRun`, `MemoryGraphSchemaProposal`, `MemoryGraphReviewItem` и команды `memory_graph_extract`, `memory_graph_schema_discover` выводятся из кода после экспорта накопленных данных (если они представляют ценность).
- Взамен принимается путь §6 концепта: агент эмитит типизированные рёбра в блок `relations:` frontmatter в момент синтеза, против контролируемого словаря типов; детерминированный материализатор строит таблицу рёбер в основной БД (PostgreSQL по ADR-0029; в SQLite-fork — SQLite) без LLM.
- **Механика модерируемой схемы из ADR-0004 сохраняется в новой форме**: новый тип ребра или сущности не попадает в словарь без предложения и ревью владельцем графа. Меняется только носитель — словарь типов в конфигурации агента и валидатор вместо таблиц proposals.
- **Владелец эмиссии рёбер — семантический проход рефлексии.** `relations:` эмитятся при фоновой ре-консолидации (дешевое время, виден корпус целиком), а не в горячем пути `memory.remember`; инструкция эмиссии добавляется в конфигурацию семантического прохода вместе со словарем типов. Иначе словарь и материализатор останутся контуром без производителя рёбер.
- Пока агент не эмитит рёбра, графа нет — это честнее выключенного контура. Graph runtime search остается в `Later` backlog без изменений.

### 4. Ревизии и ревью через git

- Единый примитив изменений — `propose -> pending -> review -> stable` (§7.2 концепта): предложение агента или кандидат — git-ветка либо страница с `lifecycle: pending`; принятие — merge/фиксация авторизованным человеком; журнал — git-история; `log.md` генерируется из git, не ведется руками.
- Таблицы `MemoryKnowledgeCandidate`, `MemoryReviewAction`, `MemoryKnowledgeEvent` упраздняются. Очередь ревью строится над таблицей-очередью и `MemoryIngestionIssue`; экран ревью (ADR-0017) перенацеливается на pending-страницы и проблемы, его маршруты и назначение сохраняются.

### 5. Автоупорядочивание файлов — отдельное приложение, заморозка

- Контур File Source Auto Organization (11 моделей `MemoryFile*`, модули `file_organization*`, 6 команд, UI `/memory/files/` и `/memory/review/file-organization/`) выносится из `apps.memory` в отдельное приложение (рабочее имя `apps.filehub`) без функциональных изменений.
- После выноса контур **замораживается**: новые функции не разрабатываются до выбора реального пилотного источника и явного решения владельца. ADR-0025 остается принятым, но замороженным.

### 6. Один профиль ранжирования; ADR-0016 — управляемый долг

- В runtime остается один режим поиска знаний и один профиль ранжирования по умолчанию (гибрид FTS + вектор со слиянием RRF). Публичные параметры выбора профилей у ИИ-бота сокращаются до выбора корпуса (`knowledge` / `source_data`).
- Концепция профилей ранжирования ADR-0016 **не отменяется**: она фиксируется как отложенный архитектурный долг. Возврат к ней — после того, как `memory_eval` на реальном корпусе покажет измеримую пользу дифференциации профилей. Соответствующая запись в backlog переводится из `Next` в `Later` с этим критерием старта.
- **Критерий возврата делается проверяемым:** создается golden set eval-кейсов (минимум 20 пар «запрос -> ожидаемые знания» в `MemoryEvalCase` на реальном корпусе) и периодический прогон `memory_eval`. Без golden set условие «eval покажет пользу» не срабатывает никогда, и долг становится вечным; это относится и к критерию включения graph runtime search.

### 7. Разделение данных и знаний (data plane)

Вводится слой data store по §3.1 концепта.

**Граница** определяется детерминированным тестом, а не решением по каждому случаю:

- наблюдения, записи, time-series, требующие агрегации/SQL/as-of-запросов -> data store (append-only);
- документы, нужные для цитаты и провенанса -> raw sources (существующий корпус `source_data`);
- интерпретации и инсайты -> вики (файлы знаний).

**Управление составом data store — кандидатство на уровне датасетов, а не строк.** Операция `capture` разрешена только в заранее объявленный датасет. Дескриптор датасета — концепт-страница вики `type: Dataset`: frontmatter несет машиночитаемую часть (имя, схема полей, владелец, ретеншн, классификация, политика индексирования строк), проза — назначение; реестр датасетов в data store — производная проекция, материализуемая reconciler'ом из этих страниц, как граф из `relations:`. Новый датасет проходит тот же ревью-гейт §7.2 концепта (pending-страница -> ревью владельца данных -> merge -> материализация); инициаторы — человек, агент после отклоненного `capture` и фоновая рефлексия. Эволюция схемы — только аддитивная (новые необязательные поля) и только через ревью; несовместимое изменение = новый датасет (версия). `capture` идемпотентен: наблюдение несет дедуп-ключ (датасет + натуральный ключ/метка времени). Добавление строк в зарегистрированный датасет автоматическое и ревью не требует. Это замена прежнего механизма кандидатства применительно к данным: кандидатство знаний (персональное -> корпоративное) сохраняется отдельно (см. «Сохраняемые механики»), кандидатство данных переносится на уровень схемы (единый примитив модерируемой схемы, §7.3 концепта). Материализатор реестра выполняет только аддитивный DDL — `CREATE TABLE` для нового датасета и `ADD COLUMN` с NULL-допуском для эволюции; `DROP`, смена типов и переименования в автоматике запрещены и выполняются человеком по runbook.

**Маршрутизация наблюдений fail-safe, без staging-слоя.** Классификацию высказывания («запомни, что курс сегодня был 105» — наблюдение или интерпретация) предлагает LLM, но решает валидатор: запись в data store происходит только при совпадении высказывания со схемой зарегистрированного датасета; во всех остальных случаях создается файл знания. Наблюдения-сироты до появления датасета живут как обычные знания — вики и есть staging, отдельный промежуточный слой не вводится. Когда рефлексия замечает серию однотипных фактов, она предлагает датасет; после принятия записи мигрируют в data store, старые страницы помечаются `superseded`. Порог рекуррентности — настраиваемая эвристика рефлексии.

**Индексирование и поиск по data store:**

- строки данных по умолчанию **не** попадают в FTS/векторный индекс;
- для каждого датасета в вики создается концепт-страница (`type: Dataset`, `resource:` -> локация в data store) — она индексируется и находится обычным гибридным поиском; так пользователь и агент «находят» датасет;
- доступ к строкам — только типизированный интерфейс запросов (именованные, версионируемые запросы; агрегация/OLAP), отдельная операция агента (`query`/`capture`), не `memory.search`;
- инсайты, выведенные из данных, материализуются в вики как обычные знания с контрактом деривации `derived_from`/`as_of`/`window`/`computed_value`/`refresh` и находятся обычным поиском; утверждение из данных без пересчитываемого `derived_from` невалидно;
- полнотекстовая проекция строк конкретного датасета (например, текстовых логов) допустима только как осознанное исключение, зафиксированное в дескрипторе датасета.

Технологическая база: после принятия ADR-0029 естественный кандидат для data store — append-only таблицы в основной PostgreSQL БД (рядом с очередью и FTS-проекцией); DuckDB остается слоем OLAP-витрин и тяжелых агрегатов. Выбор фиксируется на старте этапа 5а.

### 8. Сохраняемые механики, отсутствующие в концепте v0.5

Концепт v0.5 писался как обобщенная система знаний и не покрывает ряд механик, введенных предыдущими ADR для многопользовательского портала. Они проанализированы явно; отмена по умолчанию не применяется.

| Механика | Источник | Решение |
|---|---|---|
| Персональная и корпоративная память (`scope`, владелец) | ADR-0005, ADR-0011 | **Сохранить.** Отображается на компартменты/метки §9.2 концепта: `users/<id>/` — персональный компартмент, `org/` — общий. Поля остаются в frontmatter, фильтрация — на пути чтения |
| Кандидатство персональное -> корпоративное | ADR-0005 | **Сохранить, сменить носитель.** Продвижение — это ревью-гейт §7.2: кандидат = pending-страница в `org/` (ветка или `lifecycle: pending`), принятие владельцем знаний = merge. Таблицы кандидатов упраздняются, механика и требование человеческого аудита — нет |
| Trust-политика источников (`trusted`/`review_required`/`blocked`) | ADR-0009, ADR-0010 | **Сохранить.** Гейт на ingest и retrieval; `review_required` порождает pending-страницы (review-then-publish). Простая трехуровневая схема из ADR-0010 без расширения |
| `sensitivity` + `scope_tokens`, фильтрация до контекста модели | ADR-0003 | **Сохранить.** Это ровно retrieval-time authorization §9.1 концепта. Метки переезжают в frontmatter, enforcement остается на пути чтения |
| Secret handles (секрет не попадает в знание, индекс, журнал) | ADR-0005 | **Сохранить без изменений.** Ортогонально концепту; `SecretHandle`/`SecretAccessAudit` остаются в базе — это не знание, а операционные данные |
| Обезличивание/PII-контур | ADR-0012, ADR-0018 | **Сохранить.** Соответствует §9.5 (egress-гейт) и анонимизации на ingest из стека §10 |
| Очередь проблем `MemoryIngestionIssue` + ревью-UI | ADR-0004, ADR-0017 | **Сохранить.** Соответствует паттерну очереди с DLQ §10.1; UI — операторский экран над очередью и pending-страницами |
| `SourceAdapter`/`SourceObjectEnvelope` как граница источников | ADR-0018 | **Сохранить.** Envelope — формат входа в raw sources/data plane; внутренние адаптеры (`workorders`, `waiting_list`) продолжают работать |
| Корпус `source_data` и fallback-поиск | ADR-0011, ADR-0013 | **Сохранить.** Соответствует канону raw sources |
| Типы знаний (`fact`/`preference`/`procedure`/`decision`) | ADR-0010 | **Сохранить** как обязательное поле `type` frontmatter (требование OKF) |
| `status`/`supersedes` жизненного цикла знания | ADR-0010 | **Сохранить** в frontmatter |
| Ночная рефлексия (дедуп, устаревание, кандидаты) | ADR-0005 | **Сохранить, сменить рамку.** Это семантическая дорожка ре-консолидации + `lint` (§7.1–7.2 концепта); работает по очереди `needs-reconcile`, а не по собственным таблицам запусков |
| `memory_eval` | ADR-0014 (частично) | **Сохранить** как основу для будущих решений по ранжированию; MLflow-трассировка остается отложенной |
| Claim/belief | ADR-0009, ADR-0010 | **Без изменений**: отложено за MVP решением ADR-0010; в терминах концепта возможная будущая реализация — уровень утверждений §5 |
| `MemoryKnowledgeEvent` (событийный журнал знаний) | ADR-0005 | **Удалить.** Дублирует git-историю; `log.md` генерируется из git |

### 9. Рекомендация против рецидива

**Рекомендация (не жесткое правило):** новый контур памяти (модель, воркер, команда, слой) создается только при наличии названного потребителя и оформленного ADR; при прочих равных выбирается решение, не добавляющее нового источника истины. Основание: три корректирующих цикла за два месяца показали, что слои «на будущее» в этом домене стабильно превращаются в долг. Отступление от рекомендации допустимо, но должно быть осознанным и упомянутым в соответствующем ADR или плане.

## Альтернативы

### Оставить базу каноном, файлы — проекцией (текущее состояние)

Отклонено. Два источника истины с жесткой хэш-связкой запрещают ручную правку, требуют очередей синхронизации и противоречат инвариантам №1–3 концепта v0.5, принятого как целевая архитектура.

### Полный переход на концепт v0.5 одним шагом (включая слой доступа как веб-сервис)

Отклонено как единый шаг. Слой доступа (§9.1 концепта) и компартменты-репозитории — отдельные этапы с собственными решениями; данный ADR выравнивает ядро (канон, пересборка, рёбра, ревизии, data plane), не переписывая портал.

### Удалить механики, отсутствующие в v0.5 (кандидатство, trust-гейты, secret handles)

Отклонено. Концепт v0.5 — обобщенный скелет; перечисленные механики закрывают реальные потребности многопользовательского портала и совместимы с примитивами концепта (ревью-гейт, классификация, очередь с DLQ). Их отмена была бы потерей, а не упрощением.

### Заморозить всё как есть и развивать только по концепту в новых модулях

Отклонено. Инверсия канона — активный источник хрупкости (ручная правка ломает чтение), а параллельные контуры без потребителей продолжают требовать сопровождения и тестов.

## Последствия

### Плюсы

- Один канон знаний: markdown + git; проекции сносимы и пересобираемы; ручная правка — штатный путь.
- Путь записи сокращается с шести сущностей до «файл + commit + reconciler».
- Ожидаемое сокращение: с 31 модели (33 после PostgreSQL-среза ADR-0029) до ~12–14 (Source, SourceObject, SearchDocument, MemoryFullTextIndex, IngestionRun/Issue, таблица-очередь, таблица рёбер, SecretHandle + аудиты, EvalCase), команд — с 26 до ~8–10; автоупорядочивание уходит в отдельное замороженное приложение.
- Git-история — единственный журнал ревизий; исчезает дрейф двух аудит-трейлов.
- Рёбра графа появляются дешево и детерминированно, когда появятся — без двойной оплаты LLM.
- Появляется явная граница данных и знаний с контрактом деривации и управляемым составом data store.

### Минусы

- Требуется миграция: перенос авторитета метаданных в frontmatter, пересборка проекций, удаление таблиц с сохранением данных, вынос файлового контура.
- Часть готового кода (ревью-таблицы, graph extraction, статусные машины очередей) списывается.
- Обновление тестов, e2e-сценариев, операторской документации и AGENTS.md (перечень проверок памяти).
- В окне между ручной правкой и reconcile объявленные рёбра и проекции могут отставать (мягкая деградация, предусмотренная §7.1 концепта).

Дополнение 2026-07-07 (паттерн «сверка на записи», концепт §7.4, по итогам обсуждения в параллельном kb-проекте; реализуется тем же блоком `memory-alignment-hardening`):

- Паттерн именован в концепте (§7.4): несогласованность рождается в момент записи, поэтому запись сверяется с детерминированным проверочным набором; вторая линия той же защиты — lint/семантический проход; фальсифицируемость — golden set.
- Адаптация к этой системе: `memory.remember` — захват намерения (дешевый, идемпотентный, lifecycle `pending`), нулевой шаг сверки в горячий путь не добавляется. Роль пишущего-синтезатора несет семантический проход рефлексии, поэтому протокол противоречия ложится в его инструкцию: конфликт с существующей страницей не правится молча — эмитится ребро `contradicts` с провенансом (тип уже `accepted` в словаре), разрешение за человеком через review-очередь.
- Команда `memory_impact <путь|knowledge_id>` — детерминированный проверочный набор для прямых правок канона (строго read-only): входящие рёбра (таблица рёбер как обратный индекс), исходящие `relations:` из frontmatter, топ гибридного поиска `memory_search` с сохранением trust/scope-фильтрации.
- Golden set читается двояко: качество `query` и recall проверочного набора на записи — механический кейс «начало тела страницы -> её документ» есть проверка самонахождения (черновик, похожий на существующую страницу, обязан поднять её).
- Ориентация в начале сессии (чтение `index.md` и свежего хвоста `log.md`) зафиксирована в концепте; локальный носитель появится при подключении runtime-агентов к сессионному recall-протоколу — вне hardening-блока.

Дополнение 2026-07-13 (второй проход кросс-ревью параллельного kb-проекта; реализуется тем же блоком `memory-alignment-hardening`):

- Сигнал противоречия получает механического потребителя: детерминированный детектор в `memory_reconcile` после материализации рёбер — новое ребро `contradicts` дает `MemoryIngestionIssue` вида `knowledge_contradiction` (get_or_create по паре страниц, без дублей). Правило: протокол, порождающий сигнал, обязан назначить его потребителя, иначе сигнал декоративен.
- Правило замыкания причин: детерминированные reconciler-issues снимаются автоматически при исчезновении условия (файл `canon_file_missing` вернулся, ребро `contradicts` удалено из `relations:`); ручной статус `ignored` не перезаписывается. У каждого детектора очереди/issue должен быть назван акт, гасящий его условие, — иначе причина взводится вечно.
- Область пересчета очереди при `--force` — строго per-file (`git log <базис>..HEAD -- <путь страницы>`); глобальный скан коммитов без ограничения путем запрещен: посторонний коммит (регенерация `index.md`) воскрешал бы все страницы. Все системные пути записи канона (пометки reconciler, регенерация `index.md`/`log.md`, будущие правки `relations:` семантическим проходом) коммитят от выделенного автора `memory-reconciler`; общий repo-идентити `Local Business Memory Writer` системным не считается.
