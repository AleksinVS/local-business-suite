## Что проверено

Исполнение (из executor report + независимая проверка):

- `./data:/app/data:ro` добавлен agent-runtime в `docker-compose.yml` и
  `docker-compose.prod.yml` (ровно +2 строки в каждом; `env_file` не тронут);
  `docker compose config` подтверждает `read_only: true` bind-mount в обоих файлах;
- новый `services/agent_runtime/contract_cache.py`: инвалидация по
  `(st_mtime_ns, st_size, st_ino)`, потокобезопасность через lock, путь контракта
  не мемоизируется (появление runtime-копии после старта подхватывается);
- старт-лог фактического источника контрактов (WARNING при fallback на дефолты),
  `GET /health/details` дополнен источниками контрактов (аддитивно);
- `pytest services/agent_runtime/tests -q` → **67 passed** (55 старых + 12 новых);
  независимая проверка подтвердила содержательность тестов, включая сильный
  регрессионный тест: `os.replace` с искусственно уравненными mtime/size
  инвалидирует кэш по inode;
- исполнитель дополнительно проверил реальный контейнер: `docker run` с ro-томом,
  отказ записи (`Read-only file system`), подхват правки файла хоста без перезапуска;
- scope-нарушений нет; документация (README agent-runtime, DEPLOYMENT.md,
  WINDOWS_RUN.md, `.desc.json`, PROJECT_STRUCTURE.yaml) обновлена.

## Замечания (minor, в рекомендации)

1. `/health/details` делает двойную резолюцию путей контрактов (безвредно,
   диагностический endpoint).
2. Нет unit-теста на сценарий «файл закэширован, затем удалён» (поведение вручную
   проверено верификатором — корректный `FileNotFoundError`, кэш не отдаётся).
3. Битый JSON пробрасывает `JSONDecodeError` (поведение не изменилось относительно
   до-пакетного; упрощение относительно Django-стороны осознанно задокументировано
   в contract_cache.py).
