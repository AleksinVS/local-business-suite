# ADR-0010: Упрощение MVP-архитектуры памяти

## Статус

Accepted

## Дата

2026-05-21

## Контекст

В предыдущих решениях система памяти развивалась сразу в нескольких направлениях:

- память из чата;
- документы и внешние источники;
- граф знаний;
- надежные источники;
- утверждения (`MemoryClaim`);
- принятые позиции системы (`MemoryBelief`);
- ночная обработка знаний;
- ссылки на секреты.

Такой набор возможностей нужен для зрелой корпоративной памяти, но для MVP он создает лишнюю сложность. Один и тот же факт сейчас может быть представлен сразу несколькими объектами: сохраненным знанием, событием, снимком, текстовым фрагментом, утверждением и принятой позицией системы.

Пользователь согласовал упрощение:

- главным объектом сохраненной смысловой памяти MVP становится `MemoryKnowledgeItem`;
- `MemoryBelief` удаляется из MVP и переносится на следующие этапы;
- поиск по векторам, полнотекстовому индексу и графу сохраняется, но детальная стратегия использования графа будет проработана отдельно;
- текущая команда "рефлексии" должна быть переосмыслена как обработчик очереди явных запросов на запоминание;
- граф знаний сохраняется как дополнительный слой, но его эффективное использование будет вынесено в отдельное решение;
- правила надежных источников упрощаются до понятной схемы для MVP.

Вопрос о техническом хранении сообщений чата и текущих файловых проекциях памяти специально не входит в это решение. Он будет проработан отдельно.

## Решение

### 1. Главный объект памяти MVP

Для MVP главным объектом сохраненного знания является `MemoryKnowledgeItem`.

Он отвечает на простой вопрос: "Что система действительно запомнила?"

`MemoryKnowledgeItem` хранит:

- текст знания;
- личную или общую область действия;
- владельца для личной памяти;
- права доступа через `scope_tokens`;
- чувствительность данных;
- ссылки на исходные сообщения или другой источник;
- состояние знания: активно, удалено, заменено, помещено в карантин.

Явный запрос пользователя "запомни" должен создавать или обновлять `MemoryKnowledgeItem`. Автоматическое создание дополнительных смысловых слоев в обычном пути MVP не требуется.

### 2. `MemoryClaim` остается необязательным слоем

`MemoryClaim` описывает отдельное проверяемое утверждение с источником и состоянием проверки.

Для MVP `MemoryClaim` не должен быть обязательным результатом каждого запроса "запомни" и не должен быть главным объектом поиска. Его допустимо оставить как заготовку для будущих задач:

- проверка спорных утверждений;
- кандидаты в общую память;
- аналитика из знаний;
- подготовка к будущему слою принятых позиций системы.

Если реализация усложняется из-за `MemoryClaim`, его создание в обычном пути можно отключить до отдельного этапа.

### 3. `MemoryBelief` переносится за пределы MVP

`MemoryBelief` больше не входит в обязательную MVP-архитектуру.

Причина: в текущем виде `MemoryBelief` почти дублирует `MemoryKnowledgeItem` и `MemoryClaim`, но добавляет отдельную таблицу, состояния, связи, поиск и тесты. Польза этого слоя появится позже, когда в системе будут:

- несколько противоречивых источников по одному вопросу;
- проверка утверждений владельцем знаний;
- устаревание и замена знаний;
- сбор "текущей позиции системы" из нескольких доказательств.

До этого этапа `MemoryBelief` нужно удалить из обычного пути выполнения и из поиска. Если миграции еще не зафиксированы в основной ветке, предпочтительно не вводить таблицу `MemoryBelief`. Если таблица уже есть в среде, исполнительский план должен предусмотреть безопасную миграцию удаления или выключение использования без потери данных.

### 4. Поиск в памяти

В MVP сохраняются три технических канала поиска:

- векторный поиск;
- полнотекстовый поиск;
- поиск по графу.

При этом `MemoryBelief` не должен быть отдельным источником обычного поиска.

Детальные условия эффективного использования графа не входят в это решение. Они будут оформлены отдельным планом. До этого графовый поиск должен оставаться совместимым, но не должен требовать тяжелой обработки локальной моделью в каждом запросе.

### 5. Текущая "рефлексия"

Текущая команда `memory_reflect_chats` фактически выполняет более узкую работу:

- обрабатывает очередь явных запросов "запомни";
- создает кандидаты в общую память по простым признакам;
- показывает служебные счетчики.

Это еще не полноценная ночная рефлексия, которая анализирует историю чатов, выделяет новые знания, объединяет повторы и ищет важные закономерности.

Решение для MVP:

- переименовать или разделить текущий обработчик очереди;
- оставить полноценную ночную рефлексию отдельным следующим этапом;
- не называть простую обработку очереди полноценной рефлексией в пользовательской и технической документации.

### 6. Надежные источники

Для MVP правила надежности источников должны быть понятными:

- `trusted` - источник можно использовать в обычном ответе бота после проверки прав доступа и чувствительности;
- `review_required` - источник можно сохранять и разбирать, но нельзя напрямую отдавать боту до проверки;
- `blocked` - источник нельзя использовать.

Дополнительные классы источников допустимы как поясняющие признаки, но не должны превращать MVP в сложную матрицу состояний.

Существующие значения должны быть сведены к этой простой схеме через совместимый переход:

- `trusted` остается `trusted`;
- `candidate_only` становится `review_required`;
- `quarantined` становится `review_required` с более строгими ограничениями или `blocked`, если источник опасен;
- `blocked` остается `blocked`.

## Что не входит в это решение

- Перенос сообщений чата в отдельный SQLite-файл.
- Пересмотр того, где физически хранить историю чата.
- Полная стратегия эффективного использования графового поиска.
- Удаление или изменение файловых проекций личной и общей памяти.
- Production UI для проверки знаний.
- Сложная проверка противоречий между источниками.

## Последствия

### Плюсы

- Система памяти становится проще для объяснения и сопровождения.
- У явного "запомни" появляется один главный результат: `MemoryKnowledgeItem`.
- Уменьшается дублирование данных.
- Поиск не тащит в MVP слой `MemoryBelief`, который пока не дает достаточной пользы.
- Ночная обработка знаний отделяется от простой обработки очереди.
- Правила надежных источников становятся понятнее владельцам данных.

### Минусы

- Сложные сценарии с противоречиями между источниками будут отложены.
- Принятая позиция системы по спорным вопросам пока не будет отдельным объектом.
- Часть уже созданного кода и документации нужно привести к новой MVP-границе.
- Для существующих сред может потребоваться аккуратная миграция, если таблица `MemoryBelief` уже была создана.

## Требуемые действия

- Создать проектный план упрощения MVP-памяти.
- Создать исполнительский workflow-блок.
- Убрать `MemoryBelief` из обычного пути `memory.remember`.
- Убрать `MemoryBelief` из `memory.search`.
- Проверить, нужен ли `MemoryClaim` в первом этапе или его создание тоже должно быть отключено в обычном пути.
- Переименовать или разделить текущий обработчик очереди памяти и будущую ночную рефлексию.
- Упростить контракт надежных источников или добавить совместимый слой преобразования к простой схеме.
- Обновить тесты, пользовательскую документацию и документацию развертывания.
