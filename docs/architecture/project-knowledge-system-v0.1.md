# Система проектных знаний для ИИ-агентов (project knowledge base)

**Статус:** проект (Proposed) · **Версия:** 0.5 · **Дата:** 2026-07-06
**Базовый документ:** [hybrid-knowledge-architecture-v0.5.md](hybrid-knowledge-architecture-v0.5.md) (текст версии 0.6)
**Назначение:** адаптация гибридной концепции управления знаниями под узкую задачу — локальная база знаний **о самом проекте** для агентов-архитекторов и агентов-разработчиков.
**Исполнительная документация:** [docs/planning/active/project-knowledge-system-mvp.md](../planning/active/project-knowledge-system-mvp.md)
**Конвенция имени файла:** имя файла зафиксировано и не меняется при ревизиях; версия — только в этой шапке (как у базовой концепции).

Версии до 0.2 спроектированы только от текста базовой концепции; с v0.3 добавлен анализ существующих артефактов репозитория (§11). Сверка с другими ADR — на этапе принятия.

**Изменения v0.2 (по уточнению постановки):**
- зафиксирован главный потребитель — сами агенты («агент пишет для агента»); человек читает канон только при ревью и ознакомлении (§1);
- стиль страниц переориентирован с прозы на плотный структурированный формат; введены рекомендуемые скелеты разделов по типам концептов (§5.2);
- снят гардрейл «агент не переписывает прозу человека»: страницы принадлежат агентам, свободная реструктуризация — часть обслуживания (§7.3, §9);
- фоновый контур качества оформлен явно: уровень 1 — детерминированный lint с детекторами при каждом sync, уровень 2 — периодическая сессия-уборщик (§7.4–7.5);
- добавлена детекция устаревания страниц относительно источников (source-drift) — адаптация «детекции устаревания» из контракта деривации концепции (§7.4);
- каденция обслуживания переведена из открытых вопросов в стартовые пороги триггеров (§7.5).

**Изменения v0.3 (по итогам ревью владельца):**

- анализ графа кода: полный AST-граф отклонён; отложенно — детерминированная проекция импортов для сверки рёбер `depends_on` с фактом (детектор `code-drift`, §10.1);
- новый раздел «Интеграция с существующими артефактами репозитория» (§11); прежние §11–12 стали §12–13.

**Изменения v0.4 (по обсуждению карты проекта с владельцем):**

- проекция поглощает `.desc.json` (узлы дерева в общем поиске); новый детектор `desc-mismatch` для сверки описаний узла и накрывающей страницы (§7.4, §7.5);
- карта остаётся на текущем генераторе и каноне; `kb export structure` — перспективная замена формат-в-формат с обогащением узлов ссылками на страницы вики (§11.1);
- фаза 3 плана внедрения переориентирована с миграции описаний на интеграцию карты (§12).

**Изменения v0.5 (внутреннее ревью этой системы + перенос находок ревью основной системы знаний):**
- **акторная модель sync** (`--actor`, `--session-start`, git-трейлер `KB-Actor:`): устранены ложные `human-edited` от собственных незакоммиченных правок агента и самовоспроизводство очереди правками уборщика (§7.4, §8, §8.1);
- **очередь**: составной ключ (страница, причина), операция `--defer`, dead-letter после 3 неудач (§6, §7.5);
- **детекторы разделены на два класса**: stateless (пересчитываются при `--full` из git) и инкрементальные (best-effort); добавлена матрица состояния и восстановления (§7.4, §8.2);
- **`source-drift` — батчевый**: один проход `git log` на весь sync вместо вызова на каждый путь (бюджет «дельта-sync < 1 с») (§7.4);
- **токенизатор заменён**: `trigram` → `unicode61` + префиксные индексы; `trigram` не находит термины короче 3 символов (`AI`, `UI`, «БД»), критичные для этого корпуса (§6);
- **LF-дисциплина** (перенос находки №1 ревью основной системы): `.gitattributes` в бандле, запись только LF, все хэши — по нормализованному содержимому (§3, §5.1, §6);
- **фальсифицируемые триггеры** (перенос находки №3): golden set + `kb eval`; триггер векторного индекса привязан к измеримому hit-rate (§7.6, §10);
- **владелец полноты графа — уборщик** (перенос находки №4): предупреждение «страница без исходящих рёбер», обязанность обогащения связей (§7.4, §7.5);
- **одиночность sync** (перенос находки №5): эксклюзивная блокировка SQLite, конкурентный запуск корректно пропускается (§6);
- уточнены: формулировка валидатора (блокировка протокольная, не механическая), эвристика `rename-candidate`, правило deepest-match для `covers:`, семантика `timestamp`, fallback типа концепта, провенанс без коммита, лимит `index.md`, обновление входящих ссылок при слияниях, реалистичные оценки фаз и лимит стартового корпуса (§5.2–5.3, §7.3–7.5, §12);
- место CLI зафиксировано: `scripts/dev/kb.py` (§7.1); DDL-предохранитель для будущего data plane (перенос находки №8) (§10);
- создана исполнительная документация для реализации слабой LLM: `docs/planning/active/project-knowledge-system-mvp.md`.

Находки №2 (петля `needs-reconcile`) и №7 (фрагментный поиск) ревью основной системы здесь уже закрыты конструктивно: флаг грязности живёт только в проекции (§8), извлечение по разделам — базовое решение (§6).

## 1. Задача и границы

**Цель.** Дать агентам, работающим над проектом, систему знаний о нём: устройство подсистем, принятые паттерны, ограничения среды, неочевидные факты («грабли»), связи между компонентами. Требования: простая, локальная, быстрая, полная, индексируемая.

**Потребители.** Главный потребитель — сами ИИ-агенты (архитектор, разработчик, проверяющий): **агент пишет для агента.** Человек (владелец) обращается к канону только в двух редких случаях: контроль (ревью диффов) и ознакомление. Все работают на одной машине под одной учётной записью.

Следствия «агент пишет для агента»:

- человекочитаемость — второстепенное требование: стиль страниц оптимизируется под плотность и машинную предсказуемость (§5.2), достаточный уровень — «читаемо при ревью диффа»;
- человек практически не правит канон: машинерия согласования ручных правок сведена к страховочному минимуму (§8), а агентам разрешена свободная реструктуризация страниц без человеческого гейта (§7.5).

**Не-цели (сознательно исключено):**

- слой доступа, аутентификация, классификация, компартменты, деклассификация, egress (§9 концепции) — модель угроз пуста: единственный человек-владелец, агенты работают от его имени;
- data plane / data store для наблюдений и time-series (§3.1 концепции) — в задаче «знания о проекте» наблюдений нет;
- PostgreSQL и любые серверные СУБД, веб-сервисы, очереди на выделенной инфраструктуре;
- отдельный сервис-оркестратор — семантическую работу выполняет сам рабочий агент.

| Практика | Следствие для дизайна |
|---|---|
| LLM-wiki (Karpathy) / OKF: концепт = markdown-файл с frontmatter, идентичность = путь | Канон знаний — markdown-бандл в git, совместимый с OKF на уровне контейнера |
| Прогрессивное раскрытие (OKF `index.md`, соглашение `llms.txt`, паттерн memory-bank агентных сред) | Дешёвая точка входа в сессию: агент читает только `index.md`, дальше — точечно |

| Детерминированные производные слои: пересборка с нуля — штатная операция, обнаружение изменений по content-hash | Проекцию можно снести в любой момент; классы состояния и восстановление — §8.2 |
| Кроссплатформенность (Windows + Linux): eol-конверсия git молча меняет содержимое файлов | Принудительный LF в бандле (`.gitattributes`) + нормализация перед любым хэшированием (§6) |
| Канон читают в основном LLM: плотный структурированный текст дешевле по токенам и предсказуемее прозы | Телеграфный стиль страниц, фиксированные скелеты разделов по типам концептов (§5.2) |
| Инструменты без зависимостей живут дольше | CLI на стандартной библиотеке Python (+ PyYAML для frontmatter), без каркасов и служб |

## 3. Инварианты (адаптированные)

1. **Один канон — git-репозиторий.** Вики-бандл — канон знаний; остальные файлы репозитория (код, ADR, документация) играют роль raw sources: провенанс указывает в них через `путь@коммит` (иммутабельность обеспечивает git). Отдельные каноны data store и raw sources из концепции здесь не нужны.
2. **LLM пишет в markdown, а не в БД.** Прямой записи агента в SQLite нет ни в одном сценарии.
3. **SQLite — проекция.** Полностью пересобирается из бандла; результат детерминирован. Смена токенизатора FTS (в перспективе — эмбеддинг-модели) = плановая полная переиндексация.
4. **Обнаружение изменений по content-hash от нормализованного содержимого** (CRLF/CR → LF перед хэшированием), не по `mtime`. Иначе eol-конверсия git на Windows даёт фантомные «изменения» и паразитные переиндексации.
5. **Read-path дисциплина:** индекс → граф → чтение только выбранных страниц. Сканирование тел страниц ради релевантности запрещено.
6. **Состояние производного слоя не пишется в канон** (инвариант №9 концепции, применён строже): очередь ре-консолидации, хэши, статусы индексации живут только в SQLite. Frontmatter несёт лишь присущие концепту поля и редакционный `lifecycle`.
7. **Схемы утверждаются только человеком.** Словари типов концептов и рёбер — файлы в бандле; агент эмитит только против утверждённых записей. Предлагать расширение может кто угодно, включая агента (записи со `status: proposed`), но действующей запись делает только владелец (`status: approved`) — упрощённый примитив «предложение → ревью → принятие» §7.3 концепции.
8. **OKF — цель, не блокер.** Отступления фиксируются (раздел 9 этого документа) и оформляются ADR при принятии системы.

## 4. Архитектура

```
Git-репозиторий — единый канон
 ├─ код, ADR, docs, contracts        ← «raw sources»: сюда указывает провенанс (путь@коммит)
 └─ docs/knowledge/                  ← вики-бандл (OKF-контейнер): страницы-концепты
         │
         │  kb sync — детерминированная материализация (без LLM)
         ▼
 .local/knowledge/kb.sqlite3         ← производный слой (пересобираемый)
    ├─ chunks + FTS5 (BM25)          — извлечение на уровне раздела
    ├─ edges                         — граф типизированных рёбер
    └─ meta + очередь reconcile      — состояние проекции и «грязные» страницы
         ▲
         │  query / lint / eval (детерминированные операции — CLI `kb`)
 Рабочий агент (Claude Code и др.) = оркестратор:
 ingest и семантическая ре-консолидация — протокол в AGENTS.md
```

Ключевая адаптация относительно концепции: **отдельного агента-оркестратора нет.** Три операции распределены так:

- **ingest** — рабочий агент пишет/обновляет страницы бандла по ходу задачи (протокол — §7.3), затем вызывает `kb sync`;
- **query** — CLI: гибридный поиск по индексу + обход графа, выдача списка страниц и релевантных разделов; чтение тел — агентом, точечно;
- **lint** — двухуровневый контур качества: детерминированные проверки и детекторы при каждом sync (уровень 1), семантическое обслуживание — периодической сессией-уборщиком по очереди (уровень 2, §7.4–7.5).

## 5. Канон: бандл и страница

### 5.1 Раскладка

```
docs/knowledge/
  .gitattributes            # принудительный LF для всего бандла (инвариант №4)
  index.md                  # навигация, прогрессивное раскрытие; поддерживается агентами
  schema/
    concept-types.yaml      # контролируемый словарь типов концептов (записи со status)
    edge-types.yaml         # контролируемый словарь типов рёбер (записи со status)
    templates/
      <type>.md             # рекомендуемый скелет разделов для каждого типа концепта
  eval/
    golden.yaml             # golden set: запрос → ожидаемые страницы (§7.6)
  concepts/
    <домен>/<концепт>.md    # неглубокая иерархия по доменам, файл = концепт
```

- **Идентичность концепта = путь** относительно бандла без `.md` (соглашение OKF): `concepts/core/integrations-sdk`. Разделитель в идентификаторах — всегда `/` (на Windows пути нормализуются).
- **Переименование рвёт ссылки**, поэтому не поощряется. При необходимости: на старом пути остаётся страница-заглушка с ребром `supersedes` от новой страницы и `lifecycle: superseded`; уборщик обновляет входящие ссылки (§7.5).
- `log.md` как файл **не материализуется**: хронологию отдаёт `kb log` из git-истории по запросу (см. отступления, §9).
- `index.md` ограничен ~150 строками (контроль — lint); при росте — под-индексы по доменам (прогрессивное раскрытие).

### 5.2 Страница-концепт

```yaml
---
type: component               # из schema/concept-types.yaml (обязательное, OKF)
title: Общий SDK интеграций   # обязательное
description: Транспорт, аутентификация и обработка ошибок для внешних систем  # одна строка, обязательное
timestamp: 2026-07-04T12:00:00Z   # дата последней содержательной сверки с источниками (§7.4)
lifecycle: current            # draft | current | superseded (редакционный статус)
tags: [integrations, core]
sources:                      # первоисточники синтеза страницы (пути в репозитории)
  - docs/adr/ADR-00XX-....md
  - apps/core/integrations/
covers:                       # зона владения: пути, которые страница ОПИСЫВАЕТ (см. §10.1, §11.1)
  - apps/core/integrations/
relations:                    # типизированные рёбра — эмитятся агентом в момент синтеза
  - type: part_of
    target: concepts/core/core-app
  - type: implements
    target: concepts/decisions/integration-sdk-decision
    provenance: docs/adr/ADR-00XX-....md@a1b2c3d   # путь[@коммит][#якорь], рекомендуется
---

# Общий SDK интеграций

## Назначение и границы
...синтез, разбитый на разделы — уровень чанкинга для индекса...

## Неочевидные ограничения
...
```

Четыре обязательных поля (`type`, `title`, `description`, `timestamp`) — по замечанию концепции о референс-парсере OKF. Остальные ключи — producer-defined, что OKF явно разрешает.

**Правила содержания (агент пишет для агента):**

- **Стиль — телеграфный и плотный:** списки, таблицы, короткие утверждения; повествовательная проза не нужна. Но плотность не отменяет причинность: **каждое неочевидное утверждение сопровождается причиной или ссылкой** («что» без «почему» — деградация синтеза в список фактов).
- **Предсказуемая структура:** для каждого типа концепта — рекомендуемый скелет разделов с фиксированными H2 (`schema/templates/<type>.md`). Пример для `component`: `Назначение и границы` / `Устройство` / `Зависимости и контракты` / `Ограничения и грабли` / `Проверки`. Одинаковые заголовки дают адресное чтение (`kb page <id> --section`) и стабильный чанкинг.
- Страница — **синтез, а не копия**: не дублировать текст ADR/README, а давать выжимку следствий и ссылку-источник в `sources`. Числа и точные сигнатуры не «замораживать» в тексте — ссылаться на код.
- **`sources:` и `covers:` — разные поля:** `sources` — из чего страница синтезирована (ADR, обзоры, код), `covers` — какие пути репозитория она описывает (зона владения). `covers` питает детектор `source-drift` (§7.4), сверку с картой проекта и обогащение её экспорта (§11.1), сверку с графом импортов (§10.1).
- **Вложенность `covers:` легитимна** (страница `domain` накрывает `apps/`, страница `component` — `apps/inventory/`): при разрешении «какая страница описывает путь» действует правило **deepest-match** (побеждает самое глубокое покрытие). Предупреждение lint — только при совпадении зон или частичном пересечении на одном уровне.
- **`timestamp` — дата последней содержательной сверки с источниками.** Обновляется только когда содержимое проверено/обновлено против `sources:`/`covers:`. Механические правки (перетипизация рёбер, починка ссылок) `timestamp` **не трогают** — иначе детектор `source-drift` слепнет.
- Ориентир размера — до ~200 строк; крупные темы разрезать по `part_of`.
- **Язык — русский**, как остальная документация проекта (упрощает редкое ревью владельцем); перевод корпуса на английский ради токен-экономии — возможная оптимизация, решение за владельцем (§13).
- Рёбра проставляются **в момент синтеза** (агент знает, *почему* связывает A и B); пост-фактум восстановить тип связи без повторного LLM-прохода нельзя (§6 концепции).

### 5.3 Стартовые словари

**Типы концептов** (`schema/concept-types.yaml`):

| Тип | Что описывает |
|---|---|
| `domain` | предметная область / подсистема верхнего уровня |
| `component` | приложение, модуль, сервис |
| `integration` | внешняя система и контур обмена с ней |
| `pattern` | принятый паттерн или конвенция (со ссылкой на решение) |
| `constraint` | ограничение среды, deployment, регуляторики |
| `insight` | неочевидный факт, причинность, «грабли» |
| `source-summary` | сводка объёмного первоисточника со ссылками на порождённые концепты |

**Типы рёбер** (`schema/edge-types.yaml`), направление фиксировано определением:

| Тип | Читается как | Симметрия |
|---|---|---|
| `part_of` | A входит в состав B | нет |
| `depends_on` | A использует B | нет |
| `interacts_with` | A обменивается с B | да |
| `implements` | A реализует решение/контракт B | нет |
| `constrained_by` | на A действует ограничение B | нет |
| `supersedes` | A замещает устаревший B | нет |
| `contradicts` | A противоречит B (сигнал для lint) | да |
| `relates_to` | слабая связь, когда ничто выше не подходит | да |

Обратные рёбра в страницы не пишутся — обратный индекс строит материализация. `relates_to` — резерв последней очереди: lint следит, чтобы его доля не росла (признак дрейфа таксономии).

**Расширение словарей (инвариант №7).** Каждая запись словаря несёт `status: proposed | approved`. Агент может добавить запись-предложение (`proposed`, с обоснованием и примерами применения); валидатор принимает типы и рёбра только против `approved`. До утверждения агент использует ближайший утверждённый тип: для рёбер — обычно `relates_to`, для типов концептов — `insight`. Утверждение (`approved`) и правка утверждённых записей — только коммит владельца; после утверждения сессия-уборщик перетипизирует накопившиеся временные рёбра и страницы.

### 5.4 Формат: markdown + frontmatter, а не чистый YAML

Рассмотрен полный переход страниц на YAML; **отклонён**. Причины:

- машиночитаемая часть **уже** YAML (frontmatter, словари, скелеты) — гибрид и есть правильное разделение: структура в YAML, синтез в markdown;
- тело страницы — свободный синтез; в YAML это многострочные блочные скаляры без внутренней структуры, а markdown-заголовки бесплатно дают чанкинг по разделам и адресное чтение (`--section`);
- надёжность записи: LLM устойчивее генерируют markdown; YAML чувствителен к отступам — вероятность сломать файл при правке (в т.ч. ручной, §8.1) выше;
- диффы markdown читаются построчно — а ревью диффов остаётся единственным человеческим сценарием;
- OKF: концепт = markdown-файл; чистый YAML — отступление от substrate без измеримой выгоды (токенов не экономит).

Чистый YAML уместен там, где содержимое — записи фиксированной схемы, а не синтез; такие сущности в системе уже и так YAML (словари) или отложены (data plane, §10).

## 6. Производный слой: SQLite

Один файл `.local/knowledge/kb.sqlite3` (в `.local/` — по протоколу репозитория для локальных агентных артефактов; файл не версионируется и пересобирается на любой машине).

Набросок схемы (полный DDL — в исполнительной документации):

```sql
CREATE TABLE meta (key TEXT PRIMARY KEY, value TEXT);
-- schema_version, fts_config, last_processed_commit, last_full_rebuild

CREATE TABLE pages (
  id          TEXT PRIMARY KEY,   -- путь без .md относительно бандла, разделитель '/'
  type        TEXT, title TEXT, description TEXT,
  lifecycle   TEXT, timestamp TEXT,
  front_hash  TEXT,               -- sha256 нормализованного frontmatter
  body_hash   TEXT                -- sha256 нормализованного тела
);

CREATE TABLE chunks (
  chunk_id  INTEGER PRIMARY KEY,
  page_id   TEXT NOT NULL REFERENCES pages(id),
  heading   TEXT,                 -- путь заголовков: "H1 > H2 [> H3]"
  position  INTEGER,
  text      TEXT
);

CREATE VIRTUAL TABLE chunks_fts USING fts5(
  text, heading, page_id UNINDEXED,
  content='chunks', content_rowid='chunk_id',
  tokenize='unicode61 remove_diacritics 2',
  prefix='2 3'
);

CREATE TABLE edges (
  src TEXT NOT NULL, type TEXT NOT NULL, dst TEXT NOT NULL,
  provenance TEXT,
  PRIMARY KEY (src, type, dst)
);
CREATE INDEX edges_by_dst ON edges(dst);

CREATE TABLE reconcile_queue (
  page_id     TEXT NOT NULL,
  reason      TEXT NOT NULL,      -- prose-changed | source-drift | dup-candidate | contradiction
                                  -- | invalid-frontmatter | rename-candidate | human-edited | desc-mismatch
  status      TEXT NOT NULL DEFAULT 'open',   -- open | dead
  enqueued_at TEXT,
  attempts    INTEGER DEFAULT 0,
  PRIMARY KEY (page_id, reason)
);

-- фаза 3: узлы карты проекта из .desc.json (§11.1)
CREATE TABLE tree_nodes (
  path        TEXT PRIMARY KEY,
  description TEXT,
  page_id     TEXT               -- накрывающая страница (deepest-match) или NULL
);
```

**Решения:**

- **Нормализация содержимого.** Перед любым хэшированием и записью в индекс содержимое нормализуется: `CRLF`/`CR` → `LF`. В бандле лежит `.gitattributes` (`* text eol=lf`), агенты пишут только LF. Двойная защита: даже если git на одной из платформ подменит переводы строк, хэши не изменятся (инвариант №4; находка №1 ревью основной системы — там autocrlf уже давал фантомные изменения).
- **Чанкинг** — по `##` (H2); раздел длиннее ~2000 символов дорезается по H3. `description` из frontmatter индексируется нулевым чанком страницы. Перекрытия (overlap) нет: границы разделов семантичны, а агент при попадании читает раздел целиком.
- **Токенизатор `unicode61`** (`remove_diacritics 2`) + префиксные индексы `prefix='2 3'`. Замена решения v0.2–0.4 (`trigram`): trigram не сопоставляет токены короче 3 символов, а корпус насыщен короткими терминами (`AI`, `UI`, «БД»); кроме того, фолдинг регистра кириллицы у trigram требует отдельной проверки. `unicode61` корректно фолдит кириллицу и находит короткие токены; идентификаторы с подчёркиваниями (`memory_sync_source`) распадаются на искомые слова; подстрочный поиск компенсируется префиксными индексами и итеративными запросами агента. `trigram` остаётся эволюционной опцией, если замер покажет потребность в подстрочном поиске. Конфигурация фиксируется в `meta`; смена = полная переиндексация (инвариант №3).
- **Граф** — таблица рёбер + рекурсивные CTE для обхода на 1–2 шага. NetworkX не подключается, пока не понадобится кластерный анализ в lint (упрощение относительно §10 концепции).
- **Одиночность записи (находка №5 ревью основной системы).** Все пишущие операции (`sync`) выполняются под `BEGIN IMMEDIATE`; если блокировка занята другим процессом — команда сообщает «sync уже выполняется» и завершается с кодом 0 (пропуск безвреден: pull-модель самовосстанавливается). Читающие команды (`query`, `page`, …) работают конкурентно без ограничений.
- **Зависимости CLI:** стандартная библиотека Python (sqlite3, hashlib) + PyYAML. Ничего более.

**Ориентиры производительности** (критерии приёмки, §12): дельта-`sync` < 1 с (включая батчевый обход git, §7.4); полная пересборка < 10 с на сотнях страниц; `query` < 200 мс.

## 7. Операции

### 7.1 CLI `kb` (детерминированные операции)

Размещение: **`scripts/dev/kb.py`** — рядом с существующим dev-инструментарием (`generate-structure.js`); запуск `python scripts/dev/kb.py <команда>` без поднятия Django.

| Команда | Что делает |
|---|---|
| `kb sync --actor {agent\|maintenance} [--session-start] [--full]` | pull-reconciler: материализация изменённых страниц + lint уровня 1; `--session-start` — первый sync сессии (классификация чужих правок, §8); `--full` — снос и пересборка проекции (§8.2) |
| `kb query "<текст>" [--k 10] [--hops 1] [--type T] [--json]` | поиск: FTS5/BM25 по чанкам → агрегация в страницы → расширение соседями по графу; выдача — страницы с релевантными разделами и рёбрами |
| `kb neighbors <id> [--hops N] [--type T]` | окрестность концепта в графе |
| `kb page <id> [--section "<заголовок>"]` | вывод страницы или раздела |
| `kb lint` | контур качества уровня 1 без материализации: валидатор, предупреждения, детекторы (§7.4) |
| `kb queue [--dead] [--done <id> [--reason R]] [--defer <id> --reason R]` | очередь: показ (открытые + счётчики триггеров §7.5); `--done` — снять (без `--reason` — все причины страницы); `--defer` — отложить с инкрементом попыток; 3 неудачи → `dead` |
| `kb eval [--k 3]` | прогон golden set: доля запросов с ожидаемой страницей в топ-k, список промахов (§7.6) |
| `kb log [<id>]` | хронология изменений бандла/страницы из git-истории |
| `kb export structure` | генерация `PROJECT_STRUCTURE.yaml` в текущем формате: канон описаний — `.desc.json`, узлы обогащаются ссылками на накрывающие страницы (§11.1, перспективно) |

### 7.2 query: read-path

1. FTS5 (BM25) по чанкам → топ-20 чанков → агрегация по страницам (лучший чанк = оценка страницы) → до 10 страниц-кандидатов.
2. Обход графа на 1 шаг от кандидатов по структурным рёбрам (`part_of`, `depends_on`, `implements`, `interacts_with`) — соседи добавляются с указанием типа связи.
3. Агент читает **только выбранные** страницы/разделы (`kb page`).
4. Если ответ породил новое знание — агент приземляет его страницей (ingest).

### 7.3 ingest: write-path (протокол агента)

Когда создавать/обновлять страницу:

- завершён значимый блок работы (новый модуль, интеграция, изменение границ) → обновить страницы затронутых концептов и рёбра;
- принято архитектурное решение → страница `pattern`/`constraint` со ссылкой на ADR в `sources` (ADR остаётся каноном решения, страница — его проекцией в граф знаний);
- обнаружен неочевидный факт, стоивший времени → страница `insight`;
- изучен объёмный внешний документ → `source-summary` + рёбра к затронутым концептам.

Процедура: правка markdown → при добавлении страниц обновить `index.md` → `kb sync --actor agent` → устранить ошибки валидатора. Рёбра — только против словаря. Провенанс — `путь[@коммит]`: для ещё не закоммиченных источников допускается форма без `@коммит`, уборщик дозаполняет при следующем проходе. **Коммиты, затрагивающие `docs/knowledge/`, обязаны нести git-трейлер `KB-Actor: agent`** (уборщик — `KB-Actor: maintenance`); отсутствие трейлера = правка человека (§8).

Ограничения: в `schema/*.yaml` агент только **добавляет** записи со `status: proposed` — утверждение и правка утверждённых записей остаются за владельцем (инвариант №7); прямая запись в `kb.sqlite3` запрещена в любом сценарии. **Страницы принадлежат агентам:** переписывание, слияние и разрезание страниц разрешены без согласования — качество защищают git-история, ревью диффов владельцем по его желанию и контур lint, а не неприкосновенность текста. Исключение — свежие правки владельца: их смысл не ревертится (§8.1).

### 7.4 Контур качества, уровень 1: детерминированный lint (при каждом sync)

Выполняется автоматически финальным шагом `kb sync` (и отдельно командой `kb lint`). Без LLM, секунды. Три класса результатов:

- **Ошибки (валидатор):** типы концептов/рёбер вне утверждённых записей словарей; сломанный синтаксис frontmatter; отсутствие обязательных полей. Механической блокировки нет (CLI не пишет в канон, git-хуков нет) — валидатор возвращает **ненулевой код выхода**, а обязанность устранить до завершения задачи — протокольная (AGENTS.md, приложение А). Для правок человека ошибки не блокируют ничего: страница уходит в fail-soft (§8.1).
- **Предупреждения (отчёт):** битые `target` рёбер; страницы, недостижимые из `index.md` (сироты); ссылки на `superseded`-страницы; пустые `description`; дубли `title`; страницы сверх ориентира размера; `index.md` длиннее ~150 строк; доля `relates_to` выше порога; рёбра-утверждения (`contradicts`, `supersedes`) без провенанса; **страницы без единого исходящего ребра** (владелец полноты графа — уборщик, §7.5); совпадение или одноуровневое пересечение зон `covers:` (deepest-match, §5.2).
- **Детекторы (пополняют `reconcile_queue` для уровня 2)** — делятся на два класса (§8.2):

**Класс A — stateless (вычисляются от git и канона; идентично пересчитываются при `--full`):**

- `source-drift` — пути из `sources:`/`covers:` менялись после `timestamp` страницы. Реализация — **батчевая**: один вызов `git log --name-only --since=<минимальный timestamp корпуса>` строит карту «путь → дата последнего коммита» за один проход; сравнение по префиксу пути. Отдельный `git log` на каждый путь запрещён (не укладывается в бюджет дельта-sync < 1 с).
- `human-edited` — правка человека: закоммиченные изменения страницы без трейлера `KB-Actor:` (после `timestamp` страницы — для пересчёта при `--full`; после `last_processed_commit` — в инкрементальном режиме); незакоммиченные изменения, обнаруженные sync с флагом `--session-start` (то есть существовавшие до начала сессии агента). Разбирается уборщиком первой (§7.5).
- `desc-mismatch` — файл `.desc.json`, содержащий запись узла из зоны `covers:`, коммитился после `timestamp` накрывающей страницы, либо у страницы менялся frontmatter при инкрементальном sync: описания узла и страницы могли разойтись; семантическое сравнение — за уборщиком (§11.1).

**Класс B — инкрементальные (best-effort: живут в проекции, при `--full` теряются; компенсация — повторные правки и периодический полный аудит уборщика, §8.2):**

- `prose-changed` — тело страницы изменилось, блок `relations:` — нет (раздельные хэши front/body): объявленные рёбра могли отстать от текста. **Не** взводится для правок актора `maintenance` (уборщик отвечает за консистентность своих правок сам — иначе очередь самовоспроизводится).
- `dup-candidate` — дешёвые кандидаты в дубли: совпадение `title`/`description`, общие рёбра, взаимные попадания в FTS-выдачу. Не взводится для актора `maintenance`.
- `invalid-frontmatter` — страница перестала парситься (типичный след ручной правки): в проекции сохраняется последняя валидная материализация, ошибка идёт в очередь и отчёт — см. fail-soft, §8.1.
- `rename-candidate` — страница исчезла, а новая похожа на неё: точное совпадение `body_hash`, **или** совпадение `title` + `description`, **или** ≥ 60 % общих чанковых хэшей (переименование почти всегда сопровождается правкой, точный хэш один не поймает). До починки входящие рёбра видны предупреждением; обновление `relations:` в ссылающихся страницах — задача уборщика (CLI канон не редактирует).

### 7.5 Контур качества, уровень 2: сессия-уборщик (периодический агентный проход)

Аналог фонового `reconcile`/`lint` базовой концепции (§7, §7.1). Отдельная плановая агентная сессия (`kb sync --actor maintenance`, коммиты с `KB-Actor: maintenance`); вход — `kb queue` и предупреждения уровня 1:

- `human-edited` → **разбирается первым**; правка владельца трактуется как вердикт ревью: её смысл не ревертится и не переформулируется, уборщик достраивает вокруг неё метаданные (рёбра, `index.md`); при содержательном несогласии — эскалация в отчёте владельцу, а не обратная правка;
- `invalid-frontmatter` → починить синтаксис и обязательные поля, сохранив смысл внесённой правки;
- `rename-candidate` → подтвердить переименование: обновить `target` в ссылающихся страницах либо оставить на старом пути заглушку с `supersedes`;
- `desc-mismatch` → сверить описание узла (`.desc.json`) и накрывающей страницы; при противоречии выровнять — обычно правится `.desc.json` как механическая сторона; правка человека не ревертится (§8.1);
- `prose-changed` → перечитать страницу, заново вывести рёбра из текста, обновить `relations:`;
- `source-drift` → сверить страницу с изменившимися источниками; актуализировать (это содержательная сверка — обновить `timestamp`) или пометить `superseded`;
- `dup-candidate` → подтвердить или отклонить; слияние: одна страница поглощает содержимое, вторая становится заглушкой с `supersedes`, **все входящие ссылки обновляются на поглотившую страницу** (обратный индекс это позволяет; заглушка — только для истории и внешних ссылок);
- `contradiction` → проверить пары `contradicts` и искать новые противоречия только на изменённых узлах + их окрестности в графе (O(изменённого), не O(n²) — §7 концепции);
- **полнота графа** (находка №4 ревью основной системы: у эмиссии рёбер должен быть владелец) → разобрать предупреждения «страница без исходящих рёбер»: уборщик — владелец полноты графа; рабочие агенты эмитят рёбра только для страниц, которые сами трогают;
- перетипизация временных `relates_to`/`insight` после утверждения владельцем предложенных записей словарей (§5.3);
- гигиена навигации: обновить `index.md` (с соблюдением лимита размера), разобрать сирот, вычистить битые ссылки, дозаполнить провенанс без `@коммит` (§7.3).

Правила `timestamp` при обслуживании: механические правки (перетипизация, починка ссылок, слияние без пересмотра содержания) `timestamp` **не обновляют**; содержательная сверка с источниками — обновляет (§5.2).

Позиции снимаются `kb queue --done`; нерешаемое откладывается `kb queue --defer` (3 отложенных попытки → `dead`, скрывается из счётчиков и эскалируется владельцу в отчёте уборщика). Правки самого уборщика не взводят детекторы класса B (§7.4) — очередь не самовоспроизводится.

**Триггеры запуска** (стартовые значения, уточняются по факту): открытая очередь ≥ 10 позиций; или ≥ 20 изменённых страниц с прошлого прохода (`kb queue` показывает оба счётчика); и не реже раза в неделю при активной разработке. Запуск — владельцем вручную или планировщиком агентной среды; постоянный демон не нужен. Раз в квартал — полный аудит (проход по всем страницам без фильтра очереди): компенсация потерь инкрементальных детекторов при `--full` (§8.2).

### 7.6 kb eval: фальсифицируемые триггеры качества поиска

Перенос находки №3 ревью основной системы: критерий «вернёмся к X, когда качество поиска упадёт» нефальсифицируем без эталона — долг становится вечным.

- **Golden set** — `docs/knowledge/eval/golden.yaml`: 10–20 пар «запрос → ожидаемые страницы» (растёт до ~50). Часть канона; пополняется агентами при каждом замеченном промахе поиска («не нашёл страницу, которая есть» → добавить пару), владелец видит через диффы.
- **`kb eval`** прогоняет запросы через штатный `query`-конвейер и печатает hit-rate (доля запросов, где хотя бы одна ожидаемая страница в топ-k, k=3) и список промахов.
- **Использование:** критерий приёмки фазы 2 (hit-rate ≥ 0.8); периодический прогон в сессии-уборщике; **триггер векторного индекса** (§10) — hit-rate < 0.8 на двух последовательных плановых прогонах при существующих ожидаемых страницах.

## 8. Консистентность и жизненный цикл

**Pull-reconciler (`kb sync`).** Проекция хранит хэши материализованных страниц; sync сравнивает их с текущим состоянием файлов (content-hash нормализованного содержимого, не `mtime`) и пересобирает только изменённое. Запуск: в начале сессии агента (`--session-start`) и после операций записи. Git-хуки не используются даже как ускоритель — необязательная сложность при таком размере корпуса; пропущенный запуск ничего не ломает (самовосстановление при следующем sync).

**Акторная модель.** Каждое изменение атрибутируется актору: `agent` (рабочий агент), `maintenance` (уборщик), `human` (владелец). Закоммиченные изменения классифицируются по git-трейлеру `KB-Actor:` (его отсутствие = человек); незакоммиченные — актору текущего sync, **кроме** первого sync сессии (`--session-start`): изменения, существовавшие до начала сессии, консервативно считаются человеческими. Это закрывает две петли: ложные `human-edited` на собственные ещё не закоммиченные правки агента и самовоспроизводство очереди правками уборщика.

**Две дорожки пересборки** (§7.1 концепции, сохранены):

1. **Детерминированная — сразу:** чанки, FTS, рёбра из объявленных `relations:` — при каждом sync, без LLM, идемпотентно.
2. **Семантическая — фоном, агентом:** детекторы уровня 1 (§7.4) наполняют `reconcile_queue`; разбирает её сессия-уборщик (§7.5). Основные причины устаревания здесь — не ручные правки человека (их почти нет), а дрейф относительно источников (`source-drift`), накопление дублей и небрежность самих агентов (`prose-changed`).

Отличие от концепции: признак «грязная» живёт **только в SQLite**, а не во frontmatter (`lifecycle: needs-reconcile` не используется) — иначе каждая пометка порождала бы коммит в канон, что нарушает инвариант №6 этого документа (в ревью основной системы та же петля отмечена находкой №2 — здесь она исключена конструктивно). Мягкая деградация сохраняется: в окне до ре-консолидации индекс свеж, устареть могут только объявленные рёбра.

**Ручное редактирование** остаётся возможным (git, любой редактор), но по постановке — редкое событие: владелец читает канон при ревью и ознакомлении, а не поддерживает его. Отдельного человеческого пути записи нет: любая правка (человек или агент) проходит одну и ту же машинерию sync → детекторы → очередь. Ревизионная модель — сам git: история, диффы, откаты, атрибуция актора (трейлер/его отсутствие). Механизм pending/review из §7.2 концепции не переносится — при двух акторах (владелец и его агенты) он не окупается; его роль играют просмотр диффов владельцем по желанию и поведенческие ограничения агента (§7.3).

### 8.1 Отказоустойчивость ручных правок (сценарий «владелец поправил по итогам ревью»)

Гарантии, чтобы правка человека ничего не сломала:

1. **Fail-soft sync.** Ошибка парсинга одной страницы не валит пересборку и не ломает `query`: страница остаётся в проекции в последней валидной версии, ошибка уходит в `reconcile_queue` (`invalid-frontmatter`) и в отчёт следующей агентной сессии. Ошибки валидатора для человека не блокируют ничего: git-хуков нет, человек коммитит как привык.
2. **Ничего запускать не нужно.** Pull-модель: следующий `kb sync` (старт любой агентной сессии) сам подхватит изменения — content-hash, а не события.
3. **Правка человека = вердикт ревью.** Детектор `human-edited` (акторная модель §8: коммиты без трейлера + незакоммиченное на старте сессии) ставит страницу в начало очереди. Уборщик не ревертит и не переформулирует смысл правки — только достраивает метаданные вокруг неё; несогласие эскалируется отчётом. Это точечное возвращение принципа «агент дополняет, а не переписывает» (§7.1 концепции) — ровно для человеческих правок.
4. **Рёбра не отстают молча.** Классический риск «поправил текст, забыл `relations:`» закрыт детектором `prose-changed`; правка обоих сразу — штатный путь без последствий.
5. **Переименования и удаления не каскадят.** `rename-candidate` по хэшу и похожести (§7.4); висячие рёбра — мягкая деградация (предупреждение, а не сбой), починка — рутина уборщика. Пермиссивная модель OKF это допускает.
6. **Переводы строк не рождают фантомов.** Правка в редакторе с CRLF не даёт ложных «изменений»: хэши считаются по нормализованному содержимому, бандл защищён `.gitattributes` (инвариант №4).

### 8.2 Классы состояния и восстановление (DR-матрица)

Перенос находки №6 ревью основной системы: стратегия «проекции сносимы» должна быть записана явно.

| Состояние | Класс | Восстановление |
|---|---|---|
| Бандл `docs/knowledge/` и весь репозиторий (канон) | precious | git (локально + remote); ничего дополнительно не бэкапится |
| `kb.sqlite3`: страницы, чанки, FTS, рёбра, `tree_nodes` | rebuildable | `kb sync --full` — детерминированная пересборка из канона |
| Очередь: причины класса A (`source-drift`, `human-edited`, `desc-mismatch`) | rebuildable | пересчитываются при `--full` от git-истории и `timestamp` страниц |
| Очередь: причины класса B (`prose-changed`, `dup-candidate`, `rename-candidate`, `invalid-frontmatter`) и счётчики `attempts` | best-effort | при `--full` теряются; компенсация — повторные правки взводят их заново + квартальный полный аудит уборщика (§7.5) |

Порядок полного восстановления на новой машине: `git clone` → `kb sync --full --actor agent` → готово. Ничего, кроме git-репозитория, сохранять не требуется.

| Аспект | В концепции | Здесь | Обоснование |
|---|---|---|---|

| Классификация, компартменты, high-water mark, деклассификация | метки + отдельные репо | отсутствуют | нет пользователей с разными допусками |
| Каноны | три: вики, data store, raw sources | один git-репозиторий: бандл = канон знаний, остальные файлы = raw sources | data plane вне задачи; иммутабельность источников даёт git (`путь@коммит`) |

| Человекочитаемость канона | равноправная цель (проза-синтез для людей) | второстепенна: телеграфный структурированный стиль (§5.2) | человек читает канон только при ревью и ознакомлении |
| Индексы (§3) | FTS + вектор, RRF — сразу | FTS5/BM25 на старте, вектор отложен по измеримому триггеру (`kb eval`, §7.6) | малый корпус; агент итеративно переформулирует запросы; дёшево |
| Провенанс рёбер (§6) | обязателен, валидатор отвергает | рекомендован; lint предупреждает для рёбер-утверждений | цена ложного ребра в доверенной среде низка; атрибуция — git blame |
| `lifecycle: needs-reconcile` (§7.1) | флаг в каноне | очередь только в SQLite | последовательное применение инварианта №9 концепции |
| `log.md` (§4) | генерируется из git в файл | `kb log` по запросу, файл не материализуется | избегаем самопорождающихся коммитов; пермиссивная модель OKF это терпит |
| Оркестратор (§1) | отдельный агент ingest/query/lint | рабочий агент + CLI детерминированных операций | агентная среда уже есть; отдельный сервис избыточен |
| Обход графа (§10) | SQLite + NetworkX | SQLite + рекурсивные CTE | на глубинах 1–2 библиотека не нужна |

Отступления от нормативных правил OKF (нематериализованный `log.md`) и выбор хранилища производного слоя оформить ADR при принятии системы — по политике конформности §4 концепции и правилам репозитория.

## 10. Эволюция: что отложено и когда возвращать

| Отложено | Триггер возврата |
|---|---|
| Векторный индекс + RRF (`sqlite-vec`, локальные эмбеддинги, фиксация версии модели) | **измеримо (§7.6):** `kb eval` hit-rate < 0.8 в топ-3 на двух последовательных плановых прогонах при существующих ожидаемых страницах |
| Токенизатор `trigram` (подстрочный поиск) | замеренные промахи поиска идентификаторов, которые не лечатся префиксными индексами и переформулировкой |
| Data plane (append-only SQLite, дескрипторы датасетов `type: Dataset`) | появление рекуррентных структурированных наблюдений — по маршрутизации §3.1 концепции: до этого такие факты живут страницами `insight`, «вики и есть staging». При возврате: материализатор реестра ограничен `CREATE TABLE` и `ADD COLUMN NULL`, деструктивный DDL — только ручной runbook (предохранитель, находка №8 ревью основной системы) |
| Review-gate (pending → review) | появление второго человека-редактора или недоверенного актора |
| MCP-сервер вместо/поверх CLI | необходимость подключить агентов, не умеющих в shell |
| Стабильные ID концептов (immutable `id` + редиректы) | первый болезненный случай массового переименования; до этого — заглушки с `supersedes` и обновление входящих ссылок уборщиком |
| Проекция импортов + детектор `code-drift` (§10.1) | первый случай, когда заявленные рёбра `depends_on` разошлись с фактическими зависимостями кода и это привело к ошибке проектирования |

### 10.1 Анализ: нужен ли граф кода (AST)

Ориентиры практики: карты репозитория по символам (tree-sitter «repo map»), индексы символов IDE/LSP, промышленные кодовые графы (SCIP/LSIF, CodeQL, Glean). Все они — детерминированные проекции кода, то есть инвариантам системы не противоречат (код — канон, граф — пересобираемая проекция). Вопрос не в совместимости, а в окупаемости.

**Полный AST/call-graph — не строим:**

- рабочие агенты уже имеют быстрый структурный доступ к коду (grep/glob, точечное чтение, диагностика IDE) — символьный индекс дублирует их штатные инструменты;
- Python динамичен: статический call-graph неточен, а его поддержка недёшева;
- масштаб кодовой базы умеренный; систематических провалов навигации, которые лечит только граф символов, не зафиксировано;
- плоскости разные: вики отвечает на «почему и как устроено», структуру кода агент всегда возьмёт из самого кода — тот канон и всегда свежее любой проекции.

**Что берём вместо (отложенно, дёшево):** проекция импортов — экстрактор на `ast` из стандартной библиотеки строит таблицу зависимостей уровня приложений/модулей (`code_deps` в том же SQLite). Использование:

1. lint-детектор `code-drift`: заявленное в вики ребро `depends_on` между компонентами сверяется с фактическими импортами их `covers:`-путей; расхождение в обе стороны (заявлено, но не импортируется; импортируется, но не заявлено) → очередь уборщика;
2. подсказка при bootstrap страниц `component` (черновик рёбер `depends_on` по факту импортов — агент подтверждает и типизирует).

Это единственное место, где граф кода реально усиливает граф знаний: он не подменяет знания, а **валидирует** их. Триггер включения — в таблице выше.

## 11. Интеграция с существующими артефактами репозитория

### 11.1 Карта проекта (PROJECT_STRUCTURE.yaml / `.desc.json`): интеграция вместо замещения

Сегодня описания узлов живут в `.desc.json` по директориям, а `scripts/dev/generate-structure.js` собирает из них `PROJECT_STRUCTURE.yaml` — карту проекта для агентов. Пересечение с вики частичное: те же однострочные описания компонентов, но без типизации, связей и синтеза.

**Вытеснение `.desc.json` рассмотрено (v0.3) и отклонено (v0.4).** Причины:

- **Разная гранулярность.** Узел вики = концепт (§5 концепции), узел карты = файл/директория. Вики не должна покрывать каждый файл — иначе она раздувается микространицами и перестаёт быть слоем синтеза. Замещение означало бы либо вечный fallback (два канона навсегда), либо раздутую вики. Карта — **сплошное тонкое** покрытие дерева; вики — **выборочный глубокий** синтез.
- **Локальность как дисциплина.** `.desc.json` рядом с файлами попадает в тот же дифф, что и правка кода: обновить — нулевое усилие, забыть — трудно.
- **Fail-soft.** Карта работает и без `kb`.

**Решение — два канона с разными зонами ответственности:**

1. `.desc.json` — канон **аннотаций файлового дерева**; остаются на своих местах, механизм генерации карты не ломается.
2. Вики — канон **концептного уровня** (синтез, связи, ограничения, «грабли»); привязка к дереву — через `covers:`.
3. **Поглощение в проекцию:** `kb sync` читает `.desc.json` в таблицу узлов дерева (+ FTS) — `kb query` ищет и по концептам, и по узлам карты через единый read-path.
4. **Сверка вместо миграции:** детектор `desc-mismatch` (§7.4, класс A) отслеживает расхождение описаний узла и накрывающей страницы (deepest-match, §5.2); выравнивание — рутина уборщика (§7.5). Правка `.desc.json` человеком проходит машинерию `human-edited` (§8.1).
5. **Карта как мост в вики:** экспорт обогащает узлы, накрытые страницами, ссылками вида `# → concepts/core/integrations-sdk` — агент из карты сразу видит, где лежит синтез. Перспективно `kb export structure` заменяет `generate-structure.js` формат-в-формат; канон описаний (`.desc.json`) при этом не меняется.

**Отклонённая альтернатива** — генерировать `.desc.json` из вики как проекцию в рабочее дерево: JSON не поддерживает комментарии, файл нельзя пометить «сгенерировано — не править», ручные правки неизбежно затирались бы экспортом; плюс это коммит производного состояния в репозиторий — против духа инварианта №6.

**Принятая цена:** однострочные описания дублируются для узлов с 1:1 страницей (`apps/inventory/` ↔ страница `component`); расхождения контролирует `desc-mismatch`.

### 11.2 Что теряет актуальность при внедрении вики

| Артефакт / механизм | Роль сейчас | Судьба |
|---|---|---|
| `.desc.json` по директориям | канон описаний узлов | **остаются каноном** аннотаций файлового дерева; поглощаются проекцией и сверяются с вики через `desc-mismatch` (§11.1) |
| `scripts/dev/generate-structure.js`, цель `gen-struct` | генератор карты | остаются; перспективная замена — `kb export structure` формат-в-формат с обогащением узлов ссылками на страницы (§11.1) |
| `PROJECT_STRUCTURE.yaml` | карта проекта, источник истины по структуре | остаётся картой дерева (канон — `.desc.json`); обогащается ссылками-мостами на страницы вики; концептный уровень — в вики |
| `README.md`, разделы «Текущее состояние» и «Архитектура» | входная карта для людей | сжимаются до назначения, быстрого старта и указателей (по образцу корневого `BACKLOG.md`-указателя); «как устроено сейчас» живёт в вики |
| `docs/architecture/`: ARCHITECTURE.md, DOMAIN_MODEL.md, INTEGRATIONS.md, MEMORY_MVP_CURRENT_STATE.md | описания текущего устройства | содержимое поглощается страницами `domain`/`component`/`integration`; файлы — заглушки-указатели или архив; до миграции — raw sources для bootstrap |
| `docs/architecture/*_PLAN.md`, `*_REVIEW*.md`, blueprint.md | замыслы и обзоры на момент написания | остаются raw sources (историческая ценность); вики синтезирует из них и ссылается |
| `archive/PROJECT_HANDOFF.md` | исторический обзор | уже архив; остаётся raw source |
| AGENTS.md, раздел «Sources Of Truth» | список источников истины | обновляется: «как устроен проект сейчас» → вики; протокол `kb` → приложение А |
| ADR, `docs/planning/`, `workflow/` | решения, планы, след исполнения | **не устаревают:** ADR — канон решений (вики проецирует их рёбрами `implements`/`sources`), планирование и workflow — не знания |
| `docs/guides/` | операционные инструкции для людей | остаются; вики ссылается на них |

**Важное разграничение.** Продуктовая система памяти (`apps.memory`, `data/knowledge_repo/`, PostgreSQL-индексы, memory_* команды) — функциональность продукта для его пользователей и **не затрагивается**: проектная вики — инструмент разработки с отдельным каноном `docs/knowledge/` и проекцией в `.local/`. Сливать контуры нельзя — разные потребители, жизненные циклы и требования (у продуктовой памяти есть privacy pipeline, аудит и своя модель доступа).

## 12. План внедрения

Детальная исполнительная спецификация (алгоритмы, DDL, форматы файлов, задачи с проверками — рассчитана на исполнителя со слабой LLM): [docs/planning/active/project-knowledge-system-mvp.md](../planning/active/project-knowledge-system-mvp.md).

**Фаза 0 — скелет (0,5 дня).** Каталог бандла с `.gitattributes` (LF), `index.md`, `schema/*.yaml` со стартовыми словарями (записи со `status: approved`), `schema/templates/` со скелетами, `eval/golden.yaml` (заготовка). ADR на принятие системы (хранилище проекции, отступления от OKF). Раздел протокола в AGENTS.md (приложение А).

**Фаза 1 — ядро (2–3 дня).** CLI `kb` (`scripts/dev/kb.py`): sync (fail-soft, акторная модель, LF-нормализация, эксклюзивная блокировка), query, neighbors, page, log; lint: валидатор + предупреждения + детекторы ядра (`prose-changed`, `source-drift` батчевый, `human-edited`, `invalid-frontmatter`); queue (составной ключ, `--done`/`--defer`/dead-letter). Unit-тесты по списку исполнительной спецификации; e2e: пересборка с нуля на тестовом бандле → запрос → ожидаемые страницы.

**Фаза 1.5 — расширение (1–2 дня).** Детекторы `dup-candidate`, `rename-candidate` (эвристика похожести), `desc-mismatch`; `kb eval`.

**Фаза 2 — первичное наполнение (итеративно).** Агент-архитектор синтезирует стартовый корпус из README, PROJECT_STRUCTURE.yaml, ADR, docs/architecture, верхнего уровня apps/: **не более 20–30 страниц** (домены, ключевые компоненты, ограничения) с разметкой `covers:` + рёбра — защита от big-bang-наполнения посредственными страницами; остальное растёт органически через `insight`. Golden set: 10–20 пар. Правило: синтез и ссылки, не копирование.

**Фаза 3 — интеграция карты проекта (§11.1).** Поглощение `.desc.json` в проекцию (`tree_nodes` в общей выдаче `query`); обогащение экспорта карты ссылками на накрывающие страницы. Перспективно — `kb export structure` формат-в-формат вместо `generate-structure.js`; канон описаний не меняется.

**Критерии приёмки:**

- `kb sync --full` детерминирован: повторный запуск — ноль изменений; дельта-sync < 1 с (включая батчевый git-обход), полная пересборка < 10 с, query < 200 мс на корпусе фазы 2;
- хэши не зависят от переводов строк: файл с CRLF и с LF даёт один hash (тест);
- `kb eval` на golden set фазы 2: hit-rate ≥ 0.8 в топ-3;
- fail-soft: намеренно битая страница не ломает sync/query и попадает в очередь;
- акторная модель: правка агента внутри сессии **не** попадает в очередь как `human-edited`; незакоммиченная правка, существовавшая до `--session-start`, — попадает (сценарный тест);
- конкурентный `kb sync` корректно пропускается вторым процессом;
- `source-drift` и `desc-mismatch` срабатывают на тестовых изменениях; lint без ложных срабатываний на чистом корпусе;
- фаза 3: узлы из `.desc.json` находятся через `kb query`; если реализован `kb export structure` — карта формат-совместима с текущим генератором.

## 13. Открытые вопросы

- Политика коммитов страниц знаний: вместе с рабочим коммитом задачи или отдельным коммитом с префиксом `kb:` (маркер актора решён — трейлер `KB-Actor:`, §7.3).
- Порог доли `relates_to` и порог размера страницы для lint (стартовые значения — 30 % и 200 строк, уточнить по факту).
- Стартовые пороги триггеров сессии-уборщика (§7.5) и периодичность планового `kb eval` — проверить на практике и скорректировать.
- Язык корпуса: русский (проще редкое ревью владельцем) или английский (экономия токенов при чтении агентами) — решение владельца.
- Момент замены `generate-structure.js` на `kb export structure` (формат-в-формат, канон — `.desc.json`): при реализации фазы 3 или позже, по фактической потребности в обогащении карты.
- Регламент пополнения golden set: только по промахам или также плановые пары на каждую новую страницу типа `domain`/`component`.

## Приложение А. Проект раздела для AGENTS.md (добавляется при принятии)

> ### База проектных знаний (`docs/knowledge/`)
>
> - **Начало сессии:** выполнить `kb sync --actor agent --session-start`, прочитать `docs/knowledge/index.md`. Тела страниц целиком не сканировать.
> - **Перед проектированием или правкой подсистемы:** `kb query "<вопрос>"`, прочитать только выбранные страницы (`kb page`). Если нужная страница существует, но не нашлась поиском, — добавить пару в `docs/knowledge/eval/golden.yaml`.
> - **После значимого блока работы:** создать/обновить страницы затронутых концептов и рёбра (`relations:` — только типы `approved` из `schema/edge-types.yaml`), при добавлении страниц обновить `index.md`, затем `kb sync --actor agent`; ошибки валидатора (ненулевой код выхода) устранить до завершения задачи.
> - **Коммиты:** любой коммит, затрагивающий `docs/knowledge/`, обязан нести трейлер `KB-Actor: agent` (сессия-уборщик — `KB-Actor: maintenance`). Коммит без трейлера считается правкой человека.
> - **Стиль страниц:** телеграфный, списки и таблицы вместо повествования, каждое неочевидное утверждение — с причиной или ссылкой; скелет разделов из `schema/templates/<type>.md`; главный читатель — другой агент.
> - **`timestamp`:** обновлять только при содержательной сверке страницы с её `sources:`/`covers:`; механические правки поле не трогают.
> - **Сессия-уборщик (по триггерам §7.5):** `kb sync --actor maintenance --session-start`; вход — `kb queue` и предупреждения lint; первыми разобрать `human-edited` (правка владельца = вердикт ревью: смысл не ревертить, при несогласии — эскалация в отчёте), затем `invalid-frontmatter` и `rename-candidate`, далее рёбра для `prose-changed`, сверка `source-drift`, дубли, противоречия, полнота графа (страницы без рёбер), `index.md`. Нерешаемое — `kb queue --defer`; слияние страниц обновляет все входящие ссылки. Слияние, разрезание и переписывание агентских страниц разрешены без согласования.
> - **Словари (`schema/*.yaml`):** агент может добавлять записи со `status: proposed` (с обоснованием); эмитить типы и рёбра — только против `approved`; до утверждения использовать ближайший утверждённый тип (`relates_to` / `insight`).
> - **Запрещено:** писать в `kb.sqlite3` напрямую; утверждать записи словарей (`status: approved` ставит только владелец); молча ревертить или переформулировать правки владельца; коммитить в бандл без трейлера `KB-Actor:`; дублировать в страницы содержимое ADR/README вместо ссылки; писать в бандл файлы с CRLF.
