# План реализации MVP (Корпоративный портал ВОБ №3: одна канбан-доска ТО и ремонтов)

## Этап 0. Уточнение доменной модели и правил workflow (0.5-1 день)
- Зафиксировать набор статусов заявки и допустимые переходы.
- Зафиксировать правила по `rating` и `closure_confirmed`.
- Зафиксировать обязательные поля заявки, медицинского изделия и вложений.
- Зафиксировать матрицу прав по ролям:
  - внутренний заказчик;
  - техник;
  - начальник технического отдела.

## Этап 1. Инициализация проекта и инфраструктура (1 день)
- Создать монорепозиторий на базе Django 5.x.
- Настроить Docker Compose.
- Подключить Gunicorn и WhiteNoise для MVP.
- Настроить SQLite как основную БД.
- Организовать структуру папок:
  - `/apps/`
  - `/db/`
  - `/media/`
  - `/static/`
  - `/templates/`
- Подготовить `.env`/секреты и базовые настройки окружений.

## Этап 2. Core и локальная авторизация (1-2 дня)
- Создать `core` приложение.
- Настроить базовый шаблон, общие partial templates и HTMX middleware.
- Реализовать локальную авторизацию Django по логину/паролю.
- Создать стартовые группы и базовые permissions.
- Подготовить seed/management command для начальной настройки ролей и справочников.

## Этап 3. Справочник медицинских изделий (2-3 дня)
- Создать приложение `inventory`.
- Реализовать модель `MedicalDevice` с обязательными учетными и эксплуатационными полями.
- Реализовать CRUD для справочника медицинских изделий.
- Добавить фильтры и поиск по ключевым полям:
  - подразделение;
  - наименование;
  - модель;
  - серийный номер;
  - статус эксплуатации;
  - местоположение.
- Настроить админку и валидацию данных.

## Этап 4. Заявки, workflow и policy layer (3-4 дня)
- Создать приложение `workorders`.
- Реализовать модели:
  - `WorkOrder`
  - `WorkOrderComment`
  - `WorkOrderAttachment`
  - `WorkOrderTransitionLog`
- Реализовать state machine для статусов заявки.
- Реализовать policy layer:
  - видимость заявок;
  - доступность действий;
  - проверки переходов статусов;
  - подтверждение закрытия;
  - работу с `rating`.
- Настроить аудит переходов статуса и ключевых действий.

## Этап 5. UI общей канбан-доски (2-3 дня)
- Реализовать одну общую доску как представление `WorkOrder.status`.
- Сделать карточку заявки и боковую/модальную форму просмотра деталей.
- Реализовать HTMX-действия для:
  - создания заявки;
  - редактирования;
  - смены статуса;
  - комментариев;
  - загрузки вложений;
  - выставления `rating`;
  - подтверждения закрытия.
- Реализовать фильтрацию и группировку по:
  - подразделению;
  - автору;
  - исполнителю;
  - медицинскому изделию;
  - статусу;
  - приоритету.
- Добавить полнотекстовый поиск по заявкам и связанным полям медицинского изделия.

## Этап 6. Вложения и фотографии (1 день)
- Настроить хранение файлов в `media/`.
- Реализовать загрузку вложений и фотографий.
- Добавить превью изображений и список файлов в карточке заявки.
- Ограничить допустимые типы файлов и размеры.

## Этап 7. Базовая аналитика и отчеты (1-2 дня)
- Создать приложение `analytics`.
- Реализовать отчеты через ORM/SQL-агрегации:
  - количество заявок по статусам;
  - среднее время до выполнения и закрытия;
  - распределение по подразделениям;
  - нагрузка по исполнителям;
  - статистика `rating`.
- Ограничить доступ к отчетам для начальника технического отдела.
- DuckDB не делать обязательным в первой поставке; добавить только при подтвержденной необходимости.

## Этап 8. Тестирование и эксплуатационная готовность (1-2 дня)
- Тесты на policy layer и state machine.
- Тесты на видимость заявок и права ролей.
- Тесты на загрузку вложений.
- Smoke-тесты ключевых HTMX-сценариев.
- Настроить backup/restore для SQLite и `media/`.
- Подготовить инструкции по деплою и запуску.

## Этап 9. Этап 2 MVP: LDAP-интеграция (1-2 дня после стабилизации Этапа 1)
- Подключить `django-auth-ldap`.
- Настроить маппинг атрибутов пользователя из AD.
- Проверить совместимость LDAP-login с локальной моделью ролей и policy layer.
- Провести тестирование на стенде заказчика.

## Контрольные результаты по итогу первой поставки
- Работает одна общая канбан-доска.
- Работает локальная авторизация по логину/паролю.
- Есть полноценный справочник медицинских изделий с фильтрами и поиском.
- Есть заявки, комментарии, вложения и фотографии.
- Есть `rating` и подтверждение закрытия.
- Есть фильтрация, группировка и полнотекстовый поиск.
- Есть базовые сводные отчеты.
- LDAP вынесен в следующий этап без блокировки первой поставки.
