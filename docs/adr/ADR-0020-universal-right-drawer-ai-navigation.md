# ADR-0020: Универсальное открытие правого сайдбара ИИ-ботом

## Статус

Proposed

## Дата

2026-05-28

## Контекст

В приложении уже есть повторяющийся пользовательский паттерн: объект из модуля открывается в правом сайдбаре без ухода со страницы.

Сейчас это реализовано локально в модулях:

- `workorders`: карточки канбан-доски открывают `#detail-panel` через HTMX;
- `waiting_list`: строки листа ожидания открывают `#entry-detail-panel` через HTMX;
- будущие модули, скорее всего, будут использовать такой же правый сайдбар для просмотра и работы с объектом.

Параллельно появилась контекстная AI-система:

- левый сайдбар на рабочих страницах показывает встроенный ИИ-чат;
- страница `AI чат` показывает список чат-сессий слева и текущий диалог в центре;
- `PageContextEnvelope` и `ui.get_current_context` уже позволяют ИИ понимать текущий модуль и выбранный объект;
- источники памяти и аналитики подключаются через `SourceAdapter` и `source_code`.

Новая потребность: пользователь может попросить ИИ открыть конкретный объект, например:

- "открой заявку 123";
- "покажи запись листа ожидания по КТ";
- "открой найденную заявку из результатов поиска";
- "на странице AI чат открой эту запись справа".

Если реализовать это прямыми ветками `if source_code == ...` в AI-коде, система быстро потеряет модульность: каждый новый модуль потребует правок в AI runtime, шаблонах и JavaScript. Если же сразу строить универсальную платформу событий, это будет слишком широким решением для текущей задачи.

## Решение

Ввести узкий универсальный слой правого сайдбара:

```text
AI tool / UI click
  -> RightPanelProvider registry
  -> module-owned provider
  -> safe HTMX panel descriptor
  -> global right drawer host
```

Это не полноценная платформа событий и не универсальный CRUD. Это реестр модульных провайдеров, которые умеют открыть существующий модульный фрагмент UI в общем правом контейнере.

### Основные элементы

1. Общий правый drawer-host в базовом шаблоне.

   Он должен быть доступен на всех страницах, включая `AI чат`, и иметь стабильные DOM-идентификаторы:

   - overlay;
   - drawer panel;
   - content target;
   - общий JavaScript API открытия/закрытия.

2. `RightPanelProvider` в `apps.core`.

   Минимальный интерфейс:

   ```python
   class RightPanelProvider(Protocol):
       source_code: str
       object_type: str
       supported_modes: tuple[str, ...]

       def can_open(self, user, object_id: str, mode: str = "view") -> bool: ...
       def build_panel(self, user, object_id: str, mode: str = "view") -> RightPanelDescriptor: ...
   ```

   `RightPanelDescriptor` не содержит произвольный HTML от ИИ. Он содержит только серверно сформированное описание:

   - `source_code`;
   - `object_type`;
   - `object_id`;
   - `mode`;
   - `title`;
   - `htmx_url`;
   - `target`;
   - `swap`;
   - `drawer_size`;
   - `context_hint`;
   - безопасный `PageContextEnvelope` или данные для его обновления.

3. Модульные провайдеры.

   Каждый модуль сам регистрирует provider:

   - `workorders` регистрирует `source_code=workorders`, `object_type=workorder`;
   - `waiting_list` регистрирует `source_code=waiting_list`, `object_type=waiting_list_entry`;
   - будущие модули добавляют свой provider без правок AI-ядра.

   Провайдер использует существующие доменные views, partial templates, selectors и policies. Общий слой не должен знать бизнес-модели.

4. AI tool `ui.open_right_panel`.

   Инструмент принимает:

   ```json
   {
     "source_code": "workorders",
     "object_type": "workorder",
     "object_id": "123",
     "mode": "view"
   }
   ```

   Для MVP `mode=view`.

   Инструмент:

   - ищет provider;
   - проверяет `can_open`;
   - fail-closed при неизвестном provider, объекте или отказе доступа;
   - возвращает UI-команду, которую браузер исполняет;
   - не принимает URL от ИИ;
   - не выполняет запись бизнес-данных.

5. Браузерный обработчик UI-команд.

   Chat frontend должен принимать команду вида:

   ```json
   {
     "type": "open_right_panel",
     "source_code": "workorders",
     "object_type": "workorder",
     "object_id": "123",
     "htmx_url": "/workorders/123/",
     "target": "#global-right-panel-content",
     "swap": "innerHTML",
     "drawer_size": "large"
   }
   ```

   После загрузки содержимого он:

   - открывает общий drawer;
   - выполняет HTMX-загрузку;
   - вызывает `LocalBusinessPageContext.refresh()`, чтобы AI увидел новый выбранный объект.

### Связь с SourceAdapter

`SourceAdapter` не нужно расширять UI-логикой.

Причина: источник для памяти/аналитики и UI-панель имеют разные обязанности:

- `SourceAdapter` формирует нормализованный envelope, текст для поиска, аналитику и проверку доступа к источнику;
- `RightPanelProvider` формирует UI-descriptor для правого сайдбара.

Они могут использовать один `source_code` и один доменный selector/policy, но остаются разными интерфейсами. Это сохраняет чистые границы и не делает memory/analytics слой зависимым от HTML.

### Связь с PageContextEnvelope

Открытие панели должно обновлять текущий UI-контекст:

```json
{
  "schema_version": "1",
  "page": {
    "module": "workorders",
    "view": "detail"
  },
  "selection": {
    "source_code": "workorders",
    "object_type": "workorder",
    "object_id": "123"
  },
  "ui_state": {
    "right_drawer": "open",
    "focused_region": "global_right_panel"
  }
}
```

Сервер, как и сейчас, не доверяет клиентскому объекту и заново проверяет доступ через resolver/provider.

## Альтернативы

### Оставить локальные drawer-реализации без общего слоя

Плюсы:

- минимум изменений;
- текущие модули продолжают работать как есть.

Минусы:

- ИИ не сможет открывать объекты единообразно;
- страница `AI чат` не сможет открыть правую карточку без знания конкретного модуля;
- каждый новый модуль потребует отдельной интеграции с AI runtime.

### Добавить прямые AI-tools по модулям

Пример: `workorders.open_drawer`, `waiting_list.open_drawer`.

Плюсы:

- быстро для двух модулей;
- простая трассировка.

Минусы:

- дублирует один и тот же UI-паттерн;
- раздувает tool registry;
- заставляет AI runtime знать все модули;
- хуже масштабируется для удаляемых и добавляемых модулей.

### Использовать SourceAdapter как UI-provider

Плюсы:

- уже есть `source_code`, `object_type`, `can_access`;
- меньше новых сущностей.

Минусы:

- смешивает память/аналитику и UI;
- заставляет адаптеры файлов/API знать про Django-шаблоны;
- усложняет поддержку headless источников, у которых нет UI-панели.

### Ввести универсальную платформу событий

Плюсы:

- потенциально решает activity stream, outbox, подписки, аудит UI-команд и интеграции;
- может пригодиться для future workflow automation.

Минусы:

- избыточно для открытия правого сайдбара;
- добавляет надежную доставку, хранение событий, подписчиков, ретраи и схемы событий до появления реальной потребности;
- увеличивает риск архитектурного расползания.

## Последствия

Положительные:

- ИИ сможет открывать объекты из разных модулей одинаково;
- `AI чат` сможет работать с правой панелью так же, как рабочие страницы;
- новые модули подключаются через provider, без правок AI-ядра;
- общий drawer-host уменьшает дублирование DOM/JS;
- права остаются в доменных модулях.

Ограничения:

- MVP поддерживает только `view`;
- формы редактирования, создание и переходы статусов остаются отдельными инструментами с подтверждением;
- локальные drawer-функции модулей нужно мигрировать аккуратно, чтобы не сломать существующий UX;
- tool должен вернуть UI-команду в браузер, а не только текстовый ответ модели.

Риски:

- если provider начнет возвращать произвольные URL, появится риск обхода прав. Поэтому URL строится только сервером;
- если общий drawer станет универсальным CRUD, архитектура усложнится. Для MVP это явно запрещено;
- если не обновлять `PageContextEnvelope` после открытия панели, ИИ не будет понимать, какой объект открыт.

## MVP-границы

Входит:

- общий drawer-host в `base.html`;
- `RightPanelProvider` protocol и registry;
- provider для `workorders`;
- provider для `waiting_list`;
- tool `ui.open_right_panel`;
- браузерный обработчик UI-команды;
- e2e: из `AI чат` открыть заявку и запись листа ожидания в правом сайдбаре;
- обновление контекста после открытия.

Не входит:

- универсальный event bus;
- универсальный CRUD;
- редактирование через общий tool;
- открытие произвольных URL;
- поддержка источников без UI-provider;
- автоматическое создание provider из `SourceAdapter`.
