# ADR-0019: Контекстный ИИ-чат в левой боковой панели

Статус: Accepted.

Дата: 2026-05-28.

Принято и реализовано: 2026-05-28.

Фактический scope MVP:

- глобальное меню перенесено в выпадающее меню `Все функции`;
- левая панель занята встроенным ИИ-чатом с отдельной `sidebar`-сессией;
- на странице `AI чат` левая панель вместо второго встроенного чата показывает список чат-сессий, а центральная область занята текущим диалогом;
- общий контракт `ai.chat_settings` управляет настройками полной страницы и sidebar-чата;
- `PageContextEnvelope`, `AIWindowContextSnapshot`, привязка snapshot к `ChatMessage` и инструмент `ui.get_current_context` реализованы;
- детальная серверная резолюция выбранной сущности реализована для `workorders`;
- для остальных страниц публикуется базовый контекст маршрута `module/view`, а детальные `selection`-resolvers добавляются модульно отдельными доработками;
- полноценная универсальная платформа событий остается отложенным расширением.

## Контекст

В портале уже есть отдельная страница ИИ-чата. Пользователь хочет:

- перенести главное меню из левой боковой панели в выпадающее меню `Все функции` в верхнем левом блоке;
- освободившееся место левой боковой панели занять встроенным ИИ-чатом;
- сделать встроенный чат функционально дублирующим основную страницу чата, но с отдельной рабочей sidebar-сессией;
- передавать ИИ-боту текущий контекст страницы: модуль, выбранные фильтры, открытую карточку в правой панели, текущую сущность и безопасные идентификаторы данных.

Текущий UI построен на Django templates и HTMX. Страница заявок открывает карточку в правой панели через HTMX. AI chat уже имеет серверную модель `ChatSession`, `ChatMessage`, потоковый ответ и JS-клиент `static/src/js/ai_chat.js`.

## Решение

В первом срезе не вводить универсальную платформу событий. Реализовать легкий контур контекста страницы:

```text
server-rendered page
  -> data-ai-context на контейнерах страницы и открытых сущностей
  -> browser context bridge
  -> server-side window context snapshot
  -> sidebar chat sends window_id/context_version + context hint
  -> AI calls ui.get_current_context when an answer depends on the current window
  -> server-side validation/resolution
```

### UI

1. Левый верхний блок получает кнопку `Все функции`.
2. Текущее меню из `templates/base.html` переносится в выпадающий список этой кнопки.
3. Левый сайдбар перестает быть списком навигации и становится контейнером встроенного ИИ-чата.
4. Встроенный чат использует отдельную `sidebar`-сессию пользователя, но те же `ChatSession`, `ChatMessage`, модели, команды, вложения и потоковый ответ, что и отдельная страница чата.
5. Отдельная страница чата остается доступной для полноэкранной работы и истории диалогов. На этой странице левый сайдбар переиспользуется как список чат-сессий, чтобы не показывать второй экземпляр ИИ-чата рядом с текущим диалогом.
6. Избранные функции остаются в верхней панели.
7. Левая панель после замены меню на чат поддерживает режимы `pinned` и `collapsed`.

### Контекст страницы

Ввести `PageContextEnvelope` как JSON-объект, который браузер публикует на сервер при изменении текущего окна. Сообщение чата не должно нести полный контекст каждый раз. Оно передает только:

- `window_id`;
- `context_version`;
- короткий безопасный `context_hint`, например `workorders / workorder#123`.

ИИ-бот получает отдельный read-only инструмент `ui.get_current_context`. Бот вызывает его, когда вопрос зависит от текущей страницы или выбранного объекта: `эта заявка`, `текущая карточка`, `что я сейчас смотрю`, `добавь комментарий сюда`.

Минимальная схема:

```json
{
  "schema_version": "1",
  "window_id": "browser-tab-uuid",
  "context_version": 42,
  "page": {
    "path": "/workorders/",
    "title": "Канбан заявок",
    "module": "workorders",
    "view": "board"
  },
  "selection": {
    "object_type": "workorder",
    "object_id": 123,
    "source_code": "workorders",
    "display": "123. Заменить кабель"
  },
  "filters": {
    "board": "main",
    "department": "ОРИТ",
    "status": "in_progress"
  },
  "ui_state": {
    "right_drawer": "open",
    "focused_region": "detail_panel"
  },
  "capabilities": {
    "can_comment": true,
    "can_transition": true
  }
}
```

Клиент не должен передавать в чат полный HTML, сырые поля карточек, секреты, PII или данные, которые пользователь не должен видеть. `object_id` и `source_code` считаются ссылкой на серверную резолюцию, а не доверенным содержимым.

Сервер обязан:

- валидировать форму envelope;
- проверять права пользователя на выбранный объект;
- при необходимости заново получать безопасную сводку объекта через доменный selector/source adapter;
- отбрасывать неизвестные ключи;
- логировать только безопасный digest контекста и идентификаторы, а не полный пользовательский экран.

Контекст хранится как краткоживущий снимок состояния окна, а не как журнал событий. Рекомендуемое хранение для MVP:

- модель `AIWindowContextSnapshot` в `apps.ai`;
- immutable row на каждую новую версию контекста, если изменился hash sanitized envelope;
- `window_id`;
- `user_id`;
- `context_version`;
- `context_hash`;
- sanitized envelope;
- server-resolved safe summary;
- `is_current`;
- `created_at`;
- `updated_at`;
- `expires_at`;
- индексы `(user, window_id, context_version)`, `(user, window_id, is_current)`, `(expires_at)`;
- TTL, например 24 часа или до logout/session expiry;
- cleanup command/job удаляет expired snapshots.

При отправке сообщения сервер обязан привязать к нему конкретный context snapshot:

- найти snapshot по `user/window_id/context_version`;
- если версия отсутствует или устарела, не подменять ее молча последней версией;
- записать в `ChatMessage.metadata` `context_snapshot_id`, `window_id`, `context_version`, `context_hash`, `context_hint` и безопасную краткую сводку;
- `ui.get_current_context` возвращает snapshot, привязанный к текущему chat request/message, а не просто latest snapshot по `window_id`.

Это защищает сценарий, где пользователь отправил вопрос по заявке A и сразу открыл заявку B. Ответ должен строиться по snapshot A, привязанному к сообщению. Если snapshot недоступен, runtime получает явный `context_unavailable/context_stale` и должен попросить пользователя повторить действие или уточнить объект.

Client-provided `capabilities` не являются источником прав. Они допустимы только как UI-hint. Сервер всегда пересчитывает права через доменные policies/selectors.

Публикация контекста из браузера должна быть ограничена:

- отправлять update только при изменении hash sanitized envelope;
- использовать debounce для фильтров/поиска, ориентир 250-500 ms;
- соблюдать CSRF/session auth;
- ограничить размер payload;
- применять server-side rate limit на endpoint обновления контекста.

### Sidebar-сессия и суммаризация

Встроенный чат использует отдельную `sidebar`-сессию пользователя. Это нужно, чтобы короткие рабочие вопросы из боковой панели не смешивались с полноценными длинными диалогами на отдельной странице.

Механизм:

- для каждого пользователя поддерживается одна активная `sidebar`-сессия или одна активная `sidebar`-сессия на workspace, если позже появятся workspace/project scopes;
- полная страница чата может открыть эту сессию по ссылке `Открыть полный чат`;
- обычные full-page сессии не смешиваются с sidebar-сессией автоматически.
- если открыто несколько вкладок, они могут писать в одну sidebar-сессию, но каждое сообщение хранит свой `window_id/context_snapshot_id/context_hint`; summary должна сохранять смену контекста между сообщениями, чтобы не смешивать смысл диалога.

Историю sidebar-сессии нужно компактировать:

- хранить последние `N` сообщений как обычную историю;
- `N` берется из эффективных настроек `ai.chat_settings` для поверхности `sidebar`, по умолчанию `8`;
- настройка должна валидироваться как integer с разумными границами, ориентир `4..50`;
- старые сообщения суммаризовать в `ChatSession.metadata.sidebar_summary`;
- хранить `summarized_from_message_id`, `summarized_until_message_id`, `source_message_ids`, `summary_version`, `summary_updated_at`;
- запускать compact не только при выходе со страницы, потому что `beforeunload/pagehide` ненадежны;
- основной триггер compact: после каждого нового сообщения, если число несуммаризованных сообщений превысило порог;
- дополнительный best-effort триггер: `navigator.sendBeacon` на `pagehide`/logout для ускорения суммаризации.

Настройки полной страницы чата и встроенного чата должны идти из одного runtime contract, а не из разрозненных параметров. Это снижает риск расхождения поведения и дает одну точку валидации.

Базовый контракт:

- default contract: `contracts/ai/chat_settings.json`;
- runtime override: `data/contracts/ai/chat_settings.json`;
- Settings Center descriptor: `ai.contract.chat_settings`;
- runtime helper: `get_chat_settings(surface="full_page" | "sidebar")`;
- правило разрешения: `defaults` + поверхностные overrides из `surfaces.<surface>`.

Начальная структура:

```json
{
  "schema_version": "1",
  "defaults": {
    "recent_message_limit": 20,
    "summary_enabled": true,
    "summary_trigger_messages": 24,
    "max_prompt_chars": 10000,
    "context_tool_enabled": true
  },
  "surfaces": {
    "full_page": {
      "recent_message_limit": 20,
      "summary_enabled": false
    },
    "sidebar": {
      "recent_message_limit": 8,
      "summary_enabled": true,
      "session_mode": "dedicated",
      "session_switcher": false
    }
  }
}
```

В общий контракт входят параметры, которые одинаково важны для полной страницы и sidebar: лимит последних сообщений, включение суммаризации, порог compact, максимальный размер prompt/input, доступность `ui.get_current_context`, позже - лимиты вложений, trace/audit verbosity, timeout/retry. В surface-specific override остаются только различия интерфейсов: отдельная `sidebar`-сессия, `recent_message_limit=8`, отсутствие переключателя сессий в сайдбаре.

Состояние `pinned/collapsed` левой панели не входит в этот контракт: это локальное UI-состояние, которое хранится в браузере и не должно влиять на серверное поведение чата.

Если LLM runtime недоступен, compact не должен ломать чат. Допустимый fallback: отложить compact и оставить больше последних сообщений до следующей успешной попытки.

`sidebar_summary` считается чувствительным runtime state:

- не отправлять summary в память автоматически;
- не писать raw summary в обычные audit payload/logs;
- доступ к summary только по тем же правилам, что к ChatSession;
- summary не должен содержать секреты и должен проходить тот же masking/safety pass, что chat metadata;
- summary хранит provenance через диапазон/source message ids, чтобы было понятно, какие сообщения были сжаты.

В MVP не нужен отдельный UI для просмотра или ручной перегенерации summary. Summary является внутренним runtime state sidebar-сессии. Для отладки достаточно admin/metadata и trace logs.

Встроенный sidebar не содержит переключатель chat-сессий. Под переключателем понимался компактный selector старых/параллельных sidebar-сессий прямо в боковой панели. Это откладывается: в MVP сайдбар показывает только текущую sidebar-сессию, а просмотр истории и выбор других диалогов остаются в полной странице чата.

### Взаимодействие с памятью и аналитикой

Контекст страницы не заменяет память. Он помогает ИИ-боту выбрать инструмент и начальный объект:

- если открыт `workorder`, бот может использовать `workorders.get`, `workorders.comment`, `workorders.transition`, `workorders.search`;
- если открыта страница памяти, бот выбирает инструменты памяти и ревью;
- если открыта аналитика, бот получает текущий dataset/фильтр и может запускать аналитические диагностики.

Для объектов, подключенных через ADR-0018 source adapters, `source_code` и `object_id` должны совпадать с моделью source adapter. Это позволит повторно использовать access checks и безопасный render объекта.

## Почему не универсальная платформа событий в MVP

Универсальная платформа событий означает отдельный слой для публикации, хранения и доставки событий между модулями:

```text
UI/domain event
  -> event envelope
  -> event bus/store
  -> subscribers: AI, memory, analytics, audit, notifications
```

Например: `workorder.opened`, `workorder.status_changed`, `filter.changed`, `analytics.metric_drilled_down`.

Это полезно, если нужны:

- межмодульная реакция на события без прямых зависимостей;
- надежная доставка и повторная обработка;
- общий журнал пользовательской активности;
- real-time синхронизация между окнами;
- подписки памяти, аналитики, уведомлений и ИИ на один поток событий.

Для текущей задачи это избыточно, потому что:

- нужен только актуальный контекст текущего окна для текущего сообщения в чат;
- событие `карточка открыта` не требует надежной доставки или хранения;
- у проекта уже есть доменные сервисы, audit logs, source adapters и analytics projections;
- полноценный event bus добавит новые контракты, хранилище, правила retention, privacy и отладку.

Решение: в MVP сделать `PageContextEnvelope` и легкий browser context bridge, но проектировать его как будущий `EventEnvelope`-совместимый формат. Если появятся подписчики, надежная доставка или общий activity stream, можно будет расширить этот слой до универсальной платформы событий без переписывания чата.

Важно: `ui.get_current_context` не является event platform. Это read-only tool над последним серверным снимком окна. Он не хранит поток событий, не гарантирует доставку каждого UI-действия и не запускает подписчиков.

## Рассмотренные альтернативы

### 1. Полноценная универсальная платформа событий сразу

Плюсы:

- единый поток событий для ИИ, памяти, аналитики и аудита;
- хорошая основа для real-time и automation;
- слабая связанность модулей.

Минусы:

- существенная сложность до появления нескольких подписчиков;
- нужны правила хранения, очистки, маскирования и прав доступа;
- риск сделать инфраструктуру раньше, чем понятны реальные сценарии.

Решение: отложить.

### 2. Только DOM scraping на клиенте

Плюсы:

- быстро получить видимый текст страницы.

Минусы:

- риск утечек PII/секретов;
- нестабильность при изменении верстки;
- сервер не может доверять такому контексту;
- трудно покрыть тестами.

Решение: отклонить.

### 3. Контекст через URL и query params

Плюсы:

- простая реализация для страниц списков.

Минусы:

- не покрывает открытую HTMX-карточку в правой панели;
- не описывает фокус пользователя и capabilities;
- не подходит для модулей с локальным состоянием.

Решение: использовать только как один из источников `filters`.

### 4. Передавать полный контекст в каждом сообщении

Плюсы:

- самая простая модель исполнения;
- agent runtime сразу видит текущий экран без tool call.

Минусы:

- лишний размер промта;
- риск случайно передать больше данных, чем нужно;
- сложнее контролировать свежесть контекста;
- повторяется один и тот же контекст в каждом сообщении.

Решение: отклонить как основной путь. Передавать только `window_id/context_version/context_hint`, а полный безопасный контекст читать через `ui.get_current_context` по необходимости.

## Последствия

Положительные:

- пользователь получает постоянного ИИ-помощника рядом с рабочим экраном;
- бот понимает текущую карточку и модуль без ручного копирования номера заявки;
- отдельная страница чата и встроенный чат используют одну серверную основу;
- архитектура не вводит тяжелый event bus раньше необходимости.

Ограничения:

- контекст является snapshot, привязанным к сообщению, а не надежным журналом действий;
- `ui.get_current_context` требует server-side storage текущего окна и очистки устаревших записей;
- cross-tab real-time и подписки на события не входят в MVP;
- каждый модуль должен явно публиковать безопасный `data-ai-context`.

Риски:

- если модуль забудет обновить контекст при HTMX-swap, бот будет видеть устаревшую selection;
- если сервер начнет доверять client envelope без проверки, появится риск доступа к чужим данным;
- встраивание полного чата в сайдбар может перегрузить узкую область интерфейса.
- если tool прочитает latest context вместо message-bound snapshot, ответ может относиться к другой карточке.

Меры:

- добавить единый JS context bridge и события `ai-context:update`;
- серверно резолвить выбранный объект по `source_code/object_id`;
- привязывать context snapshot к `ChatMessage` при submit и читать его в `ui.get_current_context`;
- для узкого сайдбара сделать компактный режим сообщений и ссылку `Открыть полный чат`;
- покрыть e2e: открыть заявку в drawer, спросить ИИ про `эту заявку`, сразу открыть другую заявку и проверить, что ответ первого запроса использует snapshot первой заявки.
