# Ускорение тестов без снижения покрытия

Статус: active.

Дата: 2026-06-03.

## Назначение

Документ описывает, как ускорять локальные и контрольные тесты без отказа от важных проверок. Базовая политика тестирования остается в `docs/guides/TESTING_POLICY.md`; этот guide дополняет ее практическими командами и правилами оптимизации.

## Базовый подход

1. Сохранять полный набор тестов как контроль качества.
2. Для локальной обратной связи запускать только затронутый scope.
3. Переиспользовать тестовые базы через штатный Django `--keepdb`, а не писать собственный обход создания баз.
4. Ускорять тяжелую подготовку данных и внешние зависимости, не убирая проверки поведения.

## Команды

Полный прогон с пересозданием баз:

```bash
make test
```

Полный прогон с переиспользованием тестовых баз:

```bash
make test-fast
```

Выборочный прогон по приложению или тесту:

```bash
make test-fast TEST_SCOPE=apps.ai.tests
make test-fast TEST_SCOPE=apps.accounts.tests.LDAPAuthConfigTests.test_ad_group_sync_removes_only_ad_managed_missing_roles
```

Дополнительные Django-флаги:

```bash
make test-fast TEST_FLAGS="--parallel auto"
make test-fast TEST_SCOPE=apps.memory.tests TEST_FLAGS="--verbosity 2"
```

## Тестовые базы

`make test-fast` использует `manage.py test --keepdb`.

Поведение:

- если тестовой базы нет, Django создаст ее;
- если база уже есть, Django переиспользует ее;
- если схема изменилась, Django применит нужные изменения или сообщит о проблеме;
- после завершения база сохраняется для следующего запуска.

Это лучше собственной проверки существования баз, потому что Django учитывает aliases, миграции, test runner и состояние схемы.

## Выборочные тесты

Выборочный запуск допустим для быстрой проверки текущего изменения:

- `apps.accounts.tests` после правки авторизации;
- `apps.ai.tests` после правки AI gateway;
- конкретный `TestCase.test_name` после исправления одного падения.

Выборочный запуск не заменяет полный прогон перед завершением крупной, рискованной или cross-module работы.

## Безопасные способы ускорения

- Использовать `--keepdb` для повторных локальных прогонов.
- Проверить `--parallel auto` на полном наборе и зафиксировать, если все database aliases работают стабильно.
- Переносить неизменяемую подготовку данных из `setUp` в `setUpTestData`.
- Заменять реальные внешние вызовы моками: LLM, OCR, сеть, SMTP/IMAP, API внешних систем.
- Оставлять отдельные integration/e2e smoke-тесты на реальную связку компонентов.
- Убирать лишние sleeps и polling там, где можно проверять состояние напрямую.
- Профилировать самые медленные тесты и исправлять top-10, а не сокращать весь набор.

## Что не делать по умолчанию

- Не отключать миграции глобально для полного контрольного прогона.
- Не делать тесты зависимыми от порядка выполнения.
- Не использовать общую изменяемую фикстуру между тестами без очистки.
- Не заменять integration/e2e проверки только unit-тестами, если проверяется связка компонентов.
- Не коммитить временные тестовые базы, логи и артефакты.

## Рекомендуемый цикл

Для небольшой правки:

```bash
make check
make test-fast TEST_SCOPE=<affected-tests>
make contracts
```

Для крупной или рискованной правки:

```bash
make check
make test-fast
make contracts
```

Если изменение затрагивает UI, runtime, contracts, права доступа, память или deployment, добавить соответствующий integration/e2e сценарий по `docs/guides/TESTING_POLICY.md`.
