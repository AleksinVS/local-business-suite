# ADR-0031: Единый слой доступа к runtime-контрактам и их доставка в agent-runtime

## Статус

Accepted (принят 2026-07-05 с уточнениями реализации по итогам обсуждения с владельцем)

## Дата

2026-07-04; уточнения и принятие — 2026-07-05.

## Контекст

Проект хранит бизнес-правила в JSON-контрактах: дефолты в `contracts/` (git), рабочие копии в
`data/contracts/` (runtime, редактируются через Settings Center). Архитектурное ревью 2026-07-04
(`docs/guides/ARCHITECTURE_REVIEW_2026-07-04.md`) зафиксировало четыре дефекта текущей схемы:

1. **Межпроцессная несогласованность чтения.** Контракты загружаются в константы `settings.*`
   один раз при импорте `config/settings.py`. После записи контракта обновление в памяти
   (`_refresh_inprocess_setting`, прямые присваивания `settings.LOCAL_BUSINESS_ROLE_RULES`)
   действует только в воркере, обработавшем запрос. Production работает с несколькими
   gunicorn-воркерами, поэтому изменение прав ролей применяется частично — часть запросов
   авторизуется по старым правилам до перезапуска.
2. **Два пути записи одного контракта.** `apps/core/views.RoleRulesUpdateView` пишет
   `role_rules.json` напрямую без валидации и без audit-записи `SettingsChange`, тогда как
   Settings Center и AI-инструмент используют `apply_contract_payload` с валидацией и аудитом.
3. **Agent-runtime не получает рабочие контракты.** `services/agent_runtime/config.py` читает
   `data/contracts/` с локального диска, но в Docker-конфигурациях том `./data` в контейнер
   agent-runtime не смонтирован — runtime молча использует дефолты, запечённые в образ.
4. **Нет стратегии обновления рантайм-копий.** `get_contract_path` копирует дефолт в
   `data/contracts/` один раз; последующие изменения дефолтов в git не достигают работающих
   установок и не диагностируются (дрейф проверяется только для AI tools).

Ключевая идея решения: перестать доставлять изменения в процессы (push) и начать читать из
одного места по требованию (pull) с дешёвой проверкой актуальности.

## Решение

1. **Единая точка чтения — contract store.** Вводится модуль `apps.core.contract_store`
   (рабочее название) с функциями вида `get_contract(name)`:
   - читает рабочую копию из `data/contracts/` при обращении;
   - кэширует разобранный payload в памяти процесса; ключ инвалидации — кортеж
     `(st_mtime_ns, st_size, st_ino)`, а не голый mtime: атомарная запись через
     `os.replace` меняет inode, поэтому кортеж надёжен и на файловых системах с грубым
     разрешением mtime (один системный вызов `stat` на обращение — приемлемая цена);
   - возвращает **неизменяемый** payload (глубокая копия либо рекурсивная
     read-only обёртка — вариант выбирается при реализации и фиксируется тестом):
     мутация результата вызывающим кодом не должна портить кэш процесса;
   - семантика ошибок разделена: если валидного payload в процессе ещё нет (первое
     чтение после старта) — ошибка чтения/валидации поднимается наверх (fail-fast,
     как сейчас при старте); если валидный payload уже был — store возвращает последний
     валидный, пишет ERROR в лог и выставляет **видимый сигнал деградации**
     (флаг в health check `/health/` и индикатор в Settings Center), чтобы тихий откат
     не маскировал проблему;
   - для авторизационных решений внутри одного HTTP-запроса используется снимок:
     контракт читается один раз на запрос и передаётся дальше, чтобы запрос не увидел
     две версии правил.
   Прямые чтения `settings.LOCAL_BUSINESS_ROLE_RULES` и других payload-констант из
   `settings` постепенно заменяются на contract store. Константы путей
   (`LOCAL_BUSINESS_*_FILE`, включая env-override) остаются в settings и уважаются store.
2. **Единственный путь записи — Settings Center service layer.** Любая запись рабочего
   контракта идёт через `apps.settings_center.contract_services.apply_contract_payload`
   (валидация, атомарная запись, audit `SettingsChange`). Прямые записи
   (`apps/core/views.RoleRulesUpdateView`) переводятся на этот слой.
   `_refresh_inprocess_setting` упраздняется: согласованность обеспечивает contract store.
   Против потерянного обновления при конкурентном редактировании вводится оптимистическая
   проверка: записывающий передаёт хеш исходной версии контракта; если файл уже изменился —
   запись отклоняется с предложением перечитать.
3. **Доставка контрактов в agent-runtime — двумя шагами.**
   - Шаг 1 (минимальный, в рамках remediation-блока): том `./data` монтируется в контейнер
     agent-runtime в режиме read-only в `docker-compose.yml` и `docker-compose.prod.yml`.
     Agent-runtime **перечитывает контракты по тому же инвалидационному ключу**
     `(st_mtime_ns, st_size, st_ino)`, а не только на старте: при первом старте compose
     рабочие копии создаёт Django, и агент может подняться раньше их появления
     (гонка первого старта). При недоступности рабочих копий агент работает на дефолтах
     `contracts/` и пишет предупреждение уровня WARNING с фактическим источником.
   - Шаг 2 (целевой, отдельная задача): agent-runtime получает контракты через существующий
     Django gateway (единственный источник), с кэшем и ETag/hash. Переход на шаг 2 оформляется
     обновлением этого ADR.
4. **Диагностика дрейфа default↔runtime.** `validate_architecture_contracts` дополняется
   отчётом о расхождениях между `contracts/` и `data/contracts/` для всех контрактов
   (по аналогии с `validate_ai_tools_drift`): расхождение — не ошибка, а явно видимое
   состояние с перечнем ключей. Отчёт — диагностика, а не решение: перенос обновлённых
   дефолтов в рабочую копию остаётся ручной операцией; механика слияния, если понадобится,
   будет отдельным решением.

## Альтернативы

- **Перенести контракты в PostgreSQL.** Самая серьёзная альтернатива: таблица контрактов
  с версионированием решила бы межпроцессную согласованность, конкурентные записи
  (транзакции) и доставку агенту одним механизмом. Отклонено ради file-first философии
  ADR-0002: git-история и диффы дефолтов, правка контрактов в редакторе, dev-запуск без БД,
  простота бэкапа. **Условие пересмотра:** если правки контрактов станут частыми и
  конкурентными, либо число контрактов существенно вырастет — вернуться к этому варианту.
- **Рассылать обновления по процессам (signal/IPC/Redis pub-sub/LISTEN-NOTIFY/gunicorn HUP).**
  Отклонено: появляется брокер или протокол оповещений ради задачи, которую решает один
  `stat()`; оповещения не покрывают процессы, стартовавшие позже; graceful HUP регулярно
  дёргал бы длинные SSE-стримы AI-чата.
- **Кэш по TTL (перечитывать раз в N секунд).** Проще и не зависит от точности метаданных
  ФС, но даёт фиксированное окно несогласованности. Оставлен как **запасной вариант**:
  если инвалидация по ключу метаданных окажется капризной на целевых ФС, допускается
  TTL 5-10 секунд с фиксацией в этом ADR.
- **Читать файл при каждом обращении без кэша.** Отклонено: парсинг и валидация JSON на
  каждый запрос — лишняя цена; кэш по ключу метаданных даёт ту же согласованность.
- **Передавать контракты агенту в каждом запросе от Django.** Нулевой дрейф по построению,
  но `tools.json`/`task_types.json` раздули бы каждый вызов; разумно только для малых срезов.
- **Смонтировать `./data` в agent-runtime в режиме read-write.** Отклонено: agent-runtime
  не должен иметь права записи в состояние Django-приложения.

## Последствия

Положительные:

- изменения контрактов применяются согласованно во всех воркерах без перезапуска;
- один аудируемый путь записи; невалидный контракт нельзя сохранить через UI;
  конкурентная правка не теряется молча;
- agent-runtime работает с теми же контрактами, что и Django;
- дрейф default↔runtime становится видимым при штатной проверке.

Отрицательные и риски:

- переход на contract store затрагивает много точек чтения — нужен поэтапный перевод
  с тестами по каждому домену (сначала `role_rules`/`workflow_rules` как самые критичные);
- **ограничение среды:** `data/` должна лежать на локальной файловой системе; на NFS/SMB
  атрибуты кэшируются клиентом и инвалидация по метаданным не гарантирована —
  зафиксировать в deployment-документации;
- неизменяемость payload имеет цену (копирование) либо неполноту (поверхностная обёртка) —
  выбранный вариант должен быть закреплён тестом на попытку мутации;
- шаг 1 доставки (mount) сохраняет связанность через файловую систему; это осознанный
  промежуточный компромисс до шага 2 (gateway).

## Реализационные правила

- Работы идут в блоке `workflow/archive/2026/architecture-review-remediation-2026-07-04/`
  (пакеты 01, 02, 08).
- Порядок перевода чтений: `role_rules` и `workflow_rules` → AI-контракты (`tools`,
  `task_types`, `chat_settings`) → memory/analytics контракты.
- До полного перевода допускается сосуществование: contract store для переведённых
  контрактов, `settings.*` для остальных; запрещено добавлять новые чтения payload из
  `settings`.
- Ключ инвалидации — строго `(st_mtime_ns, st_size, st_ino)`; сравнение по одному mtime
  не допускается.
- Unit-тесты обязаны покрывать: инвалидацию кэша по ключу метаданных; неизменяемость
  возвращаемого payload (попытка мутации не влияет на следующее чтение); fail-fast при
  невалидном первом чтении; возврат последнего валидного payload с выставлением сигнала
  деградации при последующей ошибке; отказ записи невалидного payload; отказ записи при
  несовпадении хеша исходной версии; поведение agent-runtime при недоступном
  `data/contracts/` и подхват изменений без перезапуска агента.
