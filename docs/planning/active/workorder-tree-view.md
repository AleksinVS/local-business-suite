# Дерево заявок и режимы просмотра доски

## Статус

MVP реализован и ожидает приемку владельцем.

Архитектурное решение: `docs/adr/ADR-0023-workorder-tree-view-and-customer-branch-access.md`.

Workflow-блок: `workflow/active/workorder-tree-view/`.

## Методологическая заметка

Дерево здесь не должно быть отдельным источником данных. Это альтернативная проекция того же серверного набора заявок, который уже проходит проверку прав и фильтры доски. Такой подход важен для безопасности: пользователь не получает лишние заявки в HTML, а интерфейс только меняет способ просмотра разрешенных данных.

## Цель

Добавить на страницу заявок второй режим просмотра - сворачиваемое дерево по организационной структуре и медизделиям, сохранив текущий стиль доски, правый сайдбар и сценарии создания/редактирования заявок.

## Пользовательская ценность

- Руководитель и инженер видят, где именно накопились заявки: по подразделению, отделению и медизделию.
- Заказчик видит только свою ветку, без лишних отделений.
- Пользователь может быстро переключаться между доской по статусам и деревом по организационному контексту.
- Детальная работа не распадается на разные экраны: заявка открывается в том же правом сайдбаре.

## Область работ

Включено:

- переключатель режимов "Доска / Дерево" на странице заявок;
- серверная проекция дерева поверх `visible_workorders_queryset`;
- ограничение роли "заказчик" веткой `User.department`;
- ограничение списков подразделений и медизделий в формах;
- создание заявки из верхней панели, подразделения или медизделия через правый сайдбар;
- открытие и редактирование заявки через существующий правый сайдбар;
- CSS для дерева в стиле текущей доски;
- unit, view и e2e-проверки.

Не включено:

- отдельная модель `Organization`;
- массовая переработка канбан-доски;
- изменение жизненного цикла статусов;
- drag-and-drop заявок внутри дерева;
- batch-операции по нескольким заявкам;
- виртуализация очень больших деревьев в первом срезе.

## Текущая база

Уже есть:

- `templates/workorders/board.html` - рабочая страница доски;
- `templates/workorders/partials/kanban_column.html` - карточки заявок;
- `templates/workorders/partials/detail_panel.html` - правый сайдбар деталей;
- `WorkOrderBoardView.get_queryset()` - фильтры поиска, статуса, исполнителя, подразделения и изделия;
- `visible_workorders_queryset()` - базовая серверная видимость;
- `Department.parent` - иерархия подразделений;
- `User.department` - привязка пользователя к подразделению;
- `MedicalDevice.department` - привязка изделия к подразделению.

Проблема текущей базы:

- дефолтная роль `customer` сейчас имеет `view_scope: all`;
- `can_edit()` не должен давать действие, если `can_view()` запрещает объект;
- формы создания/редактирования сейчас не ограничивают `department` и `device` веткой заказчика.

## UX-концепция

### Визуальная гипотеза

Спокойная рабочая поверхность в стиле текущей доски: светлая область, плотные строки, аккуратные отступы, те же статусы, приоритеты и правый сайдбар без нового визуального языка.

### План содержимого

1. Верхняя панель: поиск, фильтры, выбор доски, переключатель "Доска / Дерево", действие создания.
2. Рабочая область: канбан-колонки или древовидная таблица.
3. Правая панель: детали выбранной заявки, редактирование, комментарии, вложения и действия по статусу.
4. Пустые состояния: нет доступной ветки, нет заявок по фильтрам, нет медизделий с заявками.

### Принцип взаимодействия

- Переключение режима не сбрасывает фильтры.
- Сворачивание веток быстро меняет обзор без перехода на другие страницы.
- Выбранная заявка подсвечивается в дереве, а детали открываются в сайдбаре.

## Структура дерева

Рекомендуемая MVP-структура:

```text
Организация
  Подразделение: <Department depth 0>
    Отделение: <Department depth 1+>
      Медизделие: <MedicalDevice>
        Заявка: <WorkOrder>
```

Правила:

- "Организация" - виртуальный корневой узел.
- Названия "Подразделение" и "Отделение" в интерфейсе зависят от глубины `Department`; модель данных не меняется.
- Если у заявки нет `device`, она отображается под служебным узлом "Без привязки к медизделию".
- Узлы без заявок в MVP скрываются. Позже можно добавить переключатель "Показывать пустые изделия".
- При поиске дерево автоматически раскрывает предков найденных заявок.
- Счетчики на узлах показывают общее число заявок и число активных заявок.

## Поведение строк

### Узел организации, подразделения и отделения

Показывает:

- шеврон раскрытия;
- иконку уровня;
- название;
- счетчик заявок;
- компактный набор индикаторов по статусам;
- действие создания заявки, если пользователь имеет право.

Основное действие:

- раскрыть или свернуть ветку.

Дополнительное действие:

- "Создать заявку в этой ветке" открывает правый сайдбар с предзаполненным `department`.

### Узел медизделия

Показывает:

- название изделия;
- серийный или инвентарный номер;
- статус эксплуатации, если доступен;
- счетчик заявок;
- действие создания заявки для этого изделия.

Основное действие:

- раскрыть список заявок по изделию.

Дополнительное действие:

- создать заявку с предзаполненными `department` и `device`.

### Строка заявки

Показывает компактный вариант текущей карточки:

- номер и заголовок;
- короткое описание в одну-две строки;
- статус;
- приоритет;
- исполнитель;
- дата обновления;
- индикаторы комментариев и вложений.

Основное действие:

- открыть правый сайдбар деталей.

Редактирование:

- выполняется из сайдбара через существующую кнопку редактирования.

## Переключатель режимов

Рекомендуемый вид: сегментированный переключатель рядом с фильтрами.

Состояния:

- `Доска` - текущий канбан;
- `Дерево` - новая древовидная таблица.

Технические правила:

- выбранный режим хранится в query string: `view=board` или `view=tree`;
- ссылки переключателя сохраняют `q`, `department`, `status`, `assignee`, `device` и текущую доску;
- HTMX-обновления меняют только рабочую область, а не весь layout;
- `data-ai-context` получает `page.view = "tree"` в режиме дерева.

## Правая панель

Нужно сохранить текущий подход:

- `openKanbanDrawer()` можно переименовать или обобщить до `openWorkorderDrawer()`;
- `#detail-panel-content` остается целью HTMX;
- `detail_panel.html`, `workorder_form_partial.html`, `workorder_edit_form_partial.html` переиспользуются;
- после успешного создания из дерева возврат должен сохранить `view=tree`;
- после редактирования в сайдбаре нужно обновить дерево или выбранную строку.

Для создания из узла дерева:

```text
GET /workorders/new/?board=<id>&department=<id>&device=<id>&return_view=tree
```

Сервер обязан игнорировать недоступные `department` и `device`, даже если они пришли в query string.

## Доступ заказчика

Нужное правило:

- заказчик видит заявки только в `user.department` и потомках этого подразделения;
- заказчик может выбрать в форме только доступные подразделения и изделия;
- прямой URL чужой заявки возвращает `404` или `403` по текущей политике проекта;
- если `user.department` не заполнен, показывается пустое состояние и действие обращения к администратору.

Рекомендуемая реализация:

- добавить scope `department_branch`;
- перевести `contracts/role_rules.json` для `customer.view_scope` на `department_branch`;
- добавить helper для доступных подразделений пользователя;
- использовать helper в `visible_workorders_queryset`, формах и фильтрах;
- обновить тесты, где сейчас ожидается, что заказчик видит все.

## Архитектура реализации

Рекомендуемые файлы:

```text
apps/workorders/tree.py
apps/workorders/selectors.py
apps/workorders/policies.py
apps/workorders/forms.py
apps/workorders/views.py
templates/workorders/board.html
templates/workorders/partials/workorder_view_switcher.html
templates/workorders/partials/tree_view.html
templates/workorders/partials/tree_row.html
static/src/css/app.css
static/src/js/workorder_tree.js
apps/workorders/tests.py
scripts/e2e/tests/workorder_tree.spec.ts
```

Принципы:

- дерево строить в отдельном модуле, например `apps.workorders.tree`, чтобы не раздувать `views.py`;
- не дублировать фильтрацию из `WorkOrderBoardView.get_queryset()`;
- дерево должно принимать уже отфильтрованный queryset заявок;
- итоговая структура для шаблона должна быть простым списком видимых строк с `level`, `node_type`, `id`, `parent_id`, `expanded`, `counts`;
- для больших наборов добавить ограничение и понятное сообщение, если фильтр слишком широкий.

## Доступность

Минимум для MVP:

- контейнер дерева имеет понятное имя через `aria-label`;
- строки с потомками имеют `aria-expanded`;
- фокус виден и отличается от выбранной заявки;
- шеврон раскрытия является кнопкой;
- клавиши вверх/вниз перемещают фокус по видимым строкам;
- клавиша вправо раскрывает ветку, влево сворачивает;
- Enter на заявке открывает сайдбар;
- все иконки-действия имеют `aria-label` и tooltip.

## Риски

- Если заказчики уже создавали заявки вне своего подразделения, после изменения доступа они потеряют видимость таких заявок. Перед эксплуатацией нужно проверить данные.
- Если `User.department` не синхронизирован из AD, заказчики увидят пустой набор заявок. Нужен администраторский контроль заполненности.
- При большом числе заявок дерево может стать тяжелым. MVP должен начинать с фильтрованного серверного вывода и иметь ограничение.
- У `Department` нет типа уровня. Названия "подразделение" и "отделение" будут UI-интерпретацией глубины.

## Фактический результат MVP

- Добавлен `department_branch` scope для заявок.
- Дефолтный и текущий runtime-контракт роли `customer` переведен на `department_branch`.
- Формы создания и редактирования заявок ограничивают подразделения и медизделия доступной веткой.
- Добавлен режим `view=tree` на существующей странице заявок.
- Дерево строится серверно в `apps/workorders/tree.py` поверх текущего visible queryset.
- Фильтры, поиск и выбор доски работают с общим контейнером текущего режима.
- Правый сайдбар переиспользуется для открытия, создания и редактирования заявки.
- Добавлена клавиатурная навигация и раскрытие веток через `static/src/js/workorder_tree.js`.
- Добавлен Playwright e2e-сценарий `scripts/e2e/tests/workorder_tree.spec.ts`.
