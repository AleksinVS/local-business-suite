# ADR-0025: Автоупорядочивание файловых источников

## Статус

Accepted, контур заморожен

Решением `docs/adr/ADR-0030-memory-alignment-hybrid-knowledge-v05.md` контур выносится из `apps.memory` в отдельное приложение без функциональных изменений и замораживается до выбора пилотного источника и явного решения владельца.

## Дата

2026-06-02

## Контекст

Система памяти уже поддерживает файловые источники, `MemorySourceObject`, `source_data` поиск, review queue и универсальные адаптеры источников. Текущий контур ingestion подходит для read-only источников: система индексирует файлы, хранит ссылки, хэши и metadata, но не является владельцем физического размещения файлов.

Нужно развить файловый источник в управляемое хранилище с автоупорядочиванием:

- сначала строить виртуальную структуру над существующим хранилищем;
- создать специальный входной каталог для новых файлов;
- после первичного анализа автоматически предложить исходную оптимальную виртуальную структуру;
- накапливать статистику использования виртуальных структур;
- постепенно переносить файлы в физически управляемую структуру только после согласования;
- удалять исходные файлы для экономии места, но только через проверяемый и обратимый процесс;
- заранее предусмотреть переход к S3 или S3-совместимому backend без переписывания доменной модели.

Ключевой риск: если путь остается идентификатором файла, физический перенос превратит тот же файл в новый объект. Это сломает историю, индексы, статистику, согласования и ссылки на источник.

## Решение

Ввести отдельный контур `File Source Auto Organization` в `apps.memory` как развитие файлового ingestion, но не смешивать его с обычным поиском и записью знаний.

### 1. Разделить личность файла, версию, физическое размещение и представления

Относительный путь остается полезным адресом, но не должен быть идентификатором файла.

Целевая модель:

```text
FileObject
  stable file_id
  source
  first_seen_at
  current_version
  current_physical_ref
  lifecycle_status

FileObjectVersion
  file_object
  sha256
  size_bytes
  mtime
  storage_backend
  storage_ref
  version_status

FilePhysicalPlacement
  file_object
  storage_backend
  physical_ref
  relative_path
  path_role
  is_current

FileVirtualPlacement
  file_object
  view
  virtual_path
  placement_source
  confidence
  status
```

`MemorySourceObject` остается проекцией source ingestion, но новый контур должен иметь устойчивую идентичность файла и историю путей.

### 2. Построить исходную оптимальную виртуальную структуру после первичного анализа

После первичного discovery система должна создать `baseline` виртуальную структуру. Она является отправной точкой оптимизации, а не физической истиной.

Baseline строится из:

- текущих относительных путей и имен файлов;
- расширения, MIME type, размера, дат и частоты изменений;
- извлеченного безопасного текста, если он доступен;
- классификации документа: договор, инструкция, отчет, счет, акт, письмо, таблица, изображение, прочее;
- вероятного владельца или подразделения;
- ACL/scope tokens и чувствительности;
- дублей и похожих файлов;
- признаков проекта, периода, контрагента, оборудования, заявки или бизнес-процесса;
- ошибок и конфликтов анализа.

Каждый узел и каждое размещение должны иметь:

- объяснение;
- confidence;
- список evidence;
- список конфликтов;
- признак `auto_generated`;
- возможность ручной правки администратором.

### 3. Входной каталог становится единственным штатным путем записи новых файлов

После включения режима новые файлы должны попадать во входной каталог:

```text
<source>/incoming/
```

Система периодически разбирает входной каталог:

```text
incoming
  -> stable file check
  -> secret/privacy/ACL checks
  -> file identity
  -> baseline or learned virtual placement
  -> review if confidence is low
  -> accepted virtual placement
  -> optional physical placement job
```

Прямую запись в старые рабочие папки нужно постепенно закрывать через права доступа и операционные правила. Иначе виртуальная структура станет только маской поверх растущего хаоса.

### 4. Учиться на популярных виртуальных структурах

Пользовательские виртуальные структуры разрешены и используются как сигнал, но не как автоматическая команда к физическому переносу.

Система собирает агрегированную статистику:

- какие виртуальные пути пользователи выбирают для похожих файлов;
- как часто пользователь переносит файл повторно;
- как быстро файл находится после размещения;
- сколько ручных исправлений делает администратор;
- какие предложенные правила принимаются или отклоняются;
- какие папки устойчивы во времени;
- какие структуры конфликтуют между отделами или ролями.

Предложения общей структуры создаются только при достаточной устойчивости и объяснимости. Персональные структуры нельзя напрямую публиковать в общую структуру без privacy threshold и агрегации.

### 5. Физический перенос включать только для доказанной структуры

Физический перенос выполняется в управляемый корень:

```text
<managed_root>/
  by-function/
  by-department/
  archive/
  quarantine/
```

Безопасная последовательность:

```text
plan
  -> admin approval
  -> copy to managed root
  -> verify sha256 and size
  -> verify ACL policy
  -> update metadata and indexes
  -> mark source placement as migrated
  -> move/delete original into quarantine
  -> purge after retention window and backup checkpoint
```

Удаление исходного файла запрещено до успешной сверки, записи manifest, обновления metadata и прохождения минимального окна удержания или подтвержденного backup/snapshot.

### 6. Подготовить StorageBackend для будущего S3

MVP использует `managed_fs`. S3 или S3-совместимое хранилище не вводится как обязательная зависимость.

Но доменная модель должна зависеть от `StorageBackend`, а не от локального пути:

```text
managed_fs:
  storage_ref = /managed/root/...

s3:
  storage_ref = s3://bucket/blobs/sha256/ab/cd/<sha256>
```

Пользовательские и организационные виртуальные пути хранятся в metadata, а не кодируются как S3 object keys. Это позволяет переименовывать папки без массового copy/delete.

## Рассмотренные альтернативы

### Только виртуальная структура над исходным хранилищем

Плюсы:

- минимальный риск потери данных;
- быстрый старт;
- не требует write-доступа к старому источнику;
- хорошо подходит для сбора статистики.

Минусы:

- физический хаос сохраняется;
- пользователи могут продолжать писать мимо входного каталога;
- резервное копирование, удаление и ручная работа через проводник остаются привязаны к старой структуре;
- виртуальный интерфейс может скрыть реальные дубли и мусор.

Принято как первый этап, но не как конечное состояние.

### Физический перенос сразу

Плюсы:

- быстро убирает физический хаос;
- пользователи видят порядок в обычной файловой папке;
- можно начать экономить место.

Минусы:

- высокий риск ошибок классификации;
- риск изменения ACL;
- риск потери ссылок и пользовательских привычек;
- нужна временная емкость для copy/verify/quarantine.

Отклонено как первый этап. Физический перенос разрешается только после накопленной статистики и согласования.

### S3 сразу

Плюсы:

- хорошо поддерживает объектное хранение, версии, lifecycle и виртуальные структуры;
- удобно хранить объект по content-addressed key;
- проще масштабировать хранение и доступ через API.

Минусы:

- избыточно для первого Windows/UNC deployment;
- нет привычной NTFS-папочной модели;
- нужен новый storage backend, signed URL/API, migration plan и отдельные операционные правила.

Отложено. Архитектура должна быть готова к S3, но MVP начинается с `managed_fs`.

### Оставить относительный путь идентификатором

Плюсы:

- проще для текущего read-only ingestion.

Минусы:

- физический перенос делает тот же файл новым объектом;
- ломается история путей;
- усложняется dedup, rollback, индексация и статистика использования;
- невозможно надежно связать исходный файл, managed copy и будущий S3 object.

Отклонено для режима автоупорядочивания.

## Последствия

Положительные:

- можно начать безопасно с виртуальной структуры и входного каталога;
- стартовая оптимальная структура появляется автоматически после анализа;
- физический перенос опирается на статистику и согласование, а не на одну эвристику;
- пользовательские структуры становятся полезным сигналом, но не ломают общую структуру;
- путь к S3 сохраняется без обязательного внедрения S3 в MVP;
- стабильный `file_id` улучшает историю, индексы, аудит и rollback.

Отрицательные:

- потребуется новый доменный слой моделей и сервисов;
- потребуется изменить path-based identity для файлового auto organization;
- физический перенос потребует строгого worker-контура, manifest и retention;
- статистика пользовательских структур должна учитывать privacy и не раскрывать личные рабочие привычки;
- администратору потребуется review UI для baseline, learned proposals и переносов.
