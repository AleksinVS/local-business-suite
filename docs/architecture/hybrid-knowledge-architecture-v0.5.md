# Гибридная система управления знаниями — отправной архитектурный документ

**Статус:** черновик · **Версия:** 0.7 · **Дата:** 2026-07-07
**Назначение:** зафиксировать концептуальный скелет системы и принятые решения, чтобы важные моменты не потерялись при дальнейшей разработке. Это живой документ; раздел «Открытые вопросы» дорабатывается на последующих итерациях.
**Конвенция именования:** имя файла — стабильный идентификатор документа и при смене версии не меняется (внешние ссылки не рвутся); актуальная версия указывается только в этой шапке, поэтому файл `-v0.5` может содержать версию 0.6 и выше.

**Изменения v0.5 (по итогам архитектурного ревью):**
- зафиксированы модель угроз и точка enforcement: слой доступа — веб-сервис поверх канона (§9.1);
- WeKnora принята как референс, не как платформа (§10.1);
- инвариант источника истины уточнён: три канона — вики, data store, raw sources (§2, §3.1);
- провенанс рёбер привязан к immutable-источнику; флаги `verified` и `confidence` упразднены (§6, §7.1);
- триггер пересборки заменён с git-хука на pull-reconciler (§7.1);
- добавлены процедура деклассификации и политика git-истории (§9.3, §9.4);
- `log.md` генерируется из git, не ведётся руками (§4).

**Изменения v0.6 (по итогам применения концепта к первому проекту, ADR-0030 local-business-suite):**
- дескриптор датасета — концепт-страница `type: Dataset`; реестр датасетов — производная проекция (§3.1);
- жизненный цикл датасета: кандидатство через pending-страницу, только аддитивная эволюция схемы, идемпотентный `capture` (§3.1);
- fail-safe маршрутизация наблюдений: `capture` только по совпадению со схемой; вики как staging для наблюдений-сирот, продвижение по рекуррентности; отдельный staging-слой упразднён (§3.1);
- разделение модальностей поиска: строки данных не индексируются по умолчанию, индексирование строк — opt-in свойство датасета (§3.1);
- единый примитив модерируемой схемы для всех машиночитаемых реестров (§7.3);
- новый инвариант №9: состояние производных слоёв не записывается в канон (§2).

**Изменения v0.7 (по итогам обсуждения паттернов с параллельным kb-проектом):**
- именован паттерн «сверка на записи» (write-time reconciliation): детерминированный проверочный набор перед записью, протокол противоречия, lint как вторая линия той же защиты, ориентация в начале сессии, фальсифицируемость через golden set (§7.4);
- карта решений и открытые вопросы дополнены соответствующими строками.

## 1. Что это за система

Гибридное хранилище знаний, объединяющее три слоя, каждый из которых закрывает слабость двух других:

- **Вики** — markdown-файлы с YAML-frontmatter, которые поддерживает LLM (паттерн Karpathy «LLM-wiki», выровненный под Google OKF). Здесь живёт **смысл**: синтезированные концепты в прозе, человекочитаемые и версионируемые в git.
- **Граф типизированных рёбер** — производный слой для точного обхода связей (соседи, пути, кластеры), которого нет в плоской вики и в RAG.
- **Индексы FTS + вектор** — производный слой для извлечения на масштабе (BM25 + семантика, слияние через RRF).

Над ними — **агент-оркестратор** с тремя операциями: `ingest`, `query`, `lint`.

Доступ пользователей к системе — через **слой доступа** (веб-сервис, §9.1); канон при этом остаётся файловым (markdown + git). Прямой доступ к файлам канона — только у владельца/администратора.

Целевая среда — локальная/offline (регуляторные требования к данным). Наружу, к внешним системам, уходит только анонимизированная проекция (см. §9).

## 2. Ключевые инварианты

Это принципы, которые нельзя нарушать; всё остальное — производное от них.

1. **Три канона, производных два.** Вики — канон knowledge plane (интерпретации); data store — канон data plane (наблюдения); raw sources — канон входов (документы). Граф и индексы — единственные производные слои, полностью пересобираемые из вики. Никогда не наоборот.
2. **LLM пишет в markdown, а не в авторитетный граф-стор.** Иначе появляется два источника истины, которые разъезжаются (правка файлов руками vs устаревший LLM-граф) и теряются git-версионирование, диффы, человекочитаемость.
3. **Граф и индекс — проекции, а не каноны.** Их можно снести и пересобрать в любой момент; результат детерминирован (для векторного индекса — при зафиксированной версии эмбеддинг-модели; смена модели = плановая полная переиндексация).
4. **Change detection по content-hash, а не по `mtime`.** Иначе постоянные ложные переиндексации.
5. **Read-path дисциплина: не сканировать тела страниц ради релевантности.** Порядок: индекс → граф → чтение только выбранных страниц (см. §7).
6. **Контроль доступа должен быть архитектурным, а не прикрученным сбоку.** Синтез по своей природе сливает источники и ломает границы; поэтому разграничение проектируется в модель данных и пути записи/чтения. Enforcement — на слое доступа, defense-in-depth — компартменты-репо (см. §9).
7. **Соответствие OKF — цель, а не жёсткий блокер.** Стремимся держать бандл конформным OKF v0.1. Любое отступление допускается только через отдельное, осознанное и задокументированное архитектурное решение (ADR), а не спонтанно (см. §4).
8. **Данные и знания разделены.** Наблюдения/записи (курсы, метрики, логи) → data store (append-only). Интерпретации/инсайты → вики (проза) с **контрактом деривации** обратно к данным. Числа не «замораживаются» в прозе; знание ссылается на пересчитываемый запрос (см. §3.1).
9. **Состояние производных слоёв не пишется в канон.** Frontmatter несёт присущие концепту метаданные и workflow-флаги (`lifecycle`, `audit`); статусы индексации, версии эмбеддинг-индексов и собственные хэши файла живут в производном слое. Иначе каждая переиндексация «трогает» канон, порождая пустые коммиты, ложные конфликты и паразитные `needs-reconcile`.

## 3. Компоненты и потоки

```
                ┌────────────────────────────────┐
                │   Слой доступа (веб-сервис)    │ ◀── пользователи
                │   authn · фильтрация · аудит   │
                └───────────────┬────────────────┘
                                │
                ┌───────────────▼────────────────┐
                │      Агент / оркестратор       │
                │      ingest · query · lint     │
                └───────┬───────────────▲────────┘
                 writes │               │ retrieves
┌──────────────┐        ▼               │
│ Raw sources  │──▶  Вики (markdown  ──▶  Производный слой (пересобираемый)
│ (immutable)  │     + frontmatter)      ├─ Граф типизированных рёбер
└──────────────┘                         └─ Индексы FTS + вектор

  Data store (append-only) ◀── capture / query ── агент
```

**Raw sources** — курируемые исходники, read-only. Один источник обычно затрагивает много концептов. Канон входов: сюда указывает провенанс рёбер и цитат.

**Вики** — markdown-страница на концепт. Frontmatter несёт машиночитаемые поля и явные типизированные рёбра; тело — проза-синтез. Идентичность концепта — путь файла (соглашение OKF).

**Граф рёбер** — материализуется из **явных** рёбер, которые агент написал в страницы. Узел = концепт = страница. Рёбра типизированы.

**Индексы** — FTS5 (trigram/BM25) + векторные эмбеддинги; гибридная выдача через RRF. Чанкинг на уровне раздела.

**Data store (data plane)** — структурированные наблюдения/записи (курсы валют, метрики, логи взаимодействий), append-only, с временными метками. **Отдельный слой, не вики и не raw sources.** Оркестратор имеет к нему доступ на чтение (`query`) и добавление (`capture`). Знание *о* датасете живёт в вики как OKF-концепт (`type: Dataset`, `resource:` → локация в data store); *строки* данных живут в data store.

**Слой доступа** — веб-сервис, через который работают пользователи (§9.1). Не хранит канон; аутентифицирует, фильтрует retrieval по допуску до попадания в контекст модели, ведёт аудит.

### 3.1 Разделение данных и знаний + контракты

Разные сущности с разными жизненными циклами; смешивать их нельзя.

| | Данные (data plane) | Знания (knowledge plane) |
|---|---|---|
| Что | наблюдения, записи, time-series | интерпретации, инсайты, концепты |
| Форма | структурировано, таблицы | проза + frontmatter |
| Изменение | append-only, высокочастотно | курируемо, версионируется в git |
| Истина | наблюдённый ground truth | фаллибельная интерпретация с провенансом |
| Хранилище | БД (SQLite/DuckDB) | markdown + git |
| Запрос | агрегация/OLAP | семантическое извлечение |

**Граница плоскостей.** Raw sources — **не** data plane. Тест: нужны агрегация/SQL/as-of-запросы (строки, time-series) → data plane; нужна цитата/провенанс документа, из которого синтезировали → raw sources. Итого три канона (см. инвариант №1): raw sources — документы-входы, data store — измерения, вики — интерпретации; производные — только граф и индексы.

Пример: пользователь ежедневно спрашивает курс пары → агент фетчит и **append'ит наблюдение в data store** (`capture`), не в вики. Когда агент синтезирует инсайт («последние 3 месяца пара падает») → это **концепт в вики** с контрактом обратно к данным.

**Правило разделения:** наблюдения → data store (append); интерпретации → вики (с контрактом). Обратно нельзя: инсайт не кладётся в data store, сырые строки не кладутся в прозу.

**Контракт деривации** (frontmatter любого концепта, выведенного из данных):

- `derived_from` — датасет + запрос (лучше именованный, версионируемый в репо) или query ID;
- `as_of` — снимок данных, против которого посчитан инсайт;
- `window` — окно наблюдения (напр. 3 месяца);
- `computed_value` — посчитанная сводка, на которой стоит прозаическое утверждение (напр. тренд −4.2%);
- `refresh` — политика пересчёта/переверификации (на новые данные / по расписанию / по порогу).

Инвариант контракта: **утверждение, выведенное из данных, без пересчитываемого `derived_from` — невалидно** (отвергается валидатором), по аналогии с провенансом рёбер. Это даёт пересчёт и **детекцию устаревания:** когда новые данные материально меняют инсайт (тренд развернулся, порог пройден), фоновая проверка помечает концепт `needs-reconcile` — та же pending/review-машинерия (§7.2). Для чувствительных концептов ре-деривация остаётся pending до человеческого ревью.

**Контракт data store** (что слой гарантирует агенту): именованные датасеты со схемой; append-only или as-of/time-travel запрос; типизированный интерфейс запросов (не свободный скрейп — деривации должны быть воспроизводимы и аудируемы); каждое наблюдение с меткой времени; ретеншн и классификация — на датасет.

**Жизненный цикл датасета.** Дескриптор датасета — концепт-страница вики `type: Dataset`: frontmatter несёт машиночитаемую часть (имя, схема полей, владелец, ретеншн, классификация, политика индексирования строк), проза — назначение и смысл. Реестр датасетов в data store — **производная проекция**, материализуемая reconciler'ом из этих страниц — так же, как граф из `relations:`; у реестра нет собственного канона. Новый датасет проходит стандартный ревью-гейт (§7.2): pending-страница → ревью владельцем данных → merge → материализация. Инициаторы: человек (объявляет заранее), агент (после отклонённого `capture`), фоновая рефлексия (см. маршрутизацию ниже). Эволюция схемы зарегистрированного датасета — **только аддитивная** (новые необязательные поля) и только через ревью; несовместимое изменение = новый датасет (версия) — иначе миграции строк ломают append-only инвариант. `capture` идемпотентен: наблюдение несёт дедуп-ключ (датасет + натуральный ключ/метка времени).

**Fail-safe маршрутизация наблюдений.** Классификацию высказывания («запомни, что курс сегодня был 105» — наблюдение или интерпретация?) предлагает LLM, но **решает валидатор**: запись в data store происходит только если высказывание парсится в схему зарегистрированного датасета; во всех остальных случаях — страница вики. Ошибка в сторону вики дешева и обратима; ошибка в сторону data store невозможна без совпадения схемы — LLM предлагает, валидатор располагает. Отдельный staging-слой не нужен: наблюдения-сироты живут как обычные страницы знаний — **вики и есть staging**. Когда рефлексия (`lint`) замечает серию однотипных фактов, она предлагает датасет; после принятия записи мигрируют, старые страницы помечаются `superseded`. Порог рекуррентности — настраиваемая эвристика рефлексии, не жёсткая константа.

**Разделение модальностей поиска.** Семантический поиск (индексы) отвечает на вопрос «где данные и что они значат»: индексируются страница датасета и инсайты с контрактом деривации, но не строки. Значения достаются типизированным запросом с пересчётом — свежие числа никогда не берутся из «замороженной» прозы. Индексирование строк (FTS по текстовым датасетам: комментарии, логи) — **opt-in свойство конкретного датасета**, фиксируемое в дескрипторе, а не правило по умолчанию: высокочастотный append при сплошном эмбеддинге означает постоянную переиндексацию контента, который никто не ищет семантически, и шум соседей в общем индексе знаний.

## 4. Решение по OKF (Open Knowledge Format)

**Что это.** Открытая спецификация Google Cloud (v0.1, июнь 2026), формализующая паттерн LLM-wiki: бандл = директория markdown-файлов с YAML-frontmatter; концепт = один файл; идентичность концепта = путь файла; единственное обязательное поле — `type`; связи — обычные markdown-ссылки, образующие граф; зарезервированы `index.md` (прогрессивное раскрытие) и `log.md` (хронология изменений). Модель потребления пермиссивна: битые ссылки, неизвестные типы/ключи и отсутствующие `index.md` не отбраковывают бандл.

**Решение: принять OKF как substrate (конвенцию контейнера), но расширять своими ключами.**

Что берём:
- файл = концепт, frontmatter, `index.md`, `log.md` как базовые соглашения;
- совместимость с внешними инструментами (визуализаторы, сторонние OKF-потребители, будущая экосистема);
- `log.md` как готовый аудит-трейл (полезно для регуляторики). **Уточнение v0.5:** `log.md` **генерируется из git-истории**, а не ведётся руками — иначе два аудит-трейла разъезжаются, а сам файл становится конфликтогенным при параллельных правках.

Чего OKF **не** даёт и что мы навешиваем сами (спека это явно разрешает через producer-defined frontmatter-ключи):
- **типизированные рёбра** — в OKF ссылки нетипизированы;
- **контроль доступа** — полностью вне области формата.

Оговорки (учитывать при опоре на конформность):
- OKF даёт **структурную**, но не семантическую совместимость — смысл на стороне производителя;
- v0.1 нестабилен: спека требует одно поле (`type`), а референс-парсер Google отвергает файлы без четырёх (`type`, `title`, `description`, `timestamp`). Конформность ≠ гарантия;
- это внутренний формат для своих агентов, не веб-стандарт обнаружения; не заменяет schema.org;
- **идентичность = путь файла:** переименование/перемещение меняет ID и рвёт ссылки → стабильность путей закладывать заранее; решение о стабильных ID (immutable `id` + редирект-стабы) — кандидат на первый ADR об отступлении;
- не путать с Open Knowledge Foundation (несвязанная организация).

**Политика конформности.** Соответствие OKF v0.1 — **цель, а не жёсткий блокер.** Добавление producer-defined ключей (типизированные рёбра, метки доступа) конформности не нарушает — спека это прямо разрешает. Нарушением считается отход от нормативных правил: концепт = один файл, обязательный `type`, структура зарезервированных файлов (`index.md`/`log.md`), путь = идентичность. Любое такое отступление оформляется **отдельным ADR:** что именно из OKF нарушается, зачем это оправдано, и как деградируют внешние OKF-потребители. Спонтанные отступления не допускаются; при равных прочих выбираем конформный вариант.

## 5. Решение по гранулярности графа

Разносим гранулярность **графа** и **извлечения** на два уровня:

- **Узел графа = концепт = страница** (уровень OKF-концепта). Здесь живут типизированные рёбра и — на следующем этапе — метки классификации. Это естественная единица и для синтеза, и для контроля доступа.
- **Извлечение = уровень раздела** (heading/TOC). Чанкинг и эмбеддинг по заголовкам, чтобы в контекст читался только релевантный раздел, а не вся страница (существенная экономия токенов). Идентичность узла графа при этом остаётся страничной.
- **Уровень утверждения/триплета** — только точечно, где нужна машинно-проверяемая детекция противоречий; как inline-аннотация/frontmatter, а не как отдельные узлы (иначе граф расползётся).

Почему не грубее (весь источник как узел): источник затрагивает много концептов → узел это концепт, а источник — отдельный концепт-сводка, ссылающийся на порождённые концепты. Почему не тоньше по умолчанию: стоимость поддержки и дрейф таксономии/рёбер растут нелинейно; синтез LLM лучше всего работает на концептовой гранулярности.

## 6. Владение и извлечение рёбер

**Порождение рёбер выполняет оркестратор в момент синтеза, но приземляет их в markdown, а не в отдельный граф-стор.** Типизация ребра — семантический акт: в момент записи агент знает, *почему* связывает A и B, и проставляет тип, направление, квалификатор и провенанс. Пост-фактум парсер видит только голую ссылку и восстановить тип не может без повторного LLM-прохода. Значит, экстракция как отдельный инференс либо даёт нетипизированный граф, либо платит дважды — этого избегаем.

Пайплайн рёбер трёхстадийный:

| Стадия | Кто/как | Что делает |
|---|---|---|
| **write** | LLM, семантика | агент эмитит типизированные рёбра с провенансом прямо в страницу (typed-ссылки или блок `relations:` в frontmatter), против контролируемого словаря типов |
| **materialize** | детерминированно, дёшево | парсер строит граф-стор из явных рёбер; без LLM, воспроизводимо |
| **reconcile** | периодически, глобально | lint по всему графу: дедуп узлов, детекция противоречий, обратные рёбра, висячие ссылки |

Так стрелка `wiki → graph` становится детерминированной и почти бесплатной, а вики остаётся единственным каноном knowledge plane.

Обязательные подпорки:
- **Контролируемый словарь типов рёбер** (в `CLAUDE.md`) + валидатор. Иначе taxonomy drift: `causes` / `Causes` / `leads_to` / `results_in`. Агент эмитит свободно, но против фиксированного набора; валидатор нормализует/отвергает чужое.
- **Провенанс на ребре** — ссылка на подтверждающее предложение **immutable-источника** (raw source); ловит галлюцинированные связи на аудите. Провенанс указывает только в источник и правкой прозы страницы **не инвалидируется**; сигнал возможного отставания рёбер от изменённой прозы — страничный `lifecycle: needs-reconcile` (§7.1). Отдельные пер-рёберные флаги `verified` и `confidence` упразднены (v0.5): у них не было потребителя, а страничного флага достаточно.

## 7. Три операции агента

**ingest.** Источник → raw → агент читает → пишет/обновляет страницы (одна правка часто трогает 10–15 страниц) с типизированными рёбрами → обновляет `index.md` → триггерит дельта-materialize графа и индексов (по content-hash).

**query.** Read-path дисциплина: (1) гибридный поиск по индексу (BM25 + вектор, RRF) → ~10 кандидатов; (2) обход окрестности в графе добавляет связанные узлы — с фильтрацией по допуску внутри обхода (§9.1); (3) чтение в контекст тел только выбранных страниц; (4) синтез ответа с цитатами, опционально филуется обратно как новая страница.

**lint.** Периодический health-check. Дешёвые проверки (сироты, битые ссылки) — grep/comm по файлам. Дорогие (противоречия, near-duplicate) — только на изменённых узлах + их соседях в графе, чтобы избежать O(n²). Это гигиена графа, а не экстракция; операции принципиально глобальные и недоступны из локального окна записи.

### 7.1 Ручное редактирование и пересборка

Прямое редактирование markdown **разрешено и является частью ценности** (человекочитаемость, git, любой редактор). Запрет («всё только через агента») ломает саму идею, создаёт бутылочное горлышко и единую точку отказа. Основной путь пользователей — слой доступа (§9.1); прямая правка файлов остаётся у владельца канона и продвинутых пользователей с соответствующим доступом. Промежуточный inbox/staging-файл — это отдельный путь **захвата нового знания** (по сути ingest), а не механизм правки существующих страниц.

Пересборка при любом изменении (агент или человек) идёт в **две дорожки**:

1. **Детерминированная материализация — всегда и сразу.** **Pull-модель:** производный слой хранит hash последнего материализованного коммита; reconciler (запускается при старте агента, по таймеру и после операций записи) сравнивает его с текущим HEAD и — с гейтом по content-hash — пересобирает граф-стор из **объявленных** рёбер и переиндексирует изменённые разделы. Без LLM, дёшево, идемпотентно, самовосстанавливается: пропущенное событие ничего не ломает. Git-хук допустим как **необязательный ускоритель** мгновенной свежести, но не как основной механизм (хуки не приезжают при clone, не покрывают merge/rebase/checkout и теряют события при сбое).
2. **Семантическая ре-консолидация — асинхронно, агентом.** Проблема ручной правки: человек меняет **прозу**, но может не обновить блок `relations:` и метки — тогда объявленные рёбра отстают от текста, а детерминированный парсер новую связь в прозе не увидит. Поэтому изменённая страница помечается `lifecycle: needs-reconcile` и попадает в очередь; фоновый проход `reconcile` (часть lint) перечитывает грязные страницы, заново выводит типизированные рёбра из новой прозы, обновляет `relations:` и провенанс, перепроверяет классификацию (high-water mark) и флагует противоречия.

Гардрейлы:

- **Провенанс стабилен, сигнал — страничный.** Провенанс рёбер указывает в immutable-источник и правкой прозы не ломается (§6). Факт «объявленные рёбра могли отстать от прозы» выражается одним страничным `lifecycle: needs-reconcile` до завершения ре-консолидации.
- **Классификация — исключение из eventual consistency.** Из-за high-water mark и риска утечки прямое редактирование **не может молча понизить** классификацию/компартмент. Эти поля меняет только агент/гейт (и автоматически только вверх); до ре-консолидации страница держится на прежнем (или консервативно повышенном) уровне. Понижение — только через процедуру деклассификации (§9.3).
- **Агент дополняет, а не переписывает.** Ре-консолидация обновляет метаданные (рёбра, метки, флаги) и **флагует** семантические проблемы, но не перезаписывает прозу человека автоматически; переписывание прозы — предложение (PR-подобное), а не молчаливый оверрайт. Авторство человека сохраняется.
- **Деградация мягкая.** В окне между правкой и reconcile индекс свеж (детерминирован), устаревать могут только объявленные рёбра — качество `query` падает плавно, не катастрофически. Пермиссивная модель OKF это и предполагает (устаревшие/битые ссылки терпимы).

Итог: **прямое редактирование разрешаем**; консистентность держим двухдорожечной пересборкой (детерминированная — сразу, семантическая — фоном), а inbox/staging оставляем отдельным путём захвата нового знания.

### 7.2 Модель ревизий и review-gate (на основе принципов Wikipedia)

> **Контекст для будущих разработчиков.** Концепции этого раздела заимствованы и адаптированы из механизмов Wikipedia: flagged/pending revisions, protection levels, edit filters (AbuseFilter), patrolling, verifiability, bot-flagging. Социальная обвязка под краудсорсинг (talk-page-консенсус, голосования, антивандализм в масштабе) намеренно **не** переносится — среда доверенная, с двумя классами редакторов (пользователь и агент). Происхождение решений оставлено явно, чтобы контекст не потерялся.

Единый примитив внесения изменений — **propose → pending → review → stable** — симметричный для обеих сторон:

- **Агент ревьюит рутинные правки человека.** Правка → pending → `reconcile` выводит рёбра, проверяет провенанс/классификацию, гоняет валидаторы → авто-принятие, если всё чисто.
- **Человек ревьюит рискованные правки агента.** Удаление контента, переписывание чужой прозы, изменение классификации → человеку на принятие (PR-подобно). Это механизм под инвариант «агент дополняет, а не переписывает» (§7.1).
- **Часть правок авто-принимается** (тривиальные, низкорисковые) — как Wikipedia авто-принимает правки доверенных редакторов.

Порядок «публикация ↔ ревью» переключается **по компартменту:**

- Открытые компартменты — *publish-then-review*: `query` видит текущую версию, `reconcile` догоняет асинхронно.
- Чувствительные компартменты — *review-then-publish*: правка остаётся pending и не попадает в обслуживаемую стабильную версию, пока рецензент не принял; `query` там не видит непроверенный контент. Это и есть механизм «исключения классификации» из §7.1 (непринятое понижение метки не становится стабильной версией).

**Обязательный человеческий аудит чувствительных концептов.** Концепты, помеченные `audit: human-required` (или выводимые из классификации/компартмента), выводятся из авто-принятия: **любая** правка — и от агента (LLM), и от другого пользователя — держится pending до принятия авторизованным человеком. Опционально — сегрегация обязанностей (four-eyes): рецензент ≠ автор правки. Это покрывает требование ручного аудита в обе стороны.

Соответствие механизмов (сжато):

| Wikipedia | Наша система |
|---|---|
| revision history / diff / revert | git |
| стабильная принятая версия / pending | «reconciled» версия / `lifecycle: needs-reconcile` |
| reviewer / patroller | `reconcile` (агент) для рутины; человек для рискованного/чувствительного |
| protection level | политика правки/принятия на компартмент (§9) |
| edit filter | валидаторы (словарь рёбер, схема, метки) |
| bot flag | пометка «автор = агент» на коммите |
| verifiability | провенанс |

Прочее:
- **Конфликты** — оптимистично по content-hash: `reconcile` работает над конкретной ревизией; если страница изменилась — результат устарел, пере-ставим в очередь против новой ревизии.
- **Агент трогает только frontmatter** (`relations:`, `lifecycle`, `audit`), не прозу → его коммиты и правки человека почти не конфликтуют, история чистая.
- **Аудит-трейл** — пер-ревизионная атрибуция git + пометка актора (человек/агент, версия модели); питает security-lint (§9).
- Формат «предложенной» правки агента — **git-ветка + локальный ревью** (PR-подобный процесс без форджа: валидаторы гоняются на ветке как CI-шаг, принятие = merge в main, обслуживаемая версия = main — рабочее дерево не содержит непринятого). Прочие варианты — патч-в-очереди, pending-ref, sidecar — рассмотрены и на текущем этапе отклонены.

### 7.3 Единый примитив модерируемой схемы

Всё, что расширяет **машиночитаемую схему** системы, проходит один и тот же механизм: **предложение (pending-страница) → ревью владельцем → принятие (merge) → детерминированная материализация реестра.** Применения: контролируемый словарь типов рёбер (§6), дескрипторы датасетов (§3.1), в перспективе — типы концептов и профили классификации.

Следствия:

- **LLM никогда не расширяет схему молча:** эмитит только против утверждённого словаря/реестра; всё новое — как кандидат через гейт. Симметрично инварианту «прямое редактирование не может молча понизить классификацию» (§7.1).
- **Реестры не имеют собственного канона:** каждый реестр — производная проекция страниц вики, пересобираемая reconciler'ом; инвариант №1 распространяется на схемы автоматически.
- **Кандидатство переносится с уровня записей на уровень схемы:** записи против утверждённой схемы (строка в датасет, ребро утверждённого типа) автоматичны и ревью не требуют; человеческий контроль стоит там, где он масштабируется, — на изменении схемы, а не каждой записи.
- **Механизм реализуется один раз,** а не по разу на реестр: один и тот же код pending/ревью/материализации обслуживает все словари.

### 7.4 Сверка на записи (write-time reconciliation)

Read-path дисциплина (инвариант №5) экономит контекст точечным чтением, но снижает общую осведомлённость читающего. Компенсирующий паттерн: **несогласованность рождается только в момент записи — поэтому сверка с существующим знанием выполняется на записи**, в самой дешёвой для этого точке (shift-left применительно к знаниям). Точка записи выгодна по трём причинам:

- **пишущий уже несёт контекст задачи** — просмотр нескольких соседних страниц при записи почти бесплатен на фоне загруженного контекста; отдельная «сверка всего со всем» стоила бы на порядки дороже;
- **записи редки относительно чтений** — плата амортизируется: платим один раз на записи, выигрыш в согласованности получают все последующие чтения;
- **эпизодическая осведомлённость превращается в постоянную структуру** — осведомлённость пишущего умирает вместе с сессией, но эмитированные в момент записи рёбра (`depends_on`, `contradicts`, `supersedes`) остаются; широкий контекст пишущего материализуется в граф, и уже граф делает точечные чтения других агентов полнее (шаг 2 read-path, §7 `query`). Компенсация точечного чтения работает не напрямую, а через артефакт.

Составные части:

1. **Детерминированный проверочный набор (impact set), а не совет «посмотри вокруг».** Перед созданием или существенной правкой страницы пишущий получает фиксированный, воспроизводимый список: входящие рёбра (таблица рёбер как обратный индекс — кто зависит от страницы), исходящие `relations:`, топ гибридного поиска по заголовку и началу черновика. Эффективность паттерна равна recall этого набора, поэтому набор — команда/функция с проверяемым выводом, а не добросовестность агента.
2. **Протокол противоречия.** Обнаружив конфликт с существующей страницей, пишущий не правит её молча и не удаляет конфликтующее утверждение, а эмитит ребро `contradicts` с провенансом; разрешение — через ревью-гейт (§7.2). Прямое следствие гардрейла «агент дополняет, а не переписывает» (§7.1).
3. **Вторая линия — lint (§7, §7.1 дорожка 2).** Проверочный набор ограничен recall'ом поиска: конфликт со страницей, лексически непохожей и несмежной в графе, проскочит — и никакой протокол «прочитай связанное» это не чинит, потому что «связанное» определяет тот же поиск. Периодические проверки lint (near-duplicate, противоречия на изменённых узлах и их соседях) — страховка этого же паттерна, ловящая пропуски первой линии, а не независимый механизм.
4. **Ориентация в начале сессии.** Большинство противоречий возникает с недавними решениями; чтение `index.md` и свежего хвоста `log.md` (генерируется из git, §4) за сотни токенов возвращает осведомлённость именно там, где точечное чтение слабее всего.
5. **Фальсифицируемость.** Recall проверочного набора меряется тем же eval-контуром, что и качество `query`: кейс golden set «черновик, похожий на существующую страницу, обязан поднять её» — прямое измерение паттерна вместо декларации.

> **Происхождение** (оставлено явно, как в §7.2): memory-слои агентов (mem0 — поиск похожих воспоминаний перед записью, затем решение add/update/merge/delete); темпоральные графы знаний (Zep/Graphiti — детекция противоречий при ingest, противоречащее ребро инвалидируется меткой времени, а не удаляется); вики-культура (норма «поищи существующую статью перед созданием», патрули как вторая линия); дисциплина ADR (`supersedes` и раздел «связанные решения» принуждают читать предыдущие ADR); blackboard-архитектуры (агенты координируются через общий изменяемый артефакт — согласованность доски есть общесистемное свойство, а не забота одного пишущего).

## 8. Модель данных (набросок)

Страница-концепт (frontmatter + тело).

```yaml
---
type: <тип концепта>          # OKF, обязательное
title: <название>
description: <одна строка>
resource: <канонический URI источника>   # опционально
tags: [<tag>, ...]
timestamp: <ISO 8601>
lifecycle: current            # current | needs-reconcile — грязная после ручной правки (§7.1)
audit: auto                   # auto | human-required — для чувствительных концептов только человек (§7.2)
# --- расширение (producer-defined) ---
relations:                    # типизированные рёбра, эмитятся агентом
  - type: causes              # из контролируемого словаря
    target: concepts/<id>     # concept ID (путь без .md)
    provenance: <raw source + предложение>   # только в immutable-источник (§6)
derived_from:                 # контракт деривации — только для концептов из данных (§3.1)
  dataset: datasets/fx_rates  # OKF-концепт датасета
  query: queries/fx_trend@v2  # именованный версионируемый запрос
  as_of: <ISO 8601>           # снимок данных
  window: P3M                 # окно наблюдения
  computed_value: -4.2%       # сводка, на которой стоит утверждение
  refresh: on-new-data        # on-new-data | schedule | threshold
classification_reason: intrinsic  # intrinsic | inherited — для синтезированных страниц;
                                  # inherited питает очередь кандидатов на деклассификацию (§9.3)
# classification: <...>       # минимальные метки; enforcement — слой доступа (§9.1)
# compartment: <...>          # для наиболее критичного — отдельный репо (§9.2)
---

# Тело: проза-синтез, разбитая на разделы (уровень чанкинга для индекса)
```

## 9. Разграничение доступа

### 9.0 Модель угроз

Защищаемся от: (а) **пользователей с разными допусками** — пользователь не должен получать (напрямую или через синтез) знание из компартментов/уровней выше своего допуска; (б) **утечки через egress** — наружу уходит только де-идентифицированная проекция. Не защищаемся от: владельца/администратора канона и компрометации машины, на которой работает агент, — это вне модели угроз данного этапа.

### 9.1 Точка enforcement — слой доступа (принято, Вариант Б)

Пользователи работают через **веб-сервис поверх канона** (WeKnora — референс UX и функциональности, см. §10.1). Сервис — единственная дверь для пользователей:

- аутентификация и допуски (OIDC — позже, на старте достаточно локальных учёток);
- **retrieval-time authorization** — урезание выдачи индекса и обхода графа по допуску **до** попадания в контекст модели; модель нельзя уговорить показать то, чего ей не дали в контекст;
- аудит доступа/синтеза/egress.

Прямой доступ к файлам канона (git, редактор) — только у владельца/администратора; для остальных пользователей фильтрация по меткам внутри общего репо становится enforceable именно потому, что файлового доступа у них нет.

**Контекст сессии — тоже канал.** Фильтрация retrieval не убирает уже прочитанное из контекста LLM. Правило: сессия агента монтирует фиксированный набор компартментов на старте; всё, что агент пишет в сессии, ложится на уровень самого закрытого из смонтированных (ниже — только через человеческий ревью).

### 9.2 Компартменты

- **По умолчанию** — метки классификации внутри общего репо + фильтрация на слое доступа (§9.1).
- **Defense-in-depth для наиболее критичного** — отдельный git-репозиторий (= отдельный OKF-бандл): физическая граница, свои граф и индексы (никогда не сливаются с общими), монтируется только в допущенные сессии. Запрет кросс-компартментного синтеза становится физическим.
- **Начинаем с двух** (открытый + чувствительный). Новый компартмент — только при появлении реальной границы need-to-know с реально разными допусками, не «на всякий случай»: каждый репо стоит дорого (свои индексы, кросс-ссылки, синхронизация словаря рёбер).
- **Ссылки между компартментами:** между уровнями — только сверху вниз (закрытый может ссылаться на открытое, не наоборот); между горизонтальными need-to-know компартментами — запрещены вовсе.
- Синтез, потребляющий несколько компартментов, приземляет результат в самый закрытый из них (физический high-water mark).

### 9.3 High-water mark и деклассификация

**High-water mark:** синтезированная страница/ребро наследует **максимальную** классификацию всех источников, из которых собрана; автоматическое движение метки — только вверх. Иначе чувствительный факт протекает в менее защищённую страницу через обобщение — главная скрытая дыра паттерна. Метки — и на рёбрах, не только на страницах.

Храповик «только вверх» без обратного хода ведёт к дрейфу всего корпуса к максимальной метке, поэтому нужна **деклассификация** — всегда инициируется и принимается человеком, автоматика только формирует очередь кандидатов:

- Поле `classification_reason: intrinsic | inherited` проставляется при синтезе; `inherited` = «уровень поднят только потому, что один из источников был закрытым».
- `lint` периодически выдаёт список inherited-страниц как кандидатов на пересмотр.
- Две операции (обе human-initiated и human-reviewed):
  1. **Перенос вниз** — метка была унаследована слишком консервативно; страница целиком переезжает в открытый компартмент (+ редирект-стаб).
  2. **Санитизированная производная** — содержимое реально смешанное; оригинал остаётся в закрытом компартменте, агент готовит очищенную версию, человек ревьюит, публикация в открытый компартмент со ссылкой из закрытого на открытый (не наоборот).

### 9.4 Git-история и удаление

История git переживает и повышение классификации, и `git rm` — старые версии остаются доступны всем, у кого есть репо. Политика:

- **Чувствительное рождается в своём компартменте:** классификация назначается на ingest, до первого коммита; миграции между компартментами — редкое исключение, а не рабочий путь.
- Для редких случаев (повышение классификации задним числом, отзыв источника, right-to-erasure) — **документированный runbook на `git filter-repo`** (переписывание истории с пере-клонированием у потребителей). Процедура ручная и аудируемая, не автоматика.

### 9.5 Egress

**Egress/анонимизация — отдельный gate.** Наружу уходит только де-идентифицированная проекция; анонимизация на входе бессмысленна, если синтез на выходе допускает реидентификацию. `restricted` не уходит никогда.

### 9.6 Обязательный человеческий аудит чувствительных концептов

Для помеченных концептов авто-принятие отключено; правки любого актора (человек/агент) держатся pending до принятия авторизованным человеком, опционально с сегрегацией обязанностей (см. §7.2).

### Отложено (сознательно, до реальной потребности)

Движок политик (OPA/Cedar/OpenFGA) и полноценный PDP — не раньше, чем ролевая модель перерастёт возможности слоя доступа. OIDC — при выходе за локальные учётки. Это осознанный анти-overengineering выбор, а не пробел.

## 10. Локальный технологический стек

- **Вики:** markdown + git. `log.md` генерируется из git-истории (§4).
- **Слой доступа:** веб-сервис поверх канона — собственная тонкая реализация или адаптация готового (решение по итогам тестирования, §10.1); идентичность на старте — локальные учётки, позже OIDC.
- **Граф:** таблица рёбер в SQLite + NetworkX для обхода (легко, локально); либо встроенный графовый движок при необходимости.
- **FTS:** SQLite FTS5 (trigram/BM25).
- **Вектор:** `sqlite-vec` / ChromaDB / FAISS; эмбеддинги локальной моделью (Ollama, напр. `mxbai-embed-large`). **Версия эмбеддинг-модели фиксируется в конфиге;** смена модели = плановая полная переиндексация (инвариант №3).
- **Очередь reconcile/pending:** таблица в SQLite (статус, retry-счётчик, dead-letter) — паттерн очереди с DLQ, без выделенной инфраструктуры.
- **Конфиг агента:** `CLAUDE.md` / `AGENTS.md` — workflow ingest/query/lint, контролируемый словарь типов рёбер, соглашения по именам.
- **Доступ агента:** тонкий CLI (shell-аудит) или MCP-сервер.
- **Анонимизация (ingest/egress):** Presidio / ARX / regex-hash.
- **Политики:** отложено (§9, «Отложено»).

### 10.1 Референс WeKnora

**Решение:** WeKnora (Tencent) — **референс, не платформа.** Канон остаётся markdown + git; перенос канона в БД (Postgres/Neo4j/Redis-стек WeKnora) ломает инварианты №1–3 и модель ревизий §7.2, а операционная тяжесть стека противоречит принципу минимальной достаточности. Использование WeKnora как **read-слоя** (UI просмотра/поиска поверх экспортированного бандла) — решить позже по результатам тестирования.

Что заимствуем (паттерны и конвенции, не инфраструктуру):

- **Очередь с DLQ** для ingest/reconcile — как паттерн устойчивости к сбоям (реализация — таблица SQLite, не Redis).
- **Конвенции ингеста:** таблицы источников конвертируются в markdown-таблицы; OCR для сканов; текстовые подписи к изображениям — чтобы весь контент был досягаем для FTS/эмбеддингов.
- **Иерархия вики** (папки + навигация) — согласуется с `index.md`/прогрессивным раскрытием OKF.
- **Чек-лист функций read-слоя** для будущего тестирования: вики-браузер, визуализация графа, гибридный поиск из UI.
- **Ролевая модель как референс** для слоя доступа: небольшая матрица ролей + per-resource ownership + пер-тенантный аудит-лог.

Что сознательно **не** берём: мульти-БД стек (Postgres + Redis + Neo4j + векторный движок), канон в БД, встроенный wiki-агент (write-path остаётся за нашим оркестратором и нашими инвариантами).

## 11. Открытые вопросы

- Конкретный контролируемый словарь типов рёбер (стартовый набор + правила расширения).
- Критерии выделения нового репо-компартмента в целевом домене (когда метки внутри общего репо перестают быть достаточными).
- Схема классификации (уровни, кто/что назначает метку на ingest — ручное vs авто-классификатор).
- Стратегия чанкинга разделов (по H2/H3? overlap? как хранить связь чанк→концепт).
- Стабильные ID концептов при переименованиях (immutable `id` + редирект-стабы? — кандидат на первый ADR, §4).
- Реализация слоя доступа: собственный тонкий сервис vs адаптация готового; минимально достаточная ролевая модель.
- WeKnora как read-слой — по результатам тестирования (чек-лист в §10.1).
- Формат и место хранения ADR (решений об отступлениях от OKF).
- Каденция pull-reconciler (таймер? при старте? watch?) и приоритизация очереди `needs-reconcile`; допустимое окно eventual consistency.
- Пороги авто-принятия vs обязательного ревью; включать ли сегрегацию обязанностей (рецензент ≠ автор) и для каких концептов.
- Технология data store (SQLite vs DuckDB), способ as-of/time-travel запросов, ретеншн и классификация датасетов.
- Именование и версионирование запросов деривации (query ID); пороги «материального» изменения данных для триггера `needs-reconcile`.
- Порог рекуррентности, по которому рефлексия предлагает датасет, и правила миграции существующих страниц-наблюдений в датасет (§3.1).
- Механика opt-in индексирования строк текстовых датасетов: средствами самого data store или отдельным корпусом общего поиска — решать при появлении первого текстового датасета, не заранее.
- Состав и пороги проверочного набора на записи (§7.4): топ-N поиска, глубина обхода входящих рёбер, включать ли пересечение тегов; носитель ориентации в начале сессии для runtime-агентов.

## Приложение: краткая карта решений

| Вопрос | Решение |
|---|---|
| Источник истины | Три канона: вики (знания), data store (данные), raw sources (входы); граф и индекс — производные |
| Граница data plane | Raw sources — не data plane; тест: агрегация/SQL → data store, цитата/провенанс → raw sources |
| OKF | Принять как substrate, расширять своими frontmatter-ключами |
| Конформность OKF | Цель, не жёсткий блокер; отступления — только через задокументированный ADR |
| Модель угроз | Пользователи с разными допусками + egress; админ канона вне модели |
| Слой доступа | Веб-сервис поверх канона (Вариант Б); enforcement, фильтрация retrieval и аудит — на сервисе |
| WeKnora | Референс, не платформа; заимствуем паттерны (§10.1), не стек; read-слой — решить по итогам тестирования |
| Компартменты | Метки + фильтрация на сервисе по умолчанию; отдельный репо — defense-in-depth для наиболее критичного; старт с двух |
| Деклассификация | Только человек: перенос вниз или санитизированная производная; кандидаты — lint по `classification_reason: inherited` |
| Git-история | Чувствительное рождается в своём компартменте (классификация на ingest); редкие случаи — runbook `git filter-repo` |
| Ручное редактирование | Разрешено; двухдорожечная пересборка (детерминированная сразу + семантическая фоном) |
| Триггер пересборки | Pull-reconciler по hash последнего материализованного коммита; git-хук — необязательный ускоритель |
| Модель ревизий | propose → pending → review → stable (по принципам Wikipedia); симметрично для человека и агента |
| Аудит чувствительных концептов | Обязательный человеческий ревью; авто-принятие отключено для правок любого актора |
| Формат правки агента | git-ветка + локальный ревью без форджа (валидаторы = CI-шаг, принятие = merge) |
| Данные vs знания | Наблюдения → data store (append-only); интерпретации → вики с контрактом деривации |
| Контракт деривации | `derived_from` (датасет + версионируемый запрос) · `as_of` · `window` · `refresh`; без него data-инсайт невалиден |
| Дескриптор датасета | Концепт-страница `type: Dataset` (схема во frontmatter); реестр — производная проекция |
| Новый датасет | Ревью-гейт §7.2: pending-страница → ревью владельца данных → merge → материализация |
| Эволюция схемы датасета | Только аддитивно и через ревью; несовместимое изменение = новый датасет |
| Маршрутизация наблюдений | `capture` только по совпадению со схемой; при сомнении — вики; staging-слой не нужен (вики и есть staging) |
| Продвижение вики → data store | Рефлексия предлагает датасет по рекуррентности; после принятия — миграция, старые страницы `superseded` |
| Индексирование строк данных | По умолчанию строки не в FTS/вектор; индексирование — opt-in свойство датасета |
| Модерируемые схемы | Единый примитив: pending → ревью → merge → детерминированная материализация; реализуется один раз (§7.3) |
| Сверка на записи | Перед созданием/существенной правкой страницы пишущий просматривает детерминированный проверочный набор: входящие рёбра + `relations:` + топ гибридного поиска; recall набора меряется golden set (§7.4) |
| Протокол противоречия | Конфликт с существующей страницей не правится молча: ребро `contradicts` с провенансом, разрешение через ревью-гейт (§7.4) |
| Узел графа | Концепт = страница |
| Извлечение | Уровень раздела (чанкинг по заголовкам) |
| Кто порождает рёбра | Оркестратор в момент синтеза, приземляя в markdown |
| Граф-стор | Материализуется детерминированно из явных рёбер |
| Типы рёбер | Контролируемый словарь + валидатор |
| Провенанс рёбер | Только в immutable-источник; не инвалидируется правкой прозы; `verified`/`confidence` упразднены |
| `log.md` | Генерируется из git-истории, не ведётся руками |
| Эмбеддинги | Версия модели фиксируется; смена = плановая полная переиндексация |
