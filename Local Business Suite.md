# Корпоративный портал ВОБ №3

## Архитектурный шаблон для локальной платформы внутренних бизнес-приложений

Этот документ описывает не текущее прикладное наполнение конкретного MVP, а архитектурный шаблон для семейства локально развертываемых внутренних систем уровня department/branch/clinic/plant office.  
Он собран на основе уже принятых и проверенных решений в реализованном MVP `local-business-suite` и предназначен как:

- архитектурный blueprint для следующих модулей;
- опорный документ для ИИ-агента;
- шаблон проектирования локальных line-of-business систем без зависимости от облачной инфраструктуры;
- база для дальнейшего масштабирования в сторону аналитики, интеграций и enterprise-экосистемы.

---

## 1. Назначение архитектуры

Шаблон предназначен для построения модульной внутренней платформы, где:

- несколько прикладных модулей живут в одном монорепозитории;
- приложения разворачиваются локально на сервере предприятия или подразделения;
- основная транзакционная часть должна быть простой, надежной и малотребовательной к сопровождению;
- UI строится серверно, без тяжелого SPA;
- система должна уметь постепенно эволюционировать:
  от локального MVP к более зрелой платформе с LDAP, интеграциями, аналитикой и внешними мостами.

Архитектура ориентирована на сценарии, где:

- нет выделенной платформенной команды уровня большой enterprise-организации;
- важны локальная автономность, быстрый ввод в эксплуатацию и прозрачность сопровождения;
- часть развития выполняется ИИ-агентом, поэтому архитектура должна быть code-first, предсказуемой и легко обозримой.

---

## 2. Базовые архитектурные принципы

### 2.1 Local-first

Основная система разворачивается и работает локально.  
Критичные данные, роли, справочники, workflow и базовая аналитика должны продолжать работать без обязательной зависимости от внешних облачных сервисов.

### 2.2 Monorepo + модульность

Каждый прикладной домен оформляется как отдельный `django app`, но использует общую инфраструктуру:

- единый auth слой;
- единый layout;
- общие policy helpers;
- общие справочники и интеграционные механизмы;
- единый deployment package.

Это дает:

- простое развертывание;
- меньше организационных границ между модулями;
- единый способ сопровождения кодовой базы;
- удобный AI-agent workflow.

### 2.3 Server-driven UI

Интерфейс должен по умолчанию быть серверным:

- Django Templates;
- HTMX для частичных обновлений;
- минимальный JavaScript для специальных взаимодействий;
- CSS без тяжелого frontend runtime.

Это уменьшает сложность:

- не нужен отдельный SPA frontend и отдельный API-first слой для всего подряд;
- меньше точек отказа;
- проще сопровождать локально;
- агенту проще вносить изменения end-to-end.

### 2.4 Policy-first backend

Критичная бизнес-логика не должна жить в шаблонах или UI-условиях.

Нужны явные уровни:

- `models.py` для сущностей;
- `policies.py` для авторизации и правил доступа;
- `services.py` для бизнес-действий и state changes;
- `views.py` для request orchestration.

Проверенное правило MVP:

- UI может скрыть кнопку;
- но единственный источник истины для прав и допустимых действий находится на сервере.

### 2.5 Config-driven поведение там, где это полезно

Если набор ролей, колонок, маршрутов переходов или аналитических правил может меняться без переписывания прикладной модели, его лучше делать конфигурируемым:

- через JSON/YAML;
- через справочники в БД;
- через параметризуемые правила.

В MVP это подтвердилось на двух направлениях:

- канбан-колонки должны быть параметрическими, а не захардкоженными;
- прикладные права ролей лучше описывать конфигом, а не if-ветками по именам ролей.

## 3. Базовый стек

### 3.1 Транзакционный слой

- Framework: Django 5.x
- Runtime: Python 3.12+
- DB: SQLite для OLTP на этапе local-first MVP
- Static: WhiteNoise
- Deployment: Docker Compose / systemd / reverse proxy

### 3.2 UI слой

- Django Templates
- HTMX
- CSS без обязательного frontend bundler

Примечание по стилю:

- HTMX используется как enhancement-слой, а не как замена всей архитектуры взаимодействия;
- базовый HTTP flow должен оставаться понятным и работать без сложной client-side state machine.

### 3.3 Аналитический слой

Рекомендуемый целевой стек:

- Evidence как analytics-as-code UI
- Parquet как формат аналитического слоя
- DuckDB как встроенный аналитический движок

Подробно описано в отдельном разделе ниже.

## 5. Паттерны, подтвержденные реализацией MVP

Ниже не описание конкретного прикладного сценария, а архитектурные решения, которые уже показали себя практичными.

### 5.1 SQLite можно использовать как OLTP-слой для локального MVP

При умеренной нагрузке и локальном развертывании SQLite допустим как основная БД, если:

- включен WAL;
- выставлены `busy_timeout` и адекватный `timeout`;
- запросы и workflow не превращают систему в write-heavy high-contention сервис;
- тяжелая аналитика вынесена из live OLTP.

Проверенное решение:

```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.sqlite3",
        "NAME": BASE_DIR / "db" / "main_vault.sqlite3",
        "OPTIONS": {
            "timeout": 20,
            "init_command": (
                "PRAGMA journal_mode=WAL;"
                "PRAGMA synchronous=NORMAL;"
                "PRAGMA busy_timeout=5000;"
            ),
        },
    }
}
```

### 5.2 Soft-archive лучше физического удаления для справочников

Если объект участвует в истории операций, workflow, ссылках и аналитике, его лучше архивировать, а не удалять:

- сохраняется ссылочная целостность;
- история не ломается;
- отчеты остаются воспроизводимыми.

### 5.3 Workflow должен быть отдельно от UI

Хороший шаблон:

- модель хранит текущее состояние;
- `policies.py` решает, кто имеет право на действие;
- `services.py` выполняет бизнес-переход и аудит;
- `views.py` просто вызывает policy + service.

Это особенно важно, если:

- часть действий идет через HTMX;
- есть drag-and-drop;
- роли и transition matrix конфигурируемы.

### 5.4 Канбан/доски лучше делать параметрическими

Колонки и привязка статусов к колонкам не должны быть частью layout-only кода.  
Это должен быть либо DB-config, либо явный конфигурационный слой.

Причина:

- UI и workflow меняются независимо;
- разные модули или предприятия могут захотеть свою декомпозицию колонок;
- агент может менять поведение без переписывания view logic.

### 5.5 Права лучше описывать декларативно

Роли должны быть равны с точки зрения кода.  
Код должен знать не “техника” и “менеджера”, а:

- scopes;
- capabilities;
- transition targets;
- management flags.

Проверенный паттерн:

- JSON-конфигурация ролей;
- generic policy-engine поверх нее;
- UI для правки JSON с серверной валидацией.

### 5.6 Иерархические справочники нужны рано

Даже если в MVP кажется, что “подразделение” это просто строка, на практике быстро выясняется, что нужна иерархия:

- филиал / корпус / отделение / кабинет;
- департамент / участок / смена;
- холдинг / завод / цех / линия.

Поэтому лучше сразу поддерживать общий иерархический справочник как базовый паттерн платформы.

### 5.7 Номера сущностей должны быть человекочитаемыми

Технический timestamp-ID не должен попадать в UI как основной номер.  
Бизнес-номер должен быть:

- стабильным;
- коротким;
- понятным пользователю;
- пригодным для поиска и коммуникации.

### 5.8 AI-friendly архитектура требует code-first поверх UI-first

Там, где предполагается активное участие ИИ-агента, лучше работают:

- конфиги в JSON/YAML;
- SQL-модели;
- code-first dashboards;
- четкое разделение domain/service/policy/view.

Хуже работают:

- настройки, спрятанные только в браузерных формах;
- неявная логика в шаблонах;
- визуальные BI-конструкторы без экспорта в код.

## 6. Авторизация, роли и корпоративная идентичность

### 6.1 Этап 1: локальная auth-модель

Первая поставка должна уметь работать автономно:

- стандартная Django auth;
- логин/пароль;
- локальные группы;
- локальные прикладные роли.

### 6.2 Этап 2: LDAP / AD integration

LDAP добавляется как identity source, но не как единственный источник прикладных прав.

Правильный принцип:

- LDAP отвечает за identity;
- прикладная система отвечает за business roles и policy model.

Это позволяет:

- не ломать локальную модель ролей;
- не делать бизнес-логику зависимой от структуры AD;
- иметь fallback и миграционный путь.

### 6.3 Role engine

Рекомендуемый шаблон:

- роли определяются в `config/role_rules.json`;
- engine читает их через `settings`;
- seed-команды создают локальные группы из конфига;
- UI для изменения конфига доступен только управленческой роли.

Такой подход лучше hardcoded role branches, потому что:

- роли можно менять без переписывания policy engine;
- система подходит для разных предприятий;
- агент может безопасно редактировать права через конфиг.

## 7. UI архитектура

### 7.1 Базовая модель

UI строится как server-rendered application с progressive enhancement.

Основной шаблон:

- `base.html`
- блоки для глобальной навигации
- page templates
- partial templates
- HTMX targets для локальных обновлений

### 7.2 Когда нужен HTMX

HTMX следует использовать там, где:

- нужно быстрое частичное обновление;
- модальный/side-panel интерфейс удобнее full reload;
- требуются inline действия;
- нет смысла строить отдельный JS state management.

### 7.3 Когда нужен обычный full-page flow

Не надо насиловать HTMX там, где обычный page flow проще:

- сложные multi-field формы;
- редкие административные настройки;
- страницы с высоким когнитивным весом;
- сценарии, где пользователю удобнее явный переход.

### 7.4 Минимальный JavaScript допустим

Полный запрет на JS излишен.  
Разумный подход:

- бизнес-логика не уходит в JS;
- но точечный JS для DnD, закрытия панелей, мелких enhancement-паттернов допустим.

## 8. Модуль аналитики: Evidence + Parquet + DuckDB

Это рекомендуемая целевая архитектура локального аналитического слоя для данной платформы.

### 8.1 Почему не читать live OLTP напрямую

Тяжелая аналитика поверх live `main.sqlite` плоха как целевая архитектура:

- аналитические запросы конкурируют с транзакционными;
- отчеты читают “живое” состояние без snapshot discipline;
- BI-слой зависит от операционной схемы;
- сложнее эволюционировать дальше.

Поэтому аналитический слой должен быть отделен от OLTP хотя бы логически, а лучше и физически.

### 8.2 Роли компонентов

#### Parquet

Parquet это основной формат аналитического слоя.

Почему именно он:

- колонночный формат;
- отлично подходит для чтения агрегатных и фильтруемых выборок;
- хорошо работает с локальными файлами;
- не привязан к одному конкретному BI-движку;
- хорошо поддерживается DuckDB.

DuckDB умеет эффективно читать Parquet напрямую, используя:

- projection pushdown;
- filter pushdown;
- оптимизации чтения колонночного формата.

По официальной документации DuckDB, Parquet является естественным аналитическим форматом для read-oriented workloads и может читаться напрямую через `read_parquet(...)` или через имя файла.

#### DuckDB

DuckDB это аналитический query engine.

Его роль:

- читать `Parquet`;
- строить локальные views и marts;
- выполнять агрегаты, joins, window functions и аналитические SQL-запросы;
- служить компактным embedded OLAP слоем.

DuckDB не обязан быть “главной БД платформы”.  
В этой архитектуре он выступает как:

- локальный аналитический движок;
- тонкий локальный warehouse/runtime;
- engine для curated analytics datasets.

#### Evidence

Evidence это analytics-as-code presentation layer.

По официальной документации:

- pages пишутся как markdown;
- SQL queries живут прямо в markdown code fences;
- источники описываются в `sources/`;
- workflow code-driven и git-friendly;
- основной SQL dialect в страницах ориентирован на DuckDB.

Это делает Evidence хорошим выбором для:

- агентной доработки аналитики;
- version-controlled отчетов;
- explainable dashboards, где важен текст, контекст и narrative;
- локального self-hosted analytics UI.

### 8.3 Рекомендуемая схема данных

```text
OLTP SQLite
    |
    | export / snapshot / transform
    v
analytics_store/raw/*.parquet
    |
    | normalize / enrich / aggregate
    v
analytics_store/marts/*.parquet
    |
    | query engine
    v
DuckDB
    |
    | SQL data source
    v
Evidence
```

### 8.4 Рекомендуемые слои аналитического контура

#### Raw layer

Почти прямой экспорт из OLTP:

- справочники;
- факты;
- события;
- журналы переходов;
- документы;
- временные метки.

#### Intermediate / normalized layer

Нормализация:

- дедупликация;
- приведение имен полей;
- унификация временных зон;
- flattening и denormalization там, где это нужно аналитике.

#### Mart layer

Бизнес-ориентированные наборы данных:

- производительность;
- SLA;
- инциденты;
- утилизация оборудования;
- динамика статусов;
- backlog;
- workload by department / assignee / device class.

### 8.5 Режим обновления

Для local-first платформы предпочтителен batch-refresh, а не live-querying:

- по cron;
- по systemd timer;
- по management command;
- по событию завершения ETL.

Рекомендуемые варианты:

- каждые 5-15 минут для near-real-time аналитики;
- почасово для нетребовательной отчетности;
- nightly для тяжелых исторических расчетов.

### 8.6 Форматы интеграции с Evidence

Evidence лучше рассматривать как repository-based analytics app.

Типовая структура:

```text
analytics_evidence/
├── pages/
├── sources/
│   └── analytics_store/
│       ├── connection.yaml
│       ├── analytics.duckdb
│       └── *.sql
├── components/
└── static/
```

По документации Evidence:

- источники подключаются через `sources/.../connection.yaml`;
- SQL-запросы могут выполняться как именованные query blocks;
- страницы и запросы удобно держать в git.

### 8.7 Плюсы такого подхода

- аналитика отделена от OLTP;
- отчетность и BI становятся code-reviewable;
- агент может безопасно развивать dashboards и метрики;
- нет необходимости сразу тянуть тяжелый warehouse;
- путь миграции дальше остается открытым.

### 8.8 Ограничения

- это не замена enterprise data platform;
- нужен явный export/refresh pipeline;
- governance метрик все равно должен быть дисциплинированным;
- для self-service BI обычным пользователям Metabase может быть удобнее, чем Evidence.

## 9. Мост к существующим локальным системам предприятия

Здесь речь идет о сценарии, когда до внедрения платформы уже существуют отдельные локальные системы:

- учетные системы;
- регистратуры;
- сервисные базы;
- системы оборудования;
- ERP/склад;
- кадровые каталоги;
- файловые шары;
- локальные БД Access/SQL Server/PostgreSQL/MySQL;
- выгрузки CSV/XLSX/XML/DBF.

Ниже перечислены практичные варианты моста.

### 9.1 Файловый мост

Самый простой и часто самый реалистичный первый этап:

- импорт CSV/XLSX/XML/JSON;
- экспорт отчетов и справочников в обменные файлы;
- обмен через сетевую папку.

Плюсы:

- минимальная зависимость от legacy API;
- быстро внедряется;
- легко автоматизировать.

Минусы:

- слабая near-real-time интеграция;
- нужно контролировать формат и качество данных;
- сложнее отслеживать ошибки обмена.

Когда подходит:

- старые системы без API;
- локальные регламенты выгрузок;
- пилотный этап интеграции.

### 9.2 SQL read bridge

Подключение к существующим локальным БД в read-only режиме:

- ODBC/JDBC;
- прямой коннектор;
- периодический ETL в staging.

Плюсы:

- хорошие данные без ручных выгрузок;
- лучше для построения сводной аналитики;
- подходит для legacy MIS/ERP/registry DB.

Минусы:

- чувствительность к схеме внешней системы;
- нужны права доступа и сетевое доверие;
- нельзя строить критичный workflow поверх чужой live schema.

Рекомендация:

- читать такие системы только в staging/raw layer;
- не связывать транзакционный контур напрямую с чужими БД.

### 9.3 API bridge

Если legacy-система умеет REST/SOAP/HTTP:

- отдельный connector;
- периодическая синхронизация;
- очереди retries;
- журнал ошибок интеграции.

Плюсы:

- чище, чем file exchange;
- меньше привязка к физической БД;
- можно строить near-real-time sync.

Минусы:

- API legacy-систем часто нестабильны;
- нужно управлять idempotency, auth, retries.

### 9.4 Message bridge / queue bridge

Если в организации уже есть локальные очереди или bus:

- RabbitMQ;
- Kafka;
- NATS;
- MSMQ;
- иные локальные брокеры.

Платформа может:

- публиковать события;
- читать события legacy-систем;
- строить асинхронную интеграцию.

Это сильный вариант, но он оправдан только если enterprise-landscape уже достаточно зрелый.

### 9.5 Shared directory / DMS bridge

Частый локальный сценарий:

- файлы, документы, сканы и вложения уже живут на SMB/NFS-шарах или в DMS.

Варианты:

- хранить в платформе только метаданные и ссылки;
- копировать в локальное media-хранилище;
- строить двусторонний reference layer.

Выбор зависит от:

- требований к юридической значимости;
- резервному копированию;
- размеру файлов;
- правилам доступа.

### 9.6 LDAP / AD bridge

Это не только auth-интеграция, но и мост к оргструктуре:

- учетные записи;
- ФИО;
- email;
- подразделения;
- руководители;
- membership.

Но важное правило:

- оргданные можно импортировать;
- прикладные business rules не должны целиком зависеть от AD.

### 9.7 Bridge для аналитического слоя

Самый полезный вариант для legacy-окружения:

- не тянуть legacy-системы прямо в OLTP-модуль;
- тянуть их в analytics raw layer;
- строить сводные cross-system marts.

Это позволяет:

- получать объединенную аналитику;
- не ломать транзакционную модель платформы;
- использовать систему как единый аналитический фасад над зоопарком старых систем.

### 9.8 Рекомендуемая стратегия моста

Для local-first enterprise landscape рекомендуемый порядок такой:

1. File bridge
2. Read-only SQL bridge
3. API bridge
4. Event bridge

То есть:

- сначала берем наиболее дешевые и стабильные источники;
- не форсируем глубокую онлайн-интеграцию до стабилизации модели;
- накапливаем опыт и каталог внешних систем;
- только потом переходим к near-real-time оркестрации.

## 11. Улучшения для дальнейшего развития архитектуры

### 11.1 Вынести конфигурацию платформы в системный config layer

Сейчас уже подтверждено, что config-driven подход полезен.  
Следующий шаг:

- единый каталог конфигов;
- schema validation;
- environment-specific overlays;
- документированный hot-reload policy там, где это безопасно.

### 11.2 Ввести integration registry

Нужен единый реестр внешних систем:

- название;
- владелец;
- тип доступа;
- формат;
- SLA;
- частота синхронизации;
- чувствительность данных;
- статус интеграции.

Это особенно полезно, если платформа станет точкой консолидации вокруг старых локальных систем.

### 11.3 Разделить operational and analytical schemas формально

Даже если физически все локально, логически следует разделять:

- operational entities;
- audit/event entities;
- analytics exports;
- marts.

Иначе OLTP и BI опять начнут смешиваться.

### 11.4 Ввести формализованный event/audit слой

Не только transition logs, а общий доменный event pattern:

- что произошло;
- когда;
- над чем;
- кем;
- из какого источника.

Это сильно помогает и интеграциям, и аналитике.

### 11.5 Подготовить архитектуру к переходу с SQLite на PostgreSQL

SQLite остается хорошим local-first стартом, но шаблон должен сохранять путь миграции:

- не использовать SQLite-specific hacks как часть domain model;
- аналитический слой не должен быть завязан на live SQLite;
- репозиторий и service layer должны быть готовы к смене OLTP storage.

### 11.6 Нормализовать shared dictionaries

Иерархические справочники уровня:

- подразделения;
- локации;
- классы активов;
- контрагенты;
- категории инцидентов;
- причины простоев;

лучше рано сделать общими платформенными building blocks.

### 11.7 Добавить schema contracts для аналитики

Для `Parquet`-экспорта полезно ввести:

- явные имена датасетов;
- стабильные схемы;
- версии датасетов;
- backward compatibility policy.

Тогда BI-контур перестанет зависеть от случайных rename полей в OLTP.

### 11.8 Добавить orchestration слой для batch jobs

Минимум:

- management commands;
- cron/systemd timers;
- structured job logs.

Следующий уровень:

- Prefect / Dagster / Airflow only if justified.

Для local-first шаблона лучше не начинать с тяжелого orchestrator, если нет реальной необходимости.

### 11.9 Ввести module contracts

Чтобы монорепозиторий не выродился в “все импортирует все”, нужны правила:

- `core` не знает доменную бизнес-специфику модулей;
- модули не лезут друг другу в приватную внутреннюю логику;
- shared helpers и dictionaries выносятся осознанно, а не по привычке.

### 11.10 Подумать о multi-tenant / multi-site профиле

Если система пойдет дальше одного подразделения, полезно заранее определить:

- single-site local deployment;
- multi-site independent deployments;
- central analytic aggregation;
- центральный каталог конфигураций и role policies.

## 12. Практическая рекомендация

Для текущего класса систем рекомендуемая эволюция такая:

1. Django monorepo + SQLite WAL + HTMX + local auth
2. Config-driven roles + hierarchical dictionaries + policy/service separation
3. File/API/SQL bridges к legacy-системам
4. Parquet export layer
5. DuckDB analytical engine
6. Evidence analytics module
7. LDAP / AD как identity bridge
8. При росте нагрузки: переход OLTP на PostgreSQL без слома аналитического контура

Именно такая траектория дает лучший баланс между:

- скоростью разработки;
- локальной автономностью;
- простотой сопровождения;
- пригодностью для ИИ-агента;
- возможностью безболезненно расти дальше.
