# План упрощения MVP-архитектуры памяти

Статус: реализовано в исполнительском блоке `workflow/active/memory-mvp-simplification/`.

Дата: 2026-05-21.

Связанный ADR: `docs/adr/ADR-0010-memory-mvp-simplification.md`.

## Назначение

Этот документ описывает, как упростить текущую систему памяти без сильной потери качества и возможностей.

Цель - оставить рабочее ядро памяти, убрать лишние обязательные слои из MVP и сделать поведение системы понятным:

- что именно считается сохраненным знанием;
- какие объекты участвуют в обычном поиске;
- что является текущей обработкой очереди, а что будет настоящей ночной обработкой знаний;
- какие источники можно отдавать боту напрямую.

## Границы плана

Входит:

- `MemoryKnowledgeItem` как главный объект смысловой памяти MVP;
- удаление или отключение `MemoryBelief` до следующих этапов;
- сохранение векторного, полнотекстового и графового поиска без детальной проработки графовой стратегии;
- разделение текущего обработчика очереди и будущей ночной рефлексии;
- упрощение правил надежных источников;
- документация и проверки.

Не входит:

- перенос сообщений чата в отдельный SQLite-файл;
- изменение физического хранения истории чата;
- детальный план эффективного использования графового поиска;
- изменение файловых проекций памяти;
- production UI для проверки знаний.

## Текущее состояние

До исправления разрывов явное запоминание из чата могло синхронно создавать несколько связанных представлений:

```text
MemoryWriteRequest
  -> MemoryKnowledgeItem
  -> MemoryKnowledgeEvent
  -> memory.current.json / memory.current.md
  -> MemorySnapshot
  -> MemoryChunk
  -> MemoryClaim
  -> MemoryBelief
```

Это слишком много для MVP. Главная проблема - `MemoryBelief` пока не добавляет самостоятельной пользы в обычном пути. Он создается почти автоматически из того же текста, который уже есть в сохраненном знании.

## Целевая MVP-схема

Для явного "запомни":

```text
MemoryWriteRequest
  -> MemoryKnowledgeItem
  -> MemoryKnowledgeEvent
  -> файлы текущей личной/общей памяти
  -> MemorySnapshot
  -> MemoryChunk
  -> поисковый индекс
```

`MemoryClaim` может существовать как необязательная заготовка для будущих проверок, но не должен быть обязательным результатом каждого запроса.

`MemoryBelief` не участвует в MVP-пути.

Реализованное решение после remediation-блока: обычный `memory.remember` только ставит `MemoryWriteRequest` и `MemoryIndexJob` в очередь и возвращает `status=queued`. Совместимый обработчик очереди создает/обновляет `MemoryKnowledgeItem`, индексирует его в стандартный полнотекстовый backend и после этого знание находится через `memory.search`. Автоматическое создание `MemoryClaim` и `MemoryBelief` в этом пути отключено.

## Главный объект памяти

`MemoryKnowledgeItem` становится главным объектом сохраненного знания.

Ожидаемое поведение:

- пользователь сказал "запомни" - создается `MemoryKnowledgeItem`;
- пользователь изменил личную память - меняется или заменяется `MemoryKnowledgeItem`;
- пользователь удалил личную память - `MemoryKnowledgeItem` получает состояние `deleted`;
- кандидат в общую память ссылается на `MemoryKnowledgeItem`;
- поиск может возвращать текстовые фрагменты, созданные из `MemoryKnowledgeItem`.

## `MemoryClaim`

`MemoryClaim` не удаляется этим планом автоматически, но его роль ограничивается.

Разрешенное применение в MVP:

- будущие кандидаты в общие знания;
- ручная проверка спорных утверждений;
- заготовка для аналитики знаний.

Запрещенное для MVP поведение:

- считать `MemoryClaim` главным объектом памяти;
- создавать `MemoryClaim` как обязательный шаг каждого обычного "запомни", если это не требуется для принятого сценария;
- отдавать `MemoryClaim` как отдельный источник обычного ответа без понятной проверки.

## `MemoryBelief`

`MemoryBelief` переносится на будущие этапы и не входит в MVP-схему текущей ветки.

Требуемое изменение:

- убрать автоматическое создание `MemoryBelief` при обработке `memory.remember`;
- убрать поиск по `MemoryBelief` из обычного `memory.search`;
- убрать модель, миграцию и admin-регистрацию `MemoryBelief`, если миграция еще не опубликована;
- убрать счетчики и документы, которые создают впечатление, что `MemoryBelief` является частью готового MVP;
- если таблица уже создана в среде, подготовить безопасный путь удаления или выключения использования.

## Поиск

Сохраняются:

- векторный поиск;
- полнотекстовый поиск;
- графовый поиск.

Ограничение этого плана: как именно граф должен подключаться к обычному поиску, будет решено отдельно.

До отдельного решения:

- графовый поиск не должен требовать обязательного вызова локальной модели;
- графовый поиск не должен опираться на `MemoryBelief`;
- обычный поиск должен проходить через права доступа, чувствительность и проверку надежности источника.

## Обработка очереди и ночная рефлексия

Текущее поведение `memory_reflect_chats` нужно назвать точнее.

Предлагаемое разделение:

```text
Обработчик очереди памяти
  -> берет queued MemoryWriteRequest
  -> создает/обновляет MemoryKnowledgeItem
  -> обновляет файлы и индексы

Ночная рефлексия
  -> позже анализирует историю чатов
  -> ищет устойчивые знания
  -> объединяет повторы
  -> предлагает кандидаты в общую память
```

Для MVP команда `memory_reflect_chats` оставлена как совместимый обработчик очереди и кандидатов. Ее help text и документация прямо указывают, что это не полноценная ночная рефлексия.

## Надежные источники

Для MVP используется простая схема:

| Статус | Смысл |
| --- | --- |
| `trusted` | Можно отдавать боту после проверки прав и чувствительности. |
| `review_required` | Можно сохранять и разбирать, но нельзя отдавать боту напрямую до проверки. |
| `blocked` | Нельзя использовать. |

Совместимость с текущими статусами:

| Сейчас | MVP-схема |
| --- | --- |
| `trusted` | `trusted` |
| `candidate_only` | `review_required` |
| `quarantined` | `review_required` или `blocked` по решению владельца источника |
| `blocked` | `blocked` |

Дополнительные признаки вроде класса источника можно оставить для пояснения и ранжирования, но не как обязательную сложную матрицу.

В коде применяется совместимое отображение: `candidate_only` и `quarantined` становятся `review_required`, `trusted` и `blocked` сохраняются. Обычный прямой контекст по-прежнему разрешен только для `trusted` источников после проверок прав доступа, чувствительности и citations.

## Порядок реализации

1. Зафиксировать решение и обновить документацию.
2. Убрать `MemoryBelief` из обычного пути `memory.remember`.
3. Убрать `MemoryBelief` из `memory.search` и связанных счетчиков.
4. Проверить роль `MemoryClaim` и отключить его автоматическое создание, если оно не нужно MVP.
5. Разделить обработчик очереди памяти и будущую ночную рефлексию в командах и документации.
6. Упростить политику надежных источников через совместимый слой или изменение контрактов.
7. Обновить тесты, документацию развертывания и пользовательскую инструкцию.

## Проверки готовности

- Явное "запомни" возвращает `queued`; обработчик очереди создает главный результат `MemoryKnowledgeItem`.
- После обработки очереди сохраненное знание находится через `memory.search`.
- В обычном пути не создается и не используется `MemoryBelief`.
- `memory.search` не возвращает `MemoryBelief`.
- Векторный, полнотекстовый и графовый поиск продолжают работать в рамках текущей реализации.
- Источники без доверия не попадают в обычный ответ бота.
- Текущая команда обработки очереди не называется полноценной ночной рефлексией.
- Документация не описывает `MemoryBelief` как обязательную часть MVP.
