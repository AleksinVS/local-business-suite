# ADR-0016: Профили гибридного ранжирования памяти

## Статус

Accepted, реализация отложена

Отложен управляемо решением `docs/adr/ADR-0030-memory-alignment-hybrid-knowledge-v05.md`: в runtime остается один профиль по умолчанию (RRF). Концепция профилей — архитектурный долг; возврат к реализации после подтверждения пользы через `memory_eval` на реальном корпусе.

## Дата

2026-05-27

## Контекст

Поиск памяти уже поддерживает FTS5 для полнотекстового поиска и LanceDB для векторного поиска. Agent runtime умеет передавать `search_mode` и `include_source_data` в `memory.search`.

Текущая проблема: FTS/BM25 и vector score находятся в разных шкалах. BM25 зависит от частоты слов, длины документов и корпуса. Vector score зависит от расстояния между embedding-векторами. Эти значения нельзя надежно складывать напрямую: канал с более крупными числами может доминировать независимо от реальной релевантности.

Дополнительно нужно закрепить поведение ИИ-бота. Бот должен выбирать понятный режим поиска по намерению пользователя, а не передавать произвольные веса каналов.

## Решение

Ввести управляемые профили ранжирования памяти.

### Разделение обязанностей

- `search_mode` отвечает за корпус и каналы поиска: `knowledge`, `source_data`, vector, graph compatibility.
- `ranking_profile` отвечает за способ объединения кандидатов: нормализация, веса каналов, бонус за совпадение нескольких каналов, authority boost и штрафы.
- ИИ-бот выбирает режим и допустимый профиль по намерению пользователя.
- ИИ-боту запрещено передавать сырые веса `fulltext_weight`, `vector_weight`, `graph_weight`.

### Базовая нормализация

Для MVP использовать rank-based fusion через RRF как основной способ объединения каналов.

```text
rrf_score = 1 / (k + rank)
final_score =
  fulltext_weight * rrf_fulltext +
  vector_weight * rrf_vector +
  graph_weight * rrf_graph +
  overlap_boost +
  authority_boost -
  penalties
```

Причины выбора RRF:

- не зависит от разных числовых шкал BM25 и vector distance;
- устойчив к выбросам;
- понятен в trace;
- подходит для первого production-safe поведения без обученной калибровки.

Raw score каждого backend сохраняется только как диагностическая информация в trace/metadata. Для диагностики допускается дополнительно считать min-max normalized score внутри каждого канала, но итоговое ранжирование MVP должно опираться на RRF.

### Предлагаемые профили

```text
precise
  fulltext: 0.90
  vector:   0.10
  graph:    0.00
  Назначение: точные термины, номера, коды, названия файлов, регламенты.

balanced
  fulltext: 0.55
  vector:   0.45
  graph:    0.00
  Назначение: обычный поиск сведений по теме.

semantic_heavy
  fulltext: 0.25
  vector:   0.75
  graph:    0.00
  Назначение: поиск по смыслу, когда пользователь не знает точную формулировку.

source_content
  fulltext: 0.70
  vector:   0.30
  graph:    0.00
  Назначение: поиск по содержимому исходных файлов с приоритетом точных совпадений.

source_semantic
  fulltext: 0.30
  vector:   0.70
  graph:    0.00
  Назначение: поиск по смыслу в исходных файлах.

graph_future
  fulltext: 0.35
  vector:   0.35
  graph:    0.30
  Назначение: заготовка после включения graph runtime search.
```

### Source semantic search

Source semantic search означает embedding-индексацию и семантический поиск по `source_data`, то есть по исходным файлам и документам, которые не стали принятым знанием.

Решение: включить source semantic search в ближайший этап реализации, но без нового внешнего API и без cloud embeddings. Доступ остается через существующий `memory.search` в Django gateway и agent runtime wrapper. Embeddings строятся только локальной моделью.

Правила:

- `source_data` индексируется в LanceDB вместе с `knowledge`, но corpus разделяется через `corpus_type`, metadata и фильтры retrieval;
- source vectors используются только в режимах, где `source_data` явно допустим: `source_explicit` и source-часть `source_fallback`;
- `knowledge_semantic` по умолчанию ищет accepted knowledge и не должен незаметно подмешивать `source_data`;
- ответ с `kind="source_data"` всегда остается ссылкой на исходный документ, а не утвержденным знанием;
- новый внешний HTTP/API для поиска не вводится.

### Sensitivity, secrets и PII

Embedding-индексирование разрешено для всех sensitivity уровней, кроме документов, где найден секрет.

Правила обработки:

- если найден секрет, индексирование этого документа останавливается до записи в FTS/vector, создается issue `secret_blocked` с severity `blocker`, документ переводится в неготовый статус, администратор получает задачу на согласование, reindex продолжает следующий документ;
- если документ уже был проиндексирован и при повторной обработке в нем найден секрет, старые FTS/vector записи удаляются до постановки issue;
- PII и прочие чувствительные признаки не блокируют индексирование и не обезличиваются перед локальными FTS/LanceDB индексами, но создают audit/review issue `pii_audit` с severity `warning` или `info`; документ индексируется, а администратор получает задачу на аудит;
- issue и audit metadata не должны хранить секретное значение или необезличенную PII, только тип срабатывания, detector id, confidence, безопасный фрагмент/позицию при допустимости и ссылки на source object;
- cloud/API embeddings для таких данных запрещены.

### Reindex и удаление

Reindex должен быть идемпотентным.

Ключ записи: стабильный `document_id` `MemorySearchDocument`. Для текущего MVP source semantic search использует один vector на документ. Это временное решение: после этапа `MemorySearchSegment` vector ids должны перейти на segment-level ids.

Инкрементальная переиндексация выполняется только если изменилось хотя бы одно:

- `content_hash`;
- parser/extraction version;
- embedding profile/model/version;
- ACL/scope fingerprint;
- sensitivity;
- trust status или authority class;
- corpus type/index profile;
- явный `--force`.

При изменении документа:

1. извлечь текст и проверить его на секреты/PII;
2. выполнить DLP/secret/PII gate;
3. если найден секрет, удалить старые FTS/vector записи этого `document_id`, создать blocker issue и перейти к следующему документу;
4. если найдена PII без секрета, создать audit issue и продолжить;
5. выполнить upsert в FTS;
6. выполнить delete+upsert в LanceDB по `document_id`, чтобы не оставлять старые payload/scope/sensitivity;
7. обновить `MemorySearchDocument.metadata.index_versions`, `indexed_at`, `body_hash` и статус.

При удалении или исчезновении файла:

- `MemorySourceObject` помечается как missing/deleted согласно текущей модели discovery;
- связанный `MemorySearchDocument` переводится в `deleted`;
- FTS rows удаляются;
- LanceDB rows удаляются по `document_id`;
- открытые ingestion issues по объекту можно оставить для истории или закрывать отдельным cleanup-правилом, но они не должны возвращаться в search.

При изменении ACL или sensitivity без изменения содержимого нужно обновить FTS/vector payload через delete+upsert, чтобы старые фильтры доступа не остались в индексе.

### Гранулярность

На этом этапе индексируется документ целиком: один `MemorySearchDocument` дает один FTS-документ и один vector.

Это временное решение. Следующий этап должен перейти к поиску по крупным фрагментам через `MemorySearchSegment`: разделы документа, листы таблицы или диапазоны строк. До этого качество semantic search по длинным файлам может быть хуже и должно отражаться в eval.

### Подсказки ИИ-бота

Системная подсказка agent runtime должна закреплять правила выбора:

```text
Если пользователь просит искать точное слово, номер или фразу в исходных
файлах, документах, вложениях, таблицах или по содержимому файла, вызывай memory.search с
search_mode="source_explicit" и ranking_profile="source_content".

Если пользователь просит искать по смыслу именно в исходных файлах или
документах, вызывай memory.search с search_mode="source_explicit" и
ranking_profile="source_semantic".

Если пользователь просит найти принятое знание по смыслу, похожую инструкцию,
аналогичный случай или говорит, что не помнит точные слова, вызывай
memory.search с search_mode="knowledge_semantic" и
ranking_profile="semantic_heavy".

Если пользователь ищет точный номер, код, название, дату, файл или термин,
вызывай memory.search с search_mode="knowledge_precise" и
ranking_profile="precise".

Если пользователь хочет сначала accepted knowledge, но допускает исходные
документы как запасной вариант, вызывай memory.search с
search_mode="source_fallback" и ranking_profile="balanced".

Не передавай сырые веса каналов. Не выдавай source_data как принятое знание.
Если результат имеет kind="source_data", явно укажи, что это исходный документ,
а не утвержденное знание.
```

## Рассмотренные альтернативы

### Складывать raw BM25 и vector score

Плюсы: минимальный объем кода.

Минусы: нестабильное ранжирование, зависимость от шкал backend, сложная диагностика.

Решение: отклонено.

### Min-max normalization как основной fusion

Плюсы: просто привести оценки к `0..1`.

Минусы: чувствительно к выбросам и малому числу кандидатов; на короткой выдаче может усиливать шум.

Решение: использовать только как диагностическую метрику или будущую опцию.

### Позволить ИИ-боту передавать веса каналов

Плюсы: гибкость.

Минусы: трудно тестировать, трудно объяснять пользователю, выше риск случайно нестабильной выдачи.

Решение: отклонено. Бот выбирает режим и профиль, сервис применяет преднастроенные веса.

### Отложить source vector search

Плюсы: меньше privacy- и lifecycle-рисков на первом этапе.

Минусы: пользователь не получает поиск по смыслу в исходных файлах, хотя FTS/vector инфраструктура уже есть.

Решение: отклонено. Source semantic search включается сразу, но только через локальные embeddings, существующий `memory.search`, явные source-режимы и строгие правила для secret/PII/reindex/delete.

## Последствия

Положительные:

- результаты гибридного поиска становятся объяснимыми;
- agent runtime получает стабильные правила выбора режима;
- можно проводить e2e-сравнение профилей на одном наборе документов;
- `source_data` не смешивается с accepted knowledge.

Ограничения:

- RRF учитывает позицию, а не абсолютную силу совпадения;
- document-level source embeddings являются временным компромиссом до `MemorySearchSegment`;
- секреты должны блокировать индексирование конкретного документа и удалять ранее созданные index records;
- профили нужно проверять на eval-наборе и донастраивать по реальным запросам.

## Проверки реализации

- Unit-тесты нормализации и fusion.
- Unit-тесты выбора `ranking_profile` по `search_mode`.
- Unit-тест agent runtime wrapper payload.
- E2E с контролируемыми документами:
  - `precise` поднимает документ с точным термином;
  - `semantic_heavy` поднимает документ с близким смыслом;
  - `balanced` поднимает документ, найденный и FTS, и vector;
  - `source_content` возвращает `source_data` по точному содержимому файла;
  - `source_semantic` возвращает `source_data` по смысловому запросу к файлу;
  - документ с секретом не попадает в FTS/vector и создает blocker issue;
  - документ с PII индексируется и создает audit issue;
  - trace содержит `ranking_profile`, веса, RRF-позиции и каналы.
