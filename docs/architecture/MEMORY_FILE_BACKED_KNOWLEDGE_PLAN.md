# План целевой архитектуры памяти: знания в файлах, данные и индексы отдельно

Статус: проектный план.

Дата: 2026-05-22.

Связанный ADR: `docs/adr/ADR-0011-file-backed-knowledge-and-unified-search.md`.

Уточняющий ADR: `docs/adr/ADR-0013-file-only-knowledge-body.md` фиксирует строгое правило: текст знания хранится только в файле знания, а `MemoryKnowledgeItem` и индексы не являются источником текста для выдачи.

## Назначение

Этот документ описывает, как привести текущую систему памяти к целевой схеме:

- знания хранятся единообразно в файлах и версионируются через Git;
- исходные данные остаются в своих системах;
- временные сырые и безопасные слои удаляются после обработки;
- метаданные, индексы, очереди, чаты и аналитика разнесены;
- поиск и индексация едины для знаний и файлового хранилища;
- агент по умолчанию работает со знаниями, а не с исходными данными.

## Критическая оценка текущей архитектуры

### Что уже работает и остается полезным

Текущее ядро не нужно выбрасывать. В нем уже есть важные части:

- `MemoryKnowledgeItem` как главный объект сохраненного знания MVP;
- очередь явного `memory.remember`;
- кандидаты в корпоративные знания;
- `scope_tokens`, sensitivity и проверки доступа;
- надежные источники;
- полнотекстовый индекс SQLite FTS;
- графовые сущности и факты;
- ingestion документов;
- внешние коннекторы;
- аналитические модели и команды;
- secret handles вместо хранения секретов в памяти.

Эти блоки нужно не переписывать с нуля, а перенести на более ясные границы хранения.

### Главные разрывы

| Текущая часть | Проблема | Целевое состояние |
| --- | --- | --- |
| `data/db/main_vault.sqlite3` | В одной базе смешаны ядро портала, чаты, память, ingestion, очереди и аналитика. | Отдельные базы для чатов, знаний, аналитики и индексов. |
| `MemoryKnowledgeItem.text` | База выглядит как источник истины для знания. | Источник истины - файл знания в `data/knowledge_repo/`; база хранит метаданные. |
| `data/memory/chat_knowledge/` | Это файловая проекция, но не главный управляемый репозиторий знаний. | Единый Git-репозиторий знаний. |
| `MemorySnapshot` и `MemoryChunk` | По названию это "память", но по смыслу это снимки и фрагменты исходных данных. Для файловых знаний это лишний слой. | Убрать из активного пути и заменить на `MemorySearchDocument`. |
| `data/memory/safe_corpus/` | Safe corpus может стать постоянной копией обработанных данных. | Временная зона обработки с регламентом удаления. |
| `memory_reflect_chats` | Исторически смешивает обработку очереди и ночную рефлексию. | Отдельные процессы: writer worker, indexing worker, reflection worker. |
| Поиск | Поиск вырос вокруг memory chunks; нет явного разделения knowledge/source_data. | Единый сервис поиска с корпусами `knowledge` и `source_data`. |
| Граф | Граф строится вокруг chunks/snapshots, а не вокруг ясных корпусов. | Графовый индекс с `corpus_type` и только утвержденными типами сущностей. |
| Аналитика | Управляющие модели и аналитические данные лежат в основной базе. | Управление в отдельной базе, срезы в аналитическом хранилище, знания из аналитики в общем репозитории знаний. |
| Чаты | История чатов лежит в основной базе. | Чаты - отдельная база и отдельный источник данных для знаний. |

## Целевые понятия

### Данные

Данные - это исходные объекты:

- сообщение чата;
- файл на файловом сервере;
- письмо в почтовом ящике;
- объект внешнего API;
- документ в системе документооборота;
- аналитический факт или метрика.

Данные остаются в источнике. Система памяти хранит только ссылку, хэш, версию, время чтения и технические признаки, необходимые для проверки происхождения знания.

### Знание

Знание - это очищенный текст, который можно использовать в ответе агента.

Пример:

```text
В отделении терапии отчеты заведующих подаются до 10:00 каждого рабочего дня.
Источник: email:imap-medex:uidvalidity=123:uid=456, получено 2026-05-21.
```

Знание не должно быть дословной копией большого документа или письма. Оно должно передавать смысл, быть достаточно коротким, иметь ссылку на источник и область действия.

### Метаданные

Метаданные описывают знание или источник:

- идентификатор;
- область действия;
- владелец;
- источник;
- чувствительность;
- права доступа;
- статус;
- ссылки на исходные данные;
- хэши;
- даты;
- состояние индексации.

Метаданные хранятся в базе, чтобы быстро фильтровать и проверять права.

### Индексы

Индексы нужны только для поиска:

- полнотекстовый индекс;
- векторный индекс;
- графовый индекс.

Индексы не являются источником истины. Их можно перестроить из файлов знаний и доступных источников.

### Аналитические срезы

Аналитические срезы - это факты, агрегаты, метрики, сигналы и диагностические результаты. Они нужны для бизнес-аналитики, но не все они являются знаниями.

Если аналитический вывод должен стать знанием, он записывается в `data/knowledge_repo/` как обычное знание с источником `analytics`.

## Целевая структура хранения

### Файлы знаний

```text
data/knowledge_repo/
  .git/
  org/
    _summary.md
    sources/
      chat/<year>/<knowledge_id>.md
      email/<year>/<knowledge_id>.md
      file_share/<year>/<knowledge_id>.md
      external_api/<year>/<knowledge_id>.md
      analytics/<year>/<knowledge_id>.md
  users/
    <user_id>/
      _summary.md
      sources/
        chat/<year>/<knowledge_id>.md
        email/<year>/<knowledge_id>.md
        file_share/<year>/<knowledge_id>.md
        external_api/<year>/<knowledge_id>.md
        analytics/<year>/<knowledge_id>.md
  candidates/
    organization/<candidate_id>.md
  schemas/
    knowledge_file.schema.json
```

`data/knowledge_repo/` - runtime Git-репозиторий. Он не коммитится в основной репозиторий проекта.

### Формат файла знания

Формат: Markdown с YAML front matter.

```markdown
---
knowledge_id: k_01HX...
scope: personal
owner_user_id: 42
source_code: chat
source_kind: chat
source_refs:
  - kind: chat_message
    value: chat_session:123/message:456
sensitivity: internal
scope_tokens:
  - user:42
status: active
created_at: 2026-05-22T10:00:00+03:00
updated_at: 2026-05-22T10:00:00+03:00
text_hash: sha256:...
---

Пользователь предпочитает получать краткие технические ответы без лишних деталей.
```

Правила:

- в body хранится только знание;
- исходный документ, письмо или сообщение не копируется целиком;
- `source_refs` указывают, откуда знание получено;
- секреты не пишутся в body, только `<SECRET_HANDLE:...>`;
- изменение знания создает новый Git commit.

### Summary-файлы

Summary-файлы лежат рядом со знаниями:

```text
data/knowledge_repo/org/_summary.md
data/knowledge_repo/org/sources/email/_summary.md
data/knowledge_repo/users/<user_id>/_summary.md
data/knowledge_repo/users/<user_id>/sources/chat/_summary.md
```

Они нужны для быстрого контекста агента:

- общее по организации;
- по пользователю;
- по отделению;
- по источнику.

Summary - производный слой. Его можно пересобрать из файлов знаний. Он не должен скрывать ссылку на исходные знания.

### Базы данных

| Путь | Что хранит | Для чего нужен |
| --- | --- | --- |
| `data/db/main_vault.sqlite3` | Пользователи, роли, настройки, базовые доменные модели портала. | Ядро приложения. |
| `data/db/chat.sqlite3` | `ChatSession`, `ChatMessage`, вложения чата, журналы действий агента. | Хранение диалогов как источника данных, отдельно от ядра портала. |
| `data/db/knowledge_meta.sqlite3` | Метаданные знаний, события знаний, кандидаты, заявки на запись, аудит чтения. | Управление знаниями без хранения body как главного источника. |
| `data/db/analytics_control.sqlite3` | Источники аналитики, правила, запуски, сигналы, кейсы, кандидаты метрик. | Управляющий контур аналитики. |
| `data/analytics/duckdb/analytics.duckdb` | Факты, агрегаты, витрины, метрики, диагностические срезы. | Быстрый расчет и хранение аналитики. |
| `data/indexes/fulltext/search.sqlite3` | Полнотекстовые индексы по корпусам `knowledge` и `source_data`. | Поиск по словам и точным формулировкам. |
| `data/indexes/vector/` | Векторные представления знаний и источников. | Поиск по смысловой близости. |
| `data/indexes/graph/` | Сущности и связи утвержденных типов. | Поиск по связям. |
| `data/queues/*.sqlite3` | Очереди записи, индексации, внешних коннекторов, рефлексии и аналитики. | Надежная асинхронная обработка. |
| `data/cache/` | Кэш прав, кэш поисковых планов, кэш embeddings, временный кэш чтения файлов. | Ускорение, не источник истины. |

### Временные зоны обработки

```text
data/processing/
  raw_quarantine/
  safe_work/
  extraction_packets/
  cleanup_manifests/
```

Правила:

- сырые API-ответы, письма и распознанный текст живут только временно;
- после записи знания и индексации временный слой удаляется по регламенту;
- если обработка упала, объект остается в quarantine до разбора или автоматического удаления;
- временная зона не используется для ответа агенту.

## Источники данных

### Чат

Текущее состояние: `ChatSession` и `ChatMessage` лежат в основной базе.

Цель:

- перенести чаты в `data/db/chat.sqlite3`;
- считать чат источником данных;
- знания из чата писать в `data/knowledge_repo/users/<user_id>/sources/chat/`;
- при переходе в корпоративную память создавать кандидата в `data/knowledge_repo/candidates/organization/`.

Источник знания:

```text
source_ref = chat_session:<id>/message:<id>
```

### Файловое хранилище

Данные остаются на файловом сервере или в DMS.

Система хранит:

- source URI;
- путь или внешний id;
- хэш;
- версию;
- дату изменения;
- права доступа;
- статус индексации.

При обычном вопросе агент ищет в знаниях. Поиск по файлам включается:

- если пользователь прямо попросил найти в файлах;
- если знания ничего не нашли и политика источника разрешает fallback.

Для файлов со статусом `indexing_pending` degraded mode возвращает только безопасные метаданные: имя, путь/ссылку, источник и статус. Система не делает утверждений о содержимом файла, пока индекс не готов.

### Email

Письмо остается в почтовой системе или почтовом архиве.

Система памяти не хранит тело письма постоянно. Она может временно разобрать письмо в `data/processing/`, извлечь знания и удалить рабочий текст.

Знание хранит ссылку вида:

```text
source_ref = imap:<mailbox>:uidvalidity:<value>:uid:<value>
```

Если организация не гарантирует долговременное хранение почты, это нужно фиксировать в политике источника. Иначе ссылка на источник может стать непроверяемой.

### Внешний API

API-ответы не становятся постоянным слоем памяти.

Порядок:

```text
fetch API
  -> raw_quarantine на короткий срок
  -> нормализация
  -> extraction packet
  -> знание / аналитический факт / индекс
  -> удаление raw/safe временных файлов
```

Постоянно хранятся:

- system code;
- object id;
- version/etag;
- content hash;
- source link;
- водяной знак синхронизации;
- ошибки обработки.

### Аналитика

Аналитические данные делятся на два слоя:

- срезы и метрики в аналитическом хранилище;
- знания из аналитики в репозитории знаний.

Пример знания из аналитики:

```text
В мае 2026 года доля пропущенных звонков в регистратуре выросла выше порога 8% три рабочих дня подряд.
Источник: analytics_metric_snapshot:missed_calls:2026-05-01..2026-05-21.
```

Агент видит это как обычное знание, но ссылка ведет к аналитическому срезу.

## Единый сервис поиска

### Корпусы

| Корпус | Что содержит | Когда используется |
| --- | --- | --- |
| `knowledge` | Принятые знания и summary-файлы. | Всегда по умолчанию. |
| `source_data` | Индексы файлов, писем, API-объектов и чатов без постоянной копии содержимого. | Только явно или при пустом результате по знаниям. |

### Каналы поиска

| Канал | Для чего нужен | Ограничение |
| --- | --- | --- |
| Полнотекстовый | Точные названия, номера, формулировки. | Для исходных файлов не должен хранить полную копию текста. |
| Векторный | Поиск близкого смысла. | Векторы считаются чувствительным производным индексом и проверяются по правам. |
| Графовый | Вопросы о связях между сущностями. | Только утвержденные типы сущностей и связей. |

### Выбор режима без перегрузки локальной модели

Локальная модель не должна строить сложный план поиска на каждый запрос.

Вместо этого агент выбирает один из простых режимов:

- `knowledge_default`;
- `knowledge_precise`;
- `knowledge_semantic`;
- `knowledge_graph`;
- `source_explicit`;
- `source_fallback`.

Дальше сервис сам применяет веса и лимиты из контракта.

Граф подключается только если:

- пользователь спрашивает про связи, ответственность, зависимость или цепочку;
- запрос содержит известную сущность из словаря графа;
- полнотекстовый и векторный поиск нашли несколько знаний с общими сущностями;
- пользователь явно попросил показать связи.

### Возврат результата

Обычный результат:

```json
{
  "result_type": "knowledge",
  "knowledge_id": "k_...",
  "text": "...",
  "source_refs": ["..."],
  "source_kind": "email",
  "scope": "organization",
  "index_status": "ready"
}
```

Fallback по файлам:

```json
{
  "result_type": "source_data",
  "source_object_id": "...",
  "source_uri": "...",
  "source_kind": "file_share",
  "index_status": "indexing_pending",
  "warning": "Это исходный объект, а не принятое знание."
}
```

## Writer service

### Обязанности

Writer service отвечает за:

- прием заявок на запись;
- проверку прав;
- запись файлов знаний;
- Git commit;
- обновление metadata database;
- постановку задач индексации;
- создание событий аудита.

### Очередь записи

```text
data/queues/knowledge_writer.sqlite3
```

Для MVP можно использовать SQLite-очередь. Для production-нагрузки допускается переход на RabbitMQ/Celery, но контракт задания должен остаться тем же.

### Безопасная запись

Порядок записи файла:

1. Получить exclusive lock для `data/knowledge_repo/`.
2. Проверить, что целевой файл не изменился после подготовки задания.
3. Записать временный файл в том же каталоге.
4. Проверить хэш.
5. Выполнить `os.replace`.
6. Выполнить `git add`.
7. Выполнить `git commit`.
8. При настроенном удаленном репозитории выполнить pull/merge/push через controlled path.
9. При конфликте создать issue и остановить публикацию.

## Reader service

Reader service является единственным штатным путем чтения знаний.

Он проверяет:

- пользователь существует и активен;
- у пользователя есть нужные роли;
- `scope_tokens` знания разрешены;
- sensitivity не превышает доступ пользователя;
- источник знания имеет статус `trusted` или знание уже прошло модерацию;
- знание не удалено и не заменено.

Все чтения, которые попадают в ответ агента, пишутся в аудит.

## Кандидатство

### Персональное в корпоративное

Переход из персонального знания в корпоративное:

```text
personal knowledge
  -> reflection или явная заявка
  -> organization candidate
  -> аудит владельца базы знаний
  -> organization knowledge
```

Без аудита кандидат не публикуется в общую память.

### Новые типы графа

Новые типы сущностей и связей:

```text
наблюдения индексации
  -> статистика повторяемости
  -> schema candidate
  -> аудит владельца графа
  -> обновление контракта графа
  -> перестроение графового индекса
```

## Ночная рефлексия

Ночная рефлексия - отдельный процесс, а не обработчик явного `memory.remember`.

Процессы:

| Процесс | Что делает | Когда работает |
| --- | --- | --- |
| `knowledge_writer_worker` | Пишет знания из очереди. | Постоянно или часто. |
| `knowledge_index_worker` | Обновляет индексы. | Постоянно или по расписанию. |
| `knowledge_reflection_worker` | Обобщает, ищет повторы, кандидатов, устаревание. | В дешевое время. |
| `source_index_worker` | Индексирует файлы и источники. | По расписанию или после изменений. |
| `analytics_worker` | Пересчитывает метрики и сигналы. | По расписанию. |

## План миграции

### Этап 1. Зафиксировать границы

- Добавить ADR и этот план.
- Описать термины: данные, знания, метаданные, индексы, аналитика.
- Зафиксировать, что `MemoryKnowledgeItem` становится совместимой metadata-записью, а не главным источником body.

### Этап 2. Ввести файловый репозиторий знаний

- Создать формат файла знания.
- Добавить writer service.
- Добавить очередь записи.
- Реализовать запись из `memory.remember` в `data/knowledge_repo/users/<id>/sources/chat/`.
- Создать metadata mirror в базе.
- Добавить Git commit на запись.

### Этап 3. Перенести существующие знания

- Экспортировать активные `MemoryKnowledgeItem` в файлы.
- Сохранить старые ids как `legacy_memory_id`.
- Проверить хэши и source refs.
- Перевести чтение на reader service.
- Оставить старые поля в базе на переходный период.

### Этап 4. Единый поиск

## Проверки готовности

- `memory.remember` ставит задачу в очередь и после обработки создает файл знания.
- Файл знания появляется в `data/knowledge_repo/` и Git commit создан.
- Metadata database знает путь к файлу, статус, owner, source refs и индексацию.
- Поиск по умолчанию возвращает knowledge result, а не source_data.
- Поиск по файлам включается явно или как fallback.
- Результат knowledge содержит ссылку на источник.
- Файлы `indexing_pending` находятся в degraded mode только по безопасным метаданным.
- Персональное знание не становится корпоративным без candidate review.
- Чаты физически пишутся в `data/db/chat.sqlite3`.
- Управляющие модели аналитики пишутся в `data/db/analytics_control.sqlite3`.
- Временные raw/safe файлы удаляются по регламенту.
- Unit-тесты и e2e-тесты покрывают запись, чтение, поиск и fallback.

## Реализация MVP-миграции от 2026-05-22

Реализован первый рабочий срез целевой архитектуры.

Что включено:

- добавлены отдельные SQLite-базы `chat`, `knowledge_meta`, `analytics_control`;
- добавлен Django database router `apps.core.db_routers.LocalBusinessDatabaseRouter`;
- внешние ключи из разнесенных баз к пользователям и чатам переведены в логические связи без физического ограничения SQLite;
- `MemoryKnowledgeItem` получил metadata-поля `source_refs`, `source_code`, `source_kind`, `knowledge_file_path`, `knowledge_file_hash`, `knowledge_file_commit`, `index_status`;
- добавлен файловый слой `apps.memory.knowledge_files` с Markdown + YAML front matter, atomic replace, lock и Git commit;
- `memory.remember` по-прежнему только ставит задачу в очередь, а `knowledge_writer_worker` обрабатывает очередь и пишет файл знания;
- `memory.search` поддерживает простые режимы поиска и по умолчанию отдает знания, а не исходные данные;
- чаты пишутся в `data/db/chat.sqlite3`, память - в `data/db/knowledge_meta.sqlite3`, управляющая аналитика - в `data/db/analytics_control.sqlite3`;
- добавлены команды миграции legacy-данных и проверки файлов знаний;
- `memory_reflect_chats` оставлен как совместимый alias только для обработки очереди, ночная рефлексия вынесена в `knowledge_reflection_worker`;
- добавлена команда `memory_file_backed_e2e` для проверки полного пути `queued remember -> файл знания -> индекс -> поиск`.

Команды первичной миграции runtime:

Остаточные ограничения MVP:

- перенос аналитических срезов в DuckDB не реализован в этом срезе;
- графовый поиск временно не участвует в выдаче памяти, детальная стратегия графа будет проработана отдельно;
- удаление старых runtime-данных из уже созданных локальных БД не выполняется автоматически.

Дополнение после реализации `memory-snapshot-chunk-removal`:

- активный путь чата, document ingestion, external connector handoff и `memory.search` использует `MemorySearchDocument`;
- `MemorySnapshot`, `MemoryChunk` и привязанный к ним `MemoryGraphFact` удалены из текущей модели Django миграцией `apps/memory/migrations/0007_remove_memorychunk_snapshot_and_more.py`;
- графовый поиск временно выключен до отдельной стратегии графового индекса.
