# Уведомления: PWA-first и опциональный Tauri-клиент

## Статус

Архитектура принята. Этап 1 PWA/browser notifications реализован как MVP. Этап 2 Tauri реализован как первичный optional client и ожидает техническую проверку сборки на целевых ОС.

Архитектурное решение: `docs/adr/ADR-0026-pwa-first-and-optional-tauri-notifications.md`.

Проектный план: `docs/architecture/PWA_AND_TAURI_NOTIFICATIONS_PLAN.md`.

Workflow-блок: `workflow/active/desktop-notifications-pwa-tauri/`.

## Методологическая заметка

Серверная очередь уведомлений отделяет факт события от способа доставки. Это позволяет сначала сделать PWA-канал для всех пользователей, а позже подключить Tauri-клиент к той же очереди без переписывания доменной логики заявок.

## Цель

Ввести пользовательские уведомления об изменениях в системе.

Первый срез:

- PWA и браузерные уведомления для открытого портала;
- центр уведомлений внутри портала;
- без стороннего Web Push.

Второй срез:

- опциональное Tauri-приложение в трее для Windows/Linux;
- только для пользователей, которым нужен постоянный фоновый режим.

## Пользовательская ценность

- Пользователь быстрее узнает о новой или измененной заявке.
- Инженер видит назначение и комментарии без ручного обновления доски.
- Руководитель не теряет важные события при работе в портале.
- Система сохраняет local-first режим на первом этапе.
- Фоновый режим остается доступным как отдельная установка, а не навязывается всем.

## Область работ

### Этап 1. PWA без стороннего Web Push

Реализовано:

- новое Django-приложение `apps.notifications`;
- модели событий, получателей, настроек и browser clients;
- API получения и отметки уведомлений текущего пользователя;
- центр уведомлений в базовом layout;
- PWA manifest и service worker;
- запрос browser notification permission только по явному действию пользователя;
- клиентский polling с cursor;
- безопасные уведомления по событиям заявок;
- пользовательская инструкция по включению/отключению;
- unit, integration и e2e-проверки.

Не включено:

- Push API, VAPID, browser push endpoints;
- сторонние notification SaaS;
- гарантированные системные уведомления при закрытом браузере;
- WebSocket/Channels;
- отдельное приложение в трее.

Остаточные ограничения этапа 1:

- авторизованный Playwright UI-сценарий требует `E2E_USERNAME` и `E2E_PASSWORD` на стенде;
- production browser notifications требуют HTTPS, текущий HTTP-only профиль годится только для центра уведомлений и локальной разработки;
- массовый расчет получателей сейчас выполняется через существующие политики видимости и может потребовать оптимизации на большом количестве пользователей.

### Этап 2. Опциональный Tauri-клиент

Реализовано:

- проект Tauri-клиента в `clients/desktop-notifier/`;
- регистрация устройства через одноразовый код;
- device token с ограниченным scope;
- tray icon, меню, автозапуск;
- получение уведомлений через внутренний API;
- открытие URL портала в браузере;
- инструкции сборки и установки для Windows/Linux.

Не включено:

- хранение бизнес-данных заявок на клиенте;
- офлайн-редактирование заявок;
- замена веб-портала;
- обязательная установка всем пользователям.

Остаточные ограничения этапа 2:

- сборка Tauri требует установленных Node.js, Rust и системных зависимостей Tauri;
- tray и системные уведомления Linux зависят от окружения рабочего стола;
- Windows production rollout требует отдельной подписи установщика;
- автообновление клиента не реализовано;
- возможны дубли при одновременной работе PWA и Tauri.

Предполагаемый write scope этапа 2:

- отдельная директория клиента после выбора места, например `clients/desktop-notifier/` или `services/desktop_notifier/`;
- API device tokens в `apps.notifications`;
- deployment/guides для сборки.

## Архитектурные ограничения

- Получателей вычисляет сервер.
- PWA-клиент не фильтрует чужие события.
- Системное уведомление содержит только безопасный краткий текст.
- Если браузер не дал permission, центр уведомлений продолжает работать.
- Без стороннего Web Push нет фоновых уведомлений при закрытом браузере.
- Tauri-клиент использует отдельный device token, а не пароль пользователя.
- Все временные e2e-артефакты пишутся в `.local/`.

## Definition of Ready

Готово к реализации, когда:

- ADR-0026 принят;
- согласовано место размещения `apps.notifications`;
- подтверждено, что этап 1 не использует Web Push;
- выбран базовый transport: polling в MVP;
- согласованы первые типы событий заявок;
- согласованы тексты безопасных уведомлений;
- определены браузеры пилота: Edge/Chrome на Windows, выбранный браузер на Linux;
- подтвержден HTTPS-профиль production.

## Acceptance этапа 1

- Пользователь видит центр уведомлений в портале.
- Пользователь может включить и выключить браузерные уведомления.
- Permission prompt вызывается только после явного действия.
- Новая заявка или назначение создает уведомление только разрешенным получателям.
- Пользователь не видит чужие уведомления через API и UI.
- Клик по уведомлению открывает портал по безопасной ссылке.
- При закрытом портале события сохраняются и отображаются после следующего входа.
- При запрете browser permission центр уведомлений остается рабочим.
- Документация объясняет ограничение первого этапа без Web Push.
- Unit, integration и e2e-проверки выполнены или остаточный риск зафиксирован.

## Acceptance этапа 2

- Пользователь подключает Tauri-клиент через одноразовый код.
- Устройство отображается в профиле и может быть отозвано.
- Клиент показывает системное уведомление при закрытом браузере, если клиент запущен.
- Клик открывает портал в системном браузере.
- Токен не дает доступа к общему API портала.
- Клиент хранит токен в Stronghold vault, а сервер хранит только HMAC-хеш токена.
- Логи клиента не содержат секретов и чувствительного текста заявок.
- Есть инструкции сборки и установки Windows/Linux.
