# ADR-0026: PWA-first уведомления и опциональный Tauri-клиент

## Статус

Accepted

## Дата

2026-06-03

## Контекст

Порталу нужен канал уведомлений для конечных пользователей: новая заявка, изменение заявки, комментарий, назначение исполнителя и другие события системы. Пользователь должен увидеть короткое сообщение и перейти по ссылке на портал.

Проект развивается как local-first Django monorepo. Это накладывает ограничения:

- сервер остается источником истины по правам доступа, состоянию уведомлений и ссылкам;
- первый этап не должен зависеть от сторонних web push сервисов браузеров;
- чувствительные данные не должны попадать в текст системных уведомлений;
- решение должно работать для всех пользователей через существующий портал;
- отдельное приложение в трее допустимо как второй этап, но не должно становиться обязательным для всех;
- Windows/Linux deployment уже различается, поэтому первый этап должен минимально менять контур развертывания.

Web Push через Push API обычно использует push-сервис браузера и VAPID-ключи. Это дает фоновые уведомления при закрытом сайте, но вводит внешнюю инфраструктуру браузера. Для текущего local-first этапа это не принимается.

## Решение

Принять двухэтапную архитектуру уведомлений.

### Этап 1. PWA и браузерные уведомления без стороннего Web Push

Первый этап реализуется для всех пользователей внутри портала:

- создать серверный домен уведомлений в Django;
- добавить центр уведомлений в веб-интерфейс;
- сделать портал installable PWA через `manifest.webmanifest` и `service-worker.js`;
- показывать системные уведомления браузера через Notifications API только после явного действия пользователя;
- доставлять новые уведомления в открытую страницу портала или запущенное PWA-окно через HTTP-опрос, с возможностью перейти к SSE после проверки deployment-контура;
- сохранять уведомления на сервере, чтобы пользователь не терял события, если портал был закрыт;
- не использовать Push API, VAPID и внешние browser push endpoints в первом этапе.

Ограничение этапа 1 является осознанным: если браузер и PWA полностью закрыты, системное уведомление не появится. При следующем открытии портала пользователь увидит накопленные уведомления в центре уведомлений.

### Этап 2. Опциональный Tauri-клиент для трея

Второй этап реализуется только для пользователей, которым нужен постоянный фоновый агент:

- отдельное Tauri 2 приложение для Windows/Linux;
- иконка в трее и меню быстрых действий;
- системные уведомления ОС;
- открытие ссылки в системном браузере;
- scoped device token, привязанный к пользователю и устройству;
- получение уведомлений от портала через внутренний HTTP/SSE API без стороннего Web Push;
- возможность отозвать устройство в профиле пользователя или админке.

Tauri-клиент остается тонким: он не хранит бизнес-данные заявок, не принимает решения о правах доступа и не заменяет веб-портал.

## Архитектурные правила

### Серверный домен уведомлений

Добавить отдельное приложение `apps.notifications`.

Оно отвечает за:

- хранение событий уведомлений;
- хранение получателей и состояния `new`, `seen`, `read`, `dismissed`;
- пользовательские настройки уведомлений;
- регистрацию PWA/browser clients без внешнего Web Push;
- регистрацию Tauri-устройств на втором этапе;
- API выдачи уведомлений текущему пользователю;
- audit минимальных delivery/ack событий.

Домен `apps.notifications` не должен знать бизнес-детали заявок. Он принимает уже подготовленный payload: тип события, заголовок, короткий текст, ссылку, важность, источник и id объекта.

### Доменные производители событий

Прикладная логика, которая решает, что именно означает событие заявки и кому оно должно быть показано, остается в `apps.workorders`.

Примеры:

- `workorders.notification_events` или аналогичный модуль создает notification payload;
- получатели вычисляются через существующие политики видимости и роли;
- `apps.notifications` получает уже сформированные события и получателей.

Так сохраняется граница доменов: общий домен уведомлений не превращается в приложение-свалку.

### Приватность

Текст системного уведомления должен быть коротким и безопасным:

```text
Новая заявка №123
Открыть в портале
```

Не включать:

- персональные медицинские данные;
- длинные описания;
- вложения;
- внутренние комментарии с возможными чувствительными деталями;
- диагностические сведения, секреты, пути к файлам или имена сетевых ресурсов.

Полные детали открываются только в портале после обычной проверки прав доступа.

### Transport

Для этапа 1:

- базовый вариант: HTTP-опрос из открытой страницы или PWA-окна;
- cursor-based API: клиент передает последний полученный курсор;
- сервер возвращает только уведомления текущего пользователя;
- состояние прочтения меняется отдельными POST-запросами;
- SSE допускается как следующий шаг, если deployment поддерживает долгоживущие HTTP-соединения без ухудшения работы IIS/Linux.

Для этапа 2:

- Tauri использует отдельный device API;
- токен устройства имеет минимальные права: читать свои уведомления и отмечать их прочитанными;
- токен можно отозвать;
- секреты хранятся в защищенном хранилище ОС, а не в обычном файле.

Реализационная заметка 2026-06-03: первичный Tauri-клиент хранит device token в Stronghold vault и не пишет его в обычный JSON/config файл. Если production rollout потребует обязательной интеграции именно с системным keychain/credential manager, это должно быть оформлено отдельным hardening-срезом без изменения серверной модели токенов.

## Рассмотренные альтернативы

### Web Push на первом этапе

Плюсы:

- фоновые уведомления при закрытом портале;
- стандартный браузерный путь для PWA push.

Минусы:

- зависимость от внешнего push-сервиса браузера;
- сложнее объяснить local-first privacy model;
- требуется VAPID и хранение browser push subscriptions;
- поведение зависит от браузера, политики организации и сетевого доступа.

Отклонено для первого этапа. Может быть пересмотрено только отдельным ADR, если появится разрешенный корпоративный push-контур или изменятся требования.

### Tauri-клиент сразу для всех

Плюсы:

- настоящий трей;
- стабильнее для фоновых уведомлений;
- не зависит от открытой вкладки.

Минусы:

- нужен установщик и сопровождение клиента;
- появляется отдельный runtime и обновления;
- выше нагрузка на поддержку пользователей;
- избыточно для тех, кто постоянно работает в портале.

Отклонено как обязательный первый этап. Принято как опциональный второй этап.

### WebSocket/Channels как первый transport

Плюсы:

- двусторонняя связь;
- низкая задержка.

Минусы:

- требует ASGI/WebSocket deployment;
- может потребовать Redis/channel layer;
- для уведомлений от сервера к клиенту избыточен в MVP;
- повышает сложность Windows/IIS контура.

Отклонено для MVP. SSE или WebSocket можно рассматривать позже после измерений и проверки deployment.

### Только email

Плюсы:

- привычный корпоративный канал;
- не требует PWA или клиента.

Минусы:

- задержки, шум и сложная обратная связь;
- неудобно для оперативных заявок;
- не решает сценарий "увидел и сразу открыл портал".

Не принимается как основной канал, но может быть дополнительным escalation-каналом позже.

## Последствия

Положительные:

- первый этап быстро доступен всем пользователям портала;
- сохраняется local-first позиция без стороннего Web Push;
- серверная очередь уведомлений станет общей основой и для PWA, и для Tauri;
- Tauri можно добавить без перепроектирования домена уведомлений.

Отрицательные:

- PWA без Web Push не показывает системные уведомления, если портал полностью закрыт;
- часть пользователей может воспринимать PWA как "не настоящее приложение";
- браузерные разрешения и системные настройки уведомлений остаются внешним фактором;
- нужно аккуратно объяснить пользователю, когда уведомления работают.

Новые требования:

- добавить пользовательскую и операторскую документацию по включению уведомлений;
- добавить e2e для включения уведомлений и fallback без разрешения браузера;
- добавить проверки privacy-safe текста уведомлений;
- перед реализацией Tauri обновить deployment-документацию и добавить отдельные сборочные инструкции.
