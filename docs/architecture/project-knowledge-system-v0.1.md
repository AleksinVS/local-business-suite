# Система проектных знаний для ИИ-агентов (project knowledge base)

**Статус:** проект (Proposed) · **Версия:** 0.2 · **Дата:** 2026-07-04
**Базовый документ:** [hybrid-knowledge-architecture-v0.5.md](hybrid-knowledge-architecture-v0.5.md) (текст версии 0.6)
**Назначение:** адаптация гибридной концепции управления знаниями под узкую задачу — локальная база знаний **о самом проекте** для агентов-архитекторов и агентов-разработчиков.

Документ спроектирован только от текста базовой концепции; сверка с другими ADR выполняется на этапе принятия.

**Изменения v0.2 (по уточнению постановки):**
- зафиксирован главный потребитель — сами агенты («агент пишет для агента»); человек читает канон только при ревью и ознакомлении (§1);
- стиль страниц переориентирован с прозы на плотный структурированный формат; введены рекомендуемые скелеты разделов по типам концептов (§5.2);
- снят гардрейл «агент не переписывает прозу человека»: страницы принадлежат агентам, свободная реструктуризация — часть обслуживания (§7.3, §9);
- фоновый контур качества оформлен явно: уровень 1 — детерминированный lint с детекторами при каждом sync, уровень 2 — периодическая сессия-уборщик (§7.4–7.5);
- добавлена детекция устаревания страниц относительно источников (source-drift) — адаптация «детекции устаревания» из контракта деривации концепции (§7.4);
- каденция обслуживания переведена из открытых вопросов в стартовые пороги триггеров (§7.5).

## 1. Задача и границы

**Цель.** Дать агентам, работающим над проектом, систему знаний о нём: устройство подсистем, принятые паттерны, ограничения среды, неочевидные факты («грабли»), связи между компонентами. Требования: простая, локальная, быстрая, полная, индексируемая.

**Потребители.** Главный потребитель — сами ИИ-агенты (архитектор, разработчик, проверяющий): **агент пишет для агента.** Человек (владелец) обращается к канону только в двух редких случаях: контроль (ревью диффов) и ознакомление. Все работают на одной машине под одной учётной записью.

Следствия «агент пишет для агента»:

- человекочитаемость — второстепенное требование: стиль страниц оптимизируется под плотность и машинную предсказуемость (§5.2), достаточный уровень — «читаемо при ревью диффа»;
- человек практически не правит канон: машинерия согласования ручных правок сведена к страховочному минимуму (§8), а агентам разрешена свободная реструктуризация страниц без человеческого гейта (§7.5).

**Не-цели (сознательно исключено):**

- слой доступа, аутентификация, классификация, компартменты, деклассификация, egress (§9 концепции) — модель угроз пуста: единственный человек-владелец, агенты работают от его имени;
- data plane / data store для наблюдений и time-series (§3.1 концепции) — в задаче «знания о проекте» наблюдений нет;
- PostgreSQL и любые серверные СУБД, веб-сервисы, очереди на выделенной инфраструктуре;
- отдельный сервис-оркестратор — семантическую работу выполняет сам рабочий агент.

## 2. Анализ практик и следствия для дизайна

| Практика | Следствие для дизайна |
|---|---|
| LLM-wiki (Karpathy) / OKF: концепт = markdown-файл с frontmatter, идентичность = путь | Канон знаний — markdown-бандл в git, совместимый с OKF на уровне контейнера |
| Прогрессивное раскрытие (OKF `index.md`, соглашение `llms.txt`, паттерн memory-bank агентных сред) | Дешёвая точка входа в сессию: агент читает только `index.md`, дальше — точечно |
| Агентный поиск: итеративный лексический поиск (BM25/подстроки) агент использует эффективнее «одноразового» векторного, т.к. сам переформулирует запросы | Старт на FTS5/BM25 без эмбеддингов; вектор — отложенное расширение по измеренному триггеру (см. §10) |
| SQLite как локальный стандарт: нулевое администрирование, один файл, FTS5 встроен в стандартную библиотеку Python | Весь производный слой (индекс + граф + очередь) — один файл SQLite |
| Детерминированные производные слои: пересборка с нуля — штатная операция, обнаружение изменений по content-hash | Проекцию можно снести в любой момент; никакого состояния, которое жалко потерять |
| Канон читают в основном LLM: плотный структурированный текст дешевле по токенам и предсказуемее прозы | Телеграфный стиль страниц, фиксированные скелеты разделов по типам концептов (§5.2) |
| Инструменты без зависимостей живут дольше | CLI на стандартной библиотеке Python (+ PyYAML для frontmatter), без каркасов и служб |

## 3. Инварианты (адаптированные)

1. **Один канон — git-репозиторий.** Вики-бандл — канон знаний; остальные файлы репозитория (код, ADR, документация) играют роль raw sources: провенанс указывает в них через `путь@коммит` (иммутабельность обеспечивает git). Отдельные каноны data store и raw sources из концепции здесь не нужны.
2. **LLM пишет в markdown, а не в БД.** Прямой записи агента в SQLite нет ни в одном сценарии.
3. **SQLite — проекция.** Полностью пересобирается из бандла; результат детерминирован. Смена токенизатора FTS (в перспективе — эмбеддинг-модели) = плановая полная переиндексация.
4. **Обнаружение изменений по content-hash, не по `mtime`.**
5. **Read-path дисциплина:** индекс → граф → чтение только выбранных страниц. Сканирование тел страниц ради релевантности запрещено.
6. **Состояние производного слоя не пишется в канон** (инвариант №9 концепции, применён строже): очередь ре-консолидации, хэши, статусы индексации живут только в SQLite. Frontmatter несёт лишь присущие концепту поля и редакционный `lifecycle`.
7. **Схемы меняются только человеком.** Словари типов концептов и рёбер — файлы в бандле; агент эмитит только против утверждённого словаря, расширение — через правку словаря с ревью владельца (упрощённый примитив §7.3 концепции).
8. **OKF — цель, не блокер.** Отступления фиксируются (раздел 9 этого документа) и оформляются ADR при принятии системы.

## 4. Архитектура

```
Git-репозиторий — единый канон
 ├─ код, ADR, docs, contracts        ← «raw sources»: сюда указывает провенанс (путь@коммит)
 └─ docs/knowledge/                  ← вики-бандл (OKF-контейнер): страницы-концепты
         │
         │  kb sync — детерминированная материализация (без LLM)
         ▼
 .local/knowledge/kb.sqlite3         ← производный слой (пересобираемый)
    ├─ chunks + FTS5 (BM25)          — извлечение на уровне раздела
    ├─ edges                         — граф типизированных рёбер
    └─ meta + очередь reconcile      — состояние проекции и «грязные» страницы
         ▲
         │  query / lint (детерминированные операции — CLI `kb`)
 Рабочий агент (Claude Code и др.) = оркестратор:
 ingest и семантическая ре-консолидация — протокол в AGENTS.md
```

Ключевая адаптация относительно концепции: **отдельного агента-оркестратора нет.** Три операции распределены так:

- **ingest** — рабочий агент пишет/обновляет страницы бандла по ходу задачи (протокол — §7.3), затем вызывает `kb sync`;
- **query** — CLI: гибридный поиск по индексу + обход графа, выдача списка страниц и релевантных разделов; чтение тел — агентом, точечно;
- **lint** — двухуровневый контур качества: детерминированные проверки и детекторы при каждом sync (уровень 1), семантическое обслуживание — периодической сессией-уборщиком по очереди (уровень 2, §7.4–7.5).

## 5. Канон: бандл и страница

### 5.1 Раскладка

```
docs/knowledge/
  index.md                  # навигация, прогрессивное раскрытие; поддерживается агентами
  schema/
    concept-types.yaml      # контролируемый словарь типов концептов
    edge-types.yaml         # контролируемый словарь типов рёбер
    templates/
      <type>.md             # рекомендуемый скелет разделов для каждого типа концепта
  concepts/
    <домен>/<концепт>.md    # неглубокая иерархия по доменам, файл = концепт
```

- **Идентичность концепта = путь** относительно бандла без `.md` (соглашение OKF): `concepts/core/integrations-sdk`.
- **Переименование рвёт ссылки**, поэтому не поощряется. При необходимости: на старом пути остаётся страница-заглушка с ребром `superseded` → новый путь и `lifecycle: superseded`.
- `log.md` как файл **не материализуется**: хронологию отдаёт `kb log` из git-истории по запросу (см. отступления, §9).

### 5.2 Страница-концепт

```yaml
---
type: component               # из schema/concept-types.yaml (обязательное, OKF)
title: Общий SDK интеграций   # обязательное
description: Транспорт, аутентификация и обработка ошибок для внешних систем  # одна строка, обязательное
timestamp: 2026-07-04T12:00:00Z   # последнее содержательное обновление, обязательное
lifecycle: current            # draft | current | superseded (редакционный статус)
tags: [integrations, core]
sources:                      # первоисточники синтеза страницы (пути в репозитории)
  - docs/adr/ADR-00XX-....md
  - apps/core/integrations/
relations:                    # типизированные рёбра — эмитятся агентом в момент синтеза
  - type: part_of
    target: concepts/core/core-app
  - type: implements
    target: concepts/decisions/integration-sdk-decision
    provenance: docs/adr/ADR-00XX-....md@a1b2c3d   # путь[@коммит][#якорь], рекомендуется
---

# Общий SDK интеграций

## Назначение и границы
...проза-синтез, разбитая на разделы — уровень чанкинга для индекса...

## Неочевидные ограничения
...
```

Четыре обязательных поля (`type`, `title`, `description`, `timestamp`) — по замечанию концепции о референс-парсере OKF. Остальные ключи — producer-defined, что OKF явно разрешает.

**Правила содержания (агент пишет для агента):**

- **Стиль — телеграфный и плотный:** списки, таблицы, короткие утверждения; повествовательная проза не нужна. Оптимизируется стоимость чтения в токенах, а не литературность; достаточная человекочитаемость — уровень ревью диффа.
- **Предсказуемая структура:** для каждого типа концепта — рекомендуемый скелет разделов с фиксированными H2 (`schema/templates/<type>.md`). Пример для `component`: `Назначение и границы` / `Устройство` / `Зависимости и контракты` / `Ограничения и грабли` / `Проверки`. Одинаковые заголовки дают адресное чтение (`kb page <id> --section`) и стабильный чанкинг.
- Страница — **синтез, а не копия**: не дублировать текст ADR/README, а давать выжимку следствий и ссылку-источник в `sources`. Числа и точные сигнатуры не «замораживать» в тексте — ссылаться на код.
- Ориентир размера — до ~200 строк; крупные темы разрезать по `part_of`.
- **Язык — русский**, как остальная документация проекта (упрощает редкое ревью владельцем); перевод корпуса на английский ради токен-экономии — возможная оптимизация, решение за владельцем (§12).
- Рёбра проставляются **в момент синтеза** (агент знает, *почему* связывает A и B); пост-фактум восстановить тип связи без повторного LLM-прохода нельзя (§6 концепции).

### 5.3 Стартовые словари

**Типы концептов** (`schema/concept-types.yaml`):

| Тип | Что описывает |
|---|---|
| `domain` | предметная область / подсистема верхнего уровня |
| `component` | приложение, модуль, сервис |
| `integration` | внешняя система и контур обмена с ней |
| `pattern` | принятый паттерн или конвенция (со ссылкой на решение) |
| `constraint` | ограничение среды, deployment, регуляторики |
| `insight` | неочевидный факт, причинность, «грабли» |
| `source-summary` | сводка объёмного первоисточника со ссылками на порождённые концепты |

**Типы рёбер** (`schema/edge-types.yaml`), направление фиксировано определением:

| Тип | Читается как | Симметрия |
|---|---|---|
| `part_of` | A входит в состав B | нет |
| `depends_on` | A использует B | нет |
| `interacts_with` | A обменивается с B | да |
| `implements` | A реализует решение/контракт B | нет |
| `constrained_by` | на A действует ограничение B | нет |
| `supersedes` | A замещает устаревший B | нет |
| `contradicts` | A противоречит B (сигнал для lint) | да |
| `relates_to` | слабая связь, когда ничто выше не подходит | да |

Обратные рёбра в страницы не пишутся — обратный индекс строит материализация. `relates_to` — резерв последней очереди: lint следит, чтобы его доля не росла (признак дрейфа таксономии). Расширение словарей — только правкой yaml-файла человеком (инвариант №7).

## 6. Производный слой: SQLite

Один файл `.local/knowledge/kb.sqlite3` (в `.local/` — по протоколу репозитория для локальных агентных артефактов; файл не версионируется и пересобирается на любой машине).

Набросок схемы:

```sql
CREATE TABLE meta (key TEXT PRIMARY KEY, value TEXT);
-- schema_version, fts_tokenizer, last_sync_commit, last_full_rebuild

CREATE TABLE pages (
  id          TEXT PRIMARY KEY,   -- путь без .md относительно бандла
  type        TEXT, title TEXT, description TEXT,
  lifecycle   TEXT, timestamp TEXT,
  front_hash  TEXT,               -- sha256 frontmatter
  body_hash   TEXT                -- sha256 тела
);

CREATE TABLE chunks (
  chunk_id  INTEGER PRIMARY KEY,
  page_id   TEXT NOT NULL REFERENCES pages(id),
  heading   TEXT,                 -- путь заголовков: "H1 > H2 [> H3]"
  position  INTEGER,
  text      TEXT
);

CREATE VIRTUAL TABLE chunks_fts USING fts5(
  text, heading, page_id UNINDEXED,
  content='chunks', content_rowid='chunk_id',
  tokenize='trigram'
);

CREATE TABLE edges (
  src TEXT NOT NULL, type TEXT NOT NULL, dst TEXT NOT NULL,
  provenance TEXT,
  PRIMARY KEY (src, type, dst)
);
CREATE INDEX edges_by_dst ON edges(dst);

CREATE TABLE reconcile_queue (
  page_id     TEXT PRIMARY KEY,
  reason      TEXT,               -- prose-changed | source-drift | dup-candidate | contradiction
  enqueued_at TEXT,
  attempts    INTEGER DEFAULT 0
);
```

**Решения:**

- **Чанкинг** — по `##` (H2); раздел длиннее ~2000 символов дорезается по H3. `description` из frontmatter индексируется нулевым чанком страницы. Перекрытия (overlap) нет: границы разделов семантичны, а агент при попадании читает раздел целиком.
- **Токенизатор `trigram`:** языконезависимость (русский и английский вперемешку, идентификаторы кода, поиск подстрок), BM25 работает поверх. Цена — индекс в 3–4 раза больше текста; для корпуса в сотни страниц несущественно. Выбор фиксируется в `meta`; смена токенизатора = полная переиндексация (инвариант №3). Запасной вариант — `unicode61`, если замер на реальном корпусе покажет проблемы качества.
- **Граф** — таблица рёбер + рекурсивные CTE для обхода на 1–2 шага. NetworkX не подключается, пока не понадобится кластерный анализ в lint (упрощение относительно §10 концепции).
- **Зависимости CLI:** стандартная библиотека Python (sqlite3, hashlib) + PyYAML. Ничего более.

**Ориентиры производительности** (критерии приёмки, §11): дельта-`sync` < 1 с; полная пересборка < 10 с на сотнях страниц; `query` < 200 мс.

## 7. Операции

### 7.1 CLI `kb` (детерминированные операции)

| Команда | Что делает |
|---|---|
| `kb sync [--full]` | pull-reconciler: материализация изменённых страниц; `--full` — снос и пересборка проекции |
| `kb query "<текст>" [--k 10] [--hops 1] [--type T] [--json]` | поиск: FTS5/BM25 по чанкам → агрегация в страницы → расширение соседями по графу; выдача — страницы с релевантными разделами и рёбрами |
| `kb neighbors <id> [--hops N] [--type T]` | окрестность концепта в графе |
| `kb page <id> [--section "<заголовок>"]` | вывод страницы или раздела |
| `kb lint` | контур качества уровня 1: валидатор, предупреждения, детекторы (§7.4); выполняется и автоматически в составе `sync` |
| `kb queue [--done <id>]` | очередь семантической ре-консолидации; `--done` — снять вручную после разбора |
| `kb log [<id>]` | хронология изменений бандла/страницы из git-истории |

Место размещения CLI в дереве репозитория — вопрос реализации (по действующим правилам структуры); требование — запуск без поднятия приложения, время старта незаметное.

### 7.2 query: read-path

1. FTS5 (BM25) по чанкам → топ-20 чанков → агрегация по страницам (лучший чанк = оценка страницы) → до 10 страниц-кандидатов.
2. Обход графа на 1 шаг от кандидатов по структурным рёбрам (`part_of`, `depends_on`, `implements`, `interacts_with`) — соседи добавляются с указанием типа связи.
3. Агент читает **только выбранные** страницы/разделы (`kb page`).
4. Если ответ породил новое знание — агент приземляет его страницей (ingest).

### 7.3 ingest: write-path (протокол агента)

Когда создавать/обновлять страницу:

- завершён значимый блок работы (новый модуль, интеграция, изменение границ) → обновить страницы затронутых концептов и рёбра;
- принято архитектурное решение → страница `pattern`/`constraint` со ссылкой на ADR в `sources` (ADR остаётся каноном решения, страница — его проекцией в граф знаний);
- обнаружен неочевидный факт, стоивший времени → страница `insight`;
- изучен объёмный внешний документ → `source-summary` + рёбра к затронутым концептам.

Процедура: правка markdown → при добавлении страниц обновить `index.md` → `kb sync` → `kb lint`. Рёбра — только против словаря; провенанс — `путь[@коммит]` на подтверждающий артефакт репозитория.

Ограничения: агент не редактирует `schema/*.yaml` без явного поручения владельца; прямая запись в `kb.sqlite3` запрещена в любом сценарии. **Страницы принадлежат агентам:** переписывание, слияние и разрезание страниц разрешены без согласования — качество защищают git-история, ревью диффов владельцем по его желанию и контур lint, а не неприкосновенность текста.

### 7.4 Контур качества, уровень 1: детерминированный lint (при каждом sync)

Выполняется автоматически финальным шагом `kb sync` (и отдельно командой `kb lint`). Без LLM, секунды. Три класса результатов:

- **Ошибки (валидатор, блокируют операцию записи):** типы концептов/рёбер вне словарей; сломанный синтаксис frontmatter; отсутствие обязательных полей. Агент обязан исправить до завершения задачи.
- **Предупреждения (отчёт):** битые `target` рёбер; страницы, недостижимые из `index.md` (сироты); ссылки на `superseded`-страницы; пустые `description`; дубли `title`; страницы сверх ориентира размера; доля `relates_to` выше порога; рёбра-утверждения (`contradicts`, `supersedes`) без провенанса.
- **Детекторы (пополняют `reconcile_queue` для уровня 2):**
  - `prose-changed` — тело страницы изменилось, блок `relations:` — нет (раздельные хэши front/body): объявленные рёбра могли отстать от текста;
  - `source-drift` — источники из `sources:` менялись после `timestamp` страницы (сравнение с датой последнего коммита git, затронувшего путь): страница могла устареть относительно кода/ADR. Это адаптация «детекции устаревания» из контракта деривации концепции (§3.1): роль `derived_from` играет `sources:`, роль новых данных — новые коммиты;
  - `dup-candidate` — дешёвые кандидаты в дубли: пересечение триграмм заголовков, общие рёбра, взаимные попадания в FTS-выдачу.

### 7.5 Контур качества, уровень 2: сессия-уборщик (периодический агентный проход)

Аналог фонового `reconcile`/`lint` базовой концепции (§7, §7.1). Отдельная плановая агентная сессия; вход — `kb queue` и предупреждения уровня 1:

- `prose-changed` → перечитать страницу, заново вывести рёбра из текста, обновить `relations:`;
- `source-drift` → сверить страницу с изменившимися источниками; актуализировать или пометить `superseded`;
- `dup-candidate` → подтвердить или отклонить; слияние: одна страница поглощает содержимое, вторая становится заглушкой с `supersedes`;
- `contradiction` → проверить пары `contradicts` и искать новые противоречия только на изменённых узлах + их окрестности в графе (O(изменённого), не O(n²) — §7 концепции);
- гигиена навигации: обновить `index.md`, разобрать сирот, вычистить битые ссылки из предупреждений.

Уборщику разрешена **свободная реструктуризация** (слияние, разрезание, переписывание страниц) без человеческого гейта — следствие «агент пишет для агента»; контроль владельца — ревью диффов по желанию. Разобранные позиции снимаются из очереди (`kb queue --done` либо автоматически при следующем sync, когда хэши обновились).

**Триггеры запуска** (стартовые значения, уточняются по факту): очередь ≥ 10 позиций; или ≥ 20 изменённых страниц с прошлого прохода (`kb queue` показывает оба счётчика); и не реже раза в неделю при активной разработке. Запуск — владельцем вручную или планировщиком агентной среды; постоянный демон не нужен.

## 8. Консистентность и жизненный цикл

**Pull-reconciler (`kb sync`).** Проекция хранит хэши материализованных страниц; sync сравнивает их с текущим состоянием файлов (content-hash, не `mtime`) и пересобирает только изменённое. Запуск: в начале сессии агента и после операций записи. Git-хуки не используются даже как ускоритель — необязательная сложность при таком размере корпуса; пропущенный запуск ничего не ломает (самовосстановление при следующем sync).

**Две дорожки пересборки** (§7.1 концепции, сохранены):

1. **Детерминированная — сразу:** чанки, FTS, рёбра из объявленных `relations:` — при каждом sync, без LLM, идемпотентно.
2. **Семантическая — фоном, агентом:** детекторы уровня 1 (§7.4) наполняют `reconcile_queue`; разбирает её сессия-уборщик (§7.5). Основные причины устаревания здесь — не ручные правки человека (их почти нет), а дрейф относительно источников (`source-drift`), накопление дублей и небрежность самих агентов (`prose-changed`).

Отличие от концепции: признак «грязная» живёт **только в SQLite**, а не во frontmatter (`lifecycle: needs-reconcile` не используется) — иначе каждая пометка порождала бы коммит в канон, что нарушает инвариант №6 этого документа. Мягкая деградация сохраняется: в окне до ре-консолидации индекс свеж, устареть могут только объявленные рёбра.

**Ручное редактирование** остаётся возможным (git, любой редактор), но по постановке — редкое событие: владелец читает канон при ревью и ознакомлении, а не поддерживает его. Отдельного человеческого пути записи нет: любая правка (человек или агент) проходит одну и ту же машинерию sync → детекторы → очередь. Ревизионная модель — сам git: история, диффы, откаты, атрибуция автора коммита (человек/агент). Механизм pending/review из §7.2 концепции не переносится — при двух акторах (владелец и его агенты) он не окупается; его роль играют просмотр диффов владельцем по желанию и поведенческие ограничения агента (§7.3).

## 9. Отступления от базовой концепции

| Аспект | В концепции | Здесь | Обоснование |
|---|---|---|---|
| Слой доступа (§9) | веб-сервис: authn, фильтрация retrieval, аудит | отсутствует | один владелец, агенты под его учёткой; модель угроз пуста — по требованию задачи |
| Классификация, компартменты, high-water mark, деклассификация | метки + отдельные репо | отсутствуют | нет пользователей с разными допусками |
| Каноны | три: вики, data store, raw sources | один git-репозиторий: бандл = канон знаний, остальные файлы = raw sources | data plane вне задачи; иммутабельность источников даёт git (`путь@коммит`) |
| Data plane (§3.1) | append-only store, датасеты, контракты деривации | нет | наблюдений/time-series нет; триггер возврата — см. §10 |
| Ревизионная модель (§7.2) | propose → pending → review → stable | git-процесс + поведенческие правила агента | издержки гейта не окупаются на двух акторах |
| «Агент дополняет, а не переписывает» (§7.1) | проза человека защищена; переписывание — только как предложение | снято: страницы агентские, свободная реструктуризация в обслуживании (§7.5) | автор и потребитель — агент; правка человека в каноне — редкое исключение, защита — git-история |
| Человекочитаемость канона | равноправная цель (проза-синтез для людей) | второстепенна: телеграфный структурированный стиль (§5.2) | человек читает канон только при ревью и ознакомлении |
| Индексы (§3) | FTS + вектор, RRF — сразу | FTS5/BM25 на старте, вектор отложен | малый корпус; агент итеративно переформулирует запросы; дёшево |
| Провенанс рёбер (§6) | обязателен, валидатор отвергает | рекомендован; lint предупреждает для рёбер-утверждений | цена ложного ребра в доверенной среде низка; атрибуция — git blame |
| `lifecycle: needs-reconcile` (§7.1) | флаг в каноне | очередь только в SQLite | последовательное применение инварианта №9 концепции |
| `log.md` (§4) | генерируется из git в файл | `kb log` по запросу, файл не материализуется | избегаем самопорождающихся коммитов; пермиссивная модель OKF это терпит |
| Оркестратор (§1) | отдельный агент ingest/query/lint | рабочий агент + CLI детерминированных операций | агентная среда уже есть; отдельный сервис избыточен |
| Обход графа (§10) | SQLite + NetworkX | SQLite + рекурсивные CTE | на глубинах 1–2 библиотека не нужна |

Отступления от нормативных правил OKF (нематериализованный `log.md`) и выбор хранилища производного слоя оформить ADR при принятии системы — по политике конформности §4 концепции и правилам репозитория.

## 10. Эволюция: что отложено и когда возвращать

| Отложено | Триггер возврата |
|---|---|
| Векторный индекс + RRF (`sqlite-vec`, локальные эмбеддинги, фиксация версии модели) | измеренные провалы полноты FTS: агент регулярно не находит существующие страницы при парафразных запросах |
| Data plane (append-only SQLite, дескрипторы датасетов `type: Dataset`) | появление рекуррентных структурированных наблюдений (метрики сборок, замеры производительности) — по маршрутизации §3.1 концепции: до этого такие факты живут страницами `insight`, «вики и есть staging» |
| Review-gate (pending → review) | появление второго человека-редактора или недоверенного актора |
| MCP-сервер вместо/поверх CLI | необходимость подключить агентов, не умеющих в shell |
| Стабильные ID концептов (immutable `id` + редиректы) | первый болезненный случай массового переименования; до этого — заглушки с `supersedes` |

## 11. План внедрения

**Фаза 0 — скелет (полдня).** Каталог бандла, `index.md`, `schema/*.yaml` со стартовыми словарями и `schema/templates/` со скелетами разделов (§5.2–5.3). ADR на принятие системы (хранилище проекции, отступления от OKF). Раздел протокола в AGENTS.md (приложение А).

**Фаза 1 — инструмент (1–2 дня).** CLI `kb`: sync, query, neighbors, page, lint уровня 1 с детекторами (`prose-changed`, `source-drift`, `dup-candidate`), queue, log. Unit-тесты на парсинг frontmatter, чанкинг, идемпотентность sync, срабатывание детекторов; e2e: пересборка с нуля на тестовом бандле → запрос → ожидаемые страницы.

**Фаза 2 — первичное наполнение (итеративно).** Агент-архитектор синтезирует стартовый корпус из README, PROJECT_STRUCTURE.yaml, ADR, docs/architecture, верхнего уровня apps/: страницы `domain`/`component`/`integration` + рёбра. Правило: синтез и ссылки, не копирование. Страницы `insight` накапливаются по ходу обычной работы.

**Критерии приёмки:** `kb sync --full` детерминирован (повторный запуск — ноль изменений); дельта-sync < 1 с, полная пересборка < 10 с, query < 200 мс на корпусе фазы 2; контрольный список из ~10 вопросов о проекте — query возвращает релевантную страницу в топ-3 не менее чем для 8; lint без ложных срабатываний на чистом корпусе; детектор `source-drift` срабатывает на тестовом изменении файла-источника.

## 12. Открытые вопросы

- Точное место CLI в дереве репозитория (по правилам структуры корня).
- Политика коммитов страниц знаний: вместе с рабочим коммитом задачи или отдельным коммитом с префиксом `kb:`.
- Финальный токенизатор FTS: `trigram` против `unicode61` — замер качества на реальном двуязычном корпусе фазы 2.
- Порог доли `relates_to` и порог размера страницы для lint (стартовые значения — 30% и 200 строк, уточнить по факту).
- Стартовые пороги триггеров сессии-уборщика (§7.5) — проверить на практике и скорректировать.
- Язык корпуса: русский (проще редкое ревью владельцем) или английский (экономия токенов при чтении агентами) — решение владельца.

> ### База проектных знаний (`docs/knowledge/`)
>
> - **Начало сессии:** выполнить `kb sync`, прочитать `docs/knowledge/index.md`. Тела страниц целиком не сканировать.
> - **Перед проектированием или правкой подсистемы:** `kb query "<вопрос>"`, прочитать только выбранные страницы (`kb page`).
> - **После значимого блока работы:** создать/обновить страницы затронутых концептов и рёбра (`relations:` — только типы из `schema/edge-types.yaml`), при добавлении страниц обновить `index.md`, затем `kb sync` (lint уровня 1 выполняется автоматически; ошибки валидатора исправить до завершения задачи).
> - **Стиль страниц:** телеграфный, списки и таблицы вместо повествования, скелет разделов из `schema/templates/<type>.md`; главный читатель — другой агент.

Слияние, разрезание и переписывание страниц разрешены без согласования.
> - **Запрещено:** писать в `kb.sqlite3` напрямую; расширять `schema/*.yaml` без поручения владельца; дублировать в страницы содержимое ADR/README вместо ссылки.
