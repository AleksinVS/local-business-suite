# ADR-0011: Файловые знания, раздельные базы и единый поиск

## Статус

Accepted

Частично заменен `docs/adr/ADR-0030-memory-alignment-hybrid-knowledge-v05.md`: writer-очередь заменяется прямой записью с pull-reconciler, а метаданные в базе становятся пересобираемой проекцией, а не управляющим зеркалом. Раздельные базы, единый поиск и корпуса `knowledge`/`source_data` остаются в силе.

## Дата

2026-05-22

## Контекст

Текущая память выросла из нескольких направлений:

- явное "запомни" из чата;
- ingestion документов;
- внешние API-коннекторы;
- email и бизнес-аналитика;
- графовый индекс;
- кандидаты в общие знания;
- надежные источники.

Из-за этого в текущей реализации смешались разные понятия:

- **знание** - очищенный и обобщенный смысл, который можно отдавать агенту;
- **данные** - исходные сообщения, документы, письма, API-ответы и аналитические факты;
- **индексы** - технические структуры для поиска;
- **аналитические срезы** - расчеты, метрики, сигналы и диагностические выводы;
- **очереди и временные зоны** - служебный слой обработки.

Сейчас часть этих объектов хранится вместе в основной базе `data/db/main_vault.sqlite3`. Знания хранятся как `MemoryKnowledgeItem`, но также есть файловые проекции, снимки источников, фрагменты, графовые факты и индексные записи. Это создает риск двойного источника истины и усложняет объяснение системы.

Пользователь зафиксировал целевые принципы:

- данные остаются в исходных системах;
- временные сырые и безопасные слои удаляются после извлечения знаний;
- знания независимо от источника хранятся единообразно в файлах и версионируются через Git;
- запись знаний идет только через writer service: очередь, lock, временный файл, атомарная замена, Git commit/merge;
- чтение идет через службу с проверкой прав, чувствительности и надежности источника;
- метаданные и индексы знаний хранятся в отдельных базах;
- поиск и индексация должны быть единым сервисом для знаний и файлового хранилища;
- агент по умолчанию работает со знаниями, а к файлам обращается явно или при пустом результате по знаниям;
- знания делятся на персональные, корпоративные и по источнику данных;
- переход из персональных знаний в корпоративные идет через кандидатство;
- данные чатов и управляющие модели аналитики должны быть вынесены в две отдельные базы;
- графовый индекс строится только по заранее определенным типам сущностей, новые типы проходят кандидатство.

Дополнение: строгое правило "текст знания только в файле знания" вынесено в `docs/adr/ADR-0013-file-only-knowledge-body.md`. ADR-0013 уточняет, что `MemoryKnowledgeItem` не должен хранить текст знания, а индекс не должен быть источником текста для выдачи.

## Решение

### 1. Разделить знания и данные

Знания становятся отдельным доменным слоем.

**Знание** - это очищенный, короткий и проверяемый текст с метаданными, ссылкой на источник и областью действия. Знание хранится в файле в runtime Git-репозитории.

**Данные** - это исходные сообщения, документы, письма, API-объекты и аналитические факты. Система памяти не хранит копии этих данных как постоянный слой. Она хранит только ссылки, хэши, версии, технические признаки и результаты обработки.

### 2. Сделать файлы знаний источником истины

Главный источник истины для принятых знаний:

```text
data/knowledge_repo/
```

Это отдельный runtime Git-репозиторий. Он не является частью основного Git-репозитория проекта.

Файлы знаний хранятся единообразно:

```text
data/knowledge_repo/
  org/
    sources/<source_code>/<year>/<knowledge_id>.md
    _summary.md
  users/<user_id>/
    sources/<source_code>/<year>/<knowledge_id>.md
    _summary.md
  candidates/
    organization/<candidate_id>.md
```

Файл знания содержит:

- стабильный `knowledge_id`;
- область действия: `personal` или `organization`;
- владельца для персонального знания;
- код источника;
- тип источника: чат, файл, email, API, аналитика, ручной ввод;
- ссылки на исходные данные;
- уровень чувствительности;
- `scope_tokens` для проверки прав;
- статус знания;
- текст знания.

Метаданные в базе являются поисковым и управляющим зеркалом, но не заменяют файл знания.

### 3. Ввести writer service для безопасной записи

Любая запись знания идет через writer service.

Порядок:

```text
запрос на запись
  -> очередь записи
  -> проверка прав и политики источника
  -> временный файл рядом с целевым файлом
  -> atomic rename
  -> обновление Git
  -> обновление метаданных
  -> постановка задач индексации
```

Writer service должен:

- сериализовать запись через lock или очередь;
- не давать нескольким процессам писать один файл одновременно;
- использовать атомарную замену файла;
- создавать Git commit на каждое принятое изменение или группу связанных изменений;
- при наличии удаленного Git-репозитория выполнять controlled merge;
- при конфликте создавать задачу ручного разбора, а не перезаписывать данные.

### 4. Ввести reader service для чтения

Прямой доступ агента к файлам знаний запрещен.

Reader service:

- проверяет права пользователя;
- проверяет `scope_tokens`;
- проверяет чувствительность;
- проверяет надежность источника;
- читает файл знания;
- возвращает знание вместе со ссылками на источник данных.

По умолчанию ответ агенту должен содержать:

```text
знание + ссылка на источник данных
```

### 5. Сделать единый сервис поиска и индексации

Нужен один сервис поиска для двух корпусов:

- `knowledge` - принятые знания;
- `source_data` - файлы в хранилище и другие исходные объекты, доступные только по явному запросу или при пустом результате по знаниям.

Обычный порядок поиска:

```text
1. Искать в knowledge.
2. Если пользователь явно попросил искать в файлах, искать в source_data.
3. Если knowledge ничего не нашел, разрешен fallback в source_data по политике источника.
4. Вернуть знания с source links; для source_data вернуть ссылку на исходный объект и признак, что это не принятое знание.
```

Полнотекстовый, векторный и графовый поиск остаются тремя каналами внутри одного сервиса. Агент выбирает режим поиска, а не вручную управляет сложными весами. Сервис применяет заранее заданные профили:

- точный поиск - больше вес полнотекстового поиска;
- смысловой поиск - больше вес векторного поиска;
- вопрос о связях - подключается графовый индекс;
- явный поиск по файлам - включается корпус `source_data`;
- обычный вопрос - сначала только корпус `knowledge`.

### 5a. Заменить MemorySnapshot/MemoryChunk прямой индексной записью

Уточнение от 2026-05-22: совместимость со старым индексным слоем не является обязательной для MVP, потому что production-внедрения еще не было.

`MemorySnapshot` и `MemoryChunk` не должны оставаться обязательным путем для файловых знаний. Их роль заменяется прямой индексной записью `MemorySearchDocument`.

Целевая схема для знания:

```text
файл знания
  -> MemoryKnowledgeItem
  -> MemorySearchDocument
  -> полнотекстовый / векторный / графовый индекс
  -> memory.search
```

Целевая схема для исходных данных:

```text
источник данных
  -> MemorySourceObject
  -> временная обработка
  -> MemorySearchDocument
  -> source_data search result
```

`MemorySearchDocument` является технической записью поиска. Она хранит `document_id`, `corpus_type`, ссылки на знание или исходный объект, `source_refs`, права, чувствительность, статус индексации и безопасные метаданные. Она не хранит полный исходный текст документов, писем, API-ответов или сообщений.

Причина: после перехода к файловым знаниям `MemorySnapshot`/`MemoryChunk` снова смешивают знания, данные и индексы. Для принятого знания источник истины - файл знания, а не chunk.

### 6. Ограничить графовый индекс

Графовый индекс строится только по типам сущностей и связей, которые уже утверждены в контракте графа.

Если обработка часто встречает новый тип сущности или связи, система создает кандидата на расширение схемы. Новый тип не попадает в рабочий граф до аудита владельцем графа.

### 7. Разнести базы

Целевая схема баз:

```text
data/db/main_vault.sqlite3              # ядро портала: пользователи, роли, настройки, справочники
data/db/chat.sqlite3                    # диалоги, сообщения, вложения и журналы действий агента
data/db/knowledge_meta.sqlite3          # метаданные знаний, кандидаты, аудит, задачи записи
data/db/analytics_control.sqlite3       # управляющие модели аналитики
data/analytics/duckdb/analytics.duckdb  # аналитические срезы, витрины и расчеты
data/indexes/fulltext/search.sqlite3    # полнотекстовые индексы знаний и источников
data/indexes/vector/                    # векторные индексы
data/indexes/graph/                     # графовый индекс
data/queues/*.sqlite3                   # очереди обработки
data/cache/                             # кэши, не являющиеся источником истины
```

На первом этапе допустима совместимая миграция: старые модели остаются в `apps.memory`, но их роли переименовываются в документации и коде постепенно.

### 8. Вынести данные чатов и управляющие модели аналитики

Чаты становятся отдельным источником данных. История чатов хранится в `data/db/chat.sqlite3`, а знания из чатов - в файлах знаний.

Аналитика делится на:

- управляющие модели аналитики в `data/db/analytics_control.sqlite3`;
- аналитические факты, срезы и витрины в `data/analytics/duckdb/analytics.duckdb` или файловых наборах под `data/analytics/`;
- знания из аналитики в `data/knowledge_repo/` на тех же правилах, что и остальные знания.

### 9. Вынести ночную рефлексию в отдельный процесс

Текущий обработчик явных запросов "запомни" не должен называться полноценной рефлексией.

Нужно разделить:

- обработчик очереди записи знаний;
- обработчик индексации;
- отдельный процесс ночной рефлексии.

Ночная рефлексия работает в дешевое время и:

- объединяет повторы;
- находит устаревшие знания;
- создает кандидатов в корпоративные знания;
- предлагает новые типы сущностей графа по статистике;
- обновляет summary-файлы.

## Альтернативы

### Оставить знания в основной базе

Отклонено. Это проще для кода, но хуже для аудита, ручной проверки, резервного копирования, Git-версий и независимого чтения знаний.

### Хранить исходные данные внутри памяти

Отклонено. Это увеличивает риски безопасности, дублирует источники истины и усложняет правила удаления.

### Сделать отдельные сервисы поиска для знаний и файлов

Отклонено для целевой архитектуры. Два сервиса быстро начнут расходиться по правам, логике ранжирования и аудиту. Нужен один сервис с разными корпусами и режимами.

### Сразу переносить все модели в новые приложения

Отклонено как первый шаг. Безопаснее сначала ввести новые границы хранения, затем постепенно переименовывать модели и выносить базы через маршрутизацию Django.

## Последствия

### Плюсы

- Понятное разделение: данные отдельно, знания отдельно, индексы отдельно.
- У агента появляется ясный основной источник: принятые знания.
- Git дает историю изменений знаний и удобный аудит.
- Источники данных не копируются в память как постоянный слой.
- Можно искать и по знаниям, и по файлам через один сервис.
- Чаты и аналитика перестают раздувать основную базу портала.
- Графовый индекс ограничен утвержденной схемой и не растет хаотично.

### Минусы

- Появляется writer service и очередь записи.
- Нужна миграция существующих `MemoryKnowledgeItem` в файлы.
- Нужна маршрутизация нескольких SQLite-баз.
- Индексация должна уметь работать с двумя корпусами и несколькими индексами.
- Нужно аккуратно описать удаление временных слоев, чтобы не потерять проверяемость знаний.

## Не входит в это решение

- Выбор окончательного production backend для векторного индекса.
- Полная замена всех старых имен моделей за один этап, кроме отдельного решения убрать `MemorySnapshot`/`MemoryChunk` из активного пути.
- Полный внешний API системы памяти.
- Перенос секретов: остается ранее принятое решение про secret handles.
- Изменение бизнес-прав доступа портала, не связанных с памятью.

## Требуемые действия

- Создать проектный план миграции к файловым знаниям и раздельным базам.
- Создать workflow-блок с task packets.
- Спроектировать формат файлов знаний и правила Git-записи.
- Ввести metadata database для знаний.
- Ввести единый search/index service с корпусами `knowledge` и `source_data`.
- Вынести данные чатов в отдельную базу.
- Вынести управляющие модели аналитики в отдельную базу.
- Заменить `MemorySnapshot`/`MemoryChunk` в активном пути на `MemorySearchDocument`.
- Разделить обработчик записи знаний, индексацию и ночную рефлексию.
- Обновить документацию, тесты и сквозные проверки.
