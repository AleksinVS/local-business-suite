## **Техническое задание для AI-агента: Разработка системы "Local Business Suite"**

**Цель:** Создание модульного монорепозитория на Django для коллекции внутренних веб\-приложений с единой точкой входа, локальной авторизацией в первой поставке MVP, последующей LDAP-интеграцией и файловым хранением данных.

---

### **1\. Архитектурный стек**

* **Framework:** Django 5.x (Monorepo).  
* **Database:** SQLite (основная) \+ DuckDB (аналитический слой).  
* **Auth (Этап 1):** Локальная авторизация Django по логину/паролю.  
* **Auth (Этап 2):** `django-auth-ldap` (интеграция с Active Directory без слома локальной модели ролей).  
* **Frontend:** Django Templates \+ HTMX \+ Tailwind CSS (через библиотеку).  
* **Deployment:** Docker Compose (Сервис Django \+ WhiteNoise для статики).

---

### **2\. Конфигурация СУБД (SQLite)**

Для обеспечения работы 50 пользователей в `settings.py` необходимо настроить режим **WAL (Write-Ahead Logging)**:

Python  
DATABASES \= {  
    'default': {  
        'ENGINE': 'django.db.backends.sqlite3',  
        'NAME': BASE\_DIR / 'db' / 'main\_vault.sqlite3',  
        'OPTIONS': {  
            'timeout': 20,  
            'init\_command': (  
                'PRAGMA journal\_mode=WAL;'  
                'PRAGMA synchronous=NORMAL;'  
                'PRAGMA busy\_timeout=5000;'  
            ),  
        },  
    }  
}

---

### **3\. Модульная структура приложений (Apps)**

Каждое новое приложение создается как стандартный `django-app`. **Приоритет проектирования:**

1. **Core / Shared:** Общие стили, базовый шаблон, общие сервисы, политики доступа и подготовка к LDAP.  
2. **Inventory:** Справочник медицинских изделий.  
3. **WorkOrders:** Заявки на ТО и ремонты, комментарии, вложения, история переходов.  
4. **Analytics:** Модуль отчетов (на первом этапе на ORM/SQL, DuckDB опционально на следующем шаге).

---

### **4\. Авторизация и LDAP**

**Этап 1 (первая поставка MVP):**
- Использовать стандартную авторизацию Django по логину/паролю.
- Роли и права хранить в приложении, без зависимости от LDAP.

**Этап 2 (расширение MVP):**
- Подключить `django-auth-ldap` через `LDAPBackend`.
- Пользователи создаются в Django автоматически при первом входе.
- Атрибуты (ФИО, Email) маппить из AD.
- LDAP не должен быть единственным источником бизнес-ролей: прикладные права должны оставаться совместимыми с локальной моделью ролей и политиками доступа.

---

### **5\. Инструкции по реализации аналитического модуля (DuckDB)**

На первом этапе MVP отчеты реализуются на ORM и SQL-агрегациях поверх основной БД. DuckDB использовать только если подтверждена необходимость в более тяжелой аналитике:

Python  
import duckdb

def report\_view(request):  
    db\_path \= settings.DATABASES\['default'\]\['NAME'\]  
    \# Аналитика: среднее время выполнения задач по категориям  
    query \= f"""  
        SELECT category, AVG(completion\_time)   
        FROM sqlite\_scan('{db\_path}', 'maintenance\_task')  
        GROUP BY category  
    """  
    stats \= duckdb.query(query).to\_df()  
    \# Далее передача в контекст для отрисовки графиков

### **6\. Требования к UI (HTMX)**

* Никаких тяжелых JS-фреймворков.  
* Обновление таблиц и форм через `hx-post` / `hx-target`.  
* Модальные окна и уведомления через `django-messages` \+ HTMX.
* UI адаптивный и отвечающий современным требованиям к дизайну веб-приложений.
---

### **7\. Правила именования и папок**

* `/db/` — папка для файлов `.sqlite3` (вынести в Docker Volume).  
* `/apps/` — папка для всех внутренних приложений.  
* `/static/` и `/templates/` — глобальные ресурсы.

### **8\. MVP приложение**

* Первое приложение - одна канбан-доска внутренних заявок на технические обслуживание и ремонты.
* Полноценный справочник медицинских изделий обязателен уже в первой поставке.
* Для справочника медицинских изделий предусмотреть:
 - поиск по ключевым полям,
 - фильтрацию по подразделению, местоположению, статусу эксплуатации и другим значимым атрибутам,
 - привязку заявок к медицинским изделиям.
* 3 типа пользователей с разделением прав:
 - внутренний заказчик (просмотр всех карточек заявок, создание карточек, изменение допустимых статусов заявки, оценка результатов через `rating`, подтверждение закрытия, комментарии),
 - техник (просмотр карточек заявок, изменение допустимых статусов заявки, комментарии, работа с вложениями),
 - начальник технического отдела (просмотр и создание карточек заявок, изменение статусов заявки, оценка результатов, подтверждение закрытия, комментарии, просмотр сводных отчетов по заявкам, управление справочником медицинских изделий).
* Предусмотреть фильтрацию и группировку карточек заявок по подразделению, автору, медицинскому изделию и статусу.
* Предусмотреть полнотекстовый поиск по заявкам.
* Работа с вложениями и фотографиями обязательна уже в MVP.
