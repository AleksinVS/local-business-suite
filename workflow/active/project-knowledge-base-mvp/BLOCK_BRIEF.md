# Block Brief: база проектных знаний для ИИ-агентов (kb) — MVP

## Зачем

Проект kb спроектирован и дважды прошёл критическое ревью (версия проекта 0.8): markdown-канон `docs/knowledge/` + полностью пересобираемая SQLite-проекция (FTS5, граф рёбер, очередь обслуживания), CLI `scripts/dev/kb.py`, двухуровневый контур качества (детерминированные детекторы + сессия-уборщик) и сверка на записи (`kb impact`). Точечное чтение экономит контекст агентов, но снижает осведомлённость о смежных решениях; kb компенсирует это картой знаний, поиском и обязательной сверкой при записи. Исполнительная спецификация T0–T6 (`docs/planning/active/project-knowledge-system-mvp.md`) задаёт DDL, алгоритмы, форматы и коды выхода дословно — ради детерминизма и воспроизводимости, независимо от силы модели-исполнителя. Блок упаковывает T0–T6 в исполняемые пакеты с областями записи, проверками и приёмкой.

## Что сделать

Шесть пакетов (`task-packets/`), строго по порядку: (01) скелет канона по §3 плана + каркас CLI и базовые функции (T0–T1); (02) ядро sync, базовый lint и тестовый бандл-фикстура (T2); (03) акторная модель, детекторы ядра включая `contradiction`, очередь (T3); (04) читающие команды `query`/`neighbors`/`page`/`log` и `impact` (T4); (05) детекторы расширения и `kb eval` (T5); (06) e2e-тест, ADR, раздел в AGENTS.md, приёмка по чек-листу §11 (T6).

## Границы

- Все проектные решения зафиксированы в плане; исполнителю запрещено их менять. Расхождение плана и проектного документа → прав план (он новее), расхождение фиксируется в отчёте; пробел → остановиться и спросить владельца.
- Вне объёма: фаза 2 (наполнение корпуса, лимит 20–30 страниц, golden set 10–20 пар) и фаза 3 (`tree_nodes`, обогащение карты проекта) — отдельные задачи после приёмки MVP.
- `apps/`, `contracts/` и зависимости проекта не трогать; продуктовая система памяти (`apps.memory`) не затрагивается ни одним шагом.
- Только stdlib + PyYAML; запись в `kb.sqlite3` мимо `kb.py` и git-хуки запрещены.
