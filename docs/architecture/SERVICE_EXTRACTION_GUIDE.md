# Правила выноса сервисов

## Назначение

Документ задает правила для будущего частичного выноса технических контуров из Django в отдельный процесс или сервис. Цель - сохранить простоту local-first портала и при этом не заблокировать будущие Go/Rust/Python workers.

Архитектурное решение: `docs/adr/ADR-0024-service-extraction-readiness.md`.

## Базовое правило

Django остается системой истины для:

- пользователей, групп, ролей и SSO;
- доменных моделей заявок, справочников, настроек и workflow;
- политик доступа;
- audit и пользовательских бизнес-действий;
- runtime-контрактов в `data/contracts/`.

Отдельный сервис может выполнять техническую работу, но не должен становиться владельцем бизнес-правил.

## Когда вынос допустим

Вынос компонента можно планировать, если выполнены условия:

- есть измеренная проблема: p95, ошибки, время очереди, объем файлов, блокировки или CPU-нагрузка;
- проблема не решается проще оптимизацией Django, SQL, индексов, кэша, PostgreSQL, DuckDB, Celery или настройкой текущего backend;
- контур имеет узкую ответственность;
- есть стабильный входной и выходной контракт;
- понятны права, audit, формат ошибок и повторная обработка;
- есть e2e-проверка главного сценария.

Хорошие кандидаты:

- parser/OCR worker;
- privacy-worker;
- external connector worker;
- search/vector service;
- event/outbox consumer;
- тяжелые аналитические преобразования.

Плохие кандидаты:

- заявки и workflow как отдельный сервис без зрелой причины;
- роли и policies как отдельный сервис;
- HTML-интерфейс как отдельное SPA только ради смены технологии;
- прямой доступ к `data/db/*.sqlite3` из внешнего сервиса.

## Допустимые границы

### Очередь задач

Подходит для долгих, повторяемых и восстанавливаемых действий:

```text
Django service
  -> queue job
  -> worker
  -> result/status
  -> Django review/audit/read model
```

Обязательные поля задачи:

- `job_id`;
- `job_kind`;
- `schema_version`;
- `idempotency_key`;
- `status`;
- `attempt_count`;
- `next_attempt_at`;
- `error_code`;
- `error_message`;
- `source_ref`;
- `audit_ref`.

### Gateway/API

Подходит для коротких запросов, где нужен ответ сразу. Gateway должен быть внутренним, аутентифицированным и не должен раскрывать business tables как произвольный API.

Обязательные свойства:

- service identity;
- request id;
- timeout;
- безопасный формат ошибок;
- audit на стороне Django;
- запрет raw PII/secrets в логах.

### SourceObjectEnvelope

Подходит для памяти, аналитики и поиска:

```text
domain module / external source
  -> SourceAdapter
  -> SourceObjectEnvelope
  -> memory / analytics projection
```

Envelope должен сохранять `schema_version`, `source_code`, `object_type`, `object_id`, `content_hash`, `sensitivity`, `privacy_profile`, `access_policy` и `provenance`.

### File/export contract

Подходит для пакетной аналитики или обмена с внешними системами. Файл должен быть производным артефактом, а не скрытой копией бизнес-БД.

Допустимые форматы:

- JSONL для простого журнала событий;
- Parquet для аналитических наборов;
- CSV/XLSX/XML только для совместимости с внешней системой.

## Запреты

Запрещено для новых сервисов:

- читать или менять `data/db/main_vault.sqlite3` напрямую;
- повторно реализовывать ролевую модель вне Django;
- принимать `user_id` из тела запроса как единственное основание для прав;
- писать секреты, prompt, исходные документы или персональные данные в технические логи;
- хранить runtime-состояние в корне проекта;
- добавлять новый runtime-сервис без ADR и deployment-документации.

## Минимальный checklist перед выносом

- Создан или обновлен ADR.
- Описан контракт входа и выхода.
- Определен владелец source of truth.
- Описаны права и audit.
- Описаны retry, idempotency и ручной разбор ошибок.
- Добавлены p50/p95 или другие базовые метрики до изменения.
- Добавлены unit/integration/e2e проверки.
- Обновлены deployment и operation guides.

## Приоритет оптимизаций до выноса

Перед Go/Rust-сервисом нужно проверить более дешевые шаги:

1. SQL indexes и устранение N+1.
2. `select_related` / `prefetch_related`.
3. Ограничение выборок, пагинация, отложенная загрузка.
4. Кэширование контрактов и справочников.
5. Перенос долгих действий в worker.
6. PostgreSQL для конкурентной записи, если SQLite стал узким местом.
7. DuckDB/Parquet для тяжелой аналитики.
8. Qdrant или другой отдельный search/vector backend, если embedded-индексы перестали хватать.
