# Доводка памяти после ревью целевой архитектуры (memory-alignment-hardening)

## Статус

Active planning. Готов к исполнению. Создан 2026-07-06, дополнен 2026-07-07.

- Основание: дополнения 2026-07-06 и 2026-07-07 в `docs/adr/ADR-0030-memory-alignment-hybrid-knowledge-v05.md` (раздел «Требуемые действия») и остаточные риски из `workflow/archive/2026/memory-hybrid-knowledge-v05-alignment/RETROSPECTIVE.md`; кросс-ревью находок параллельного kb-проекта (2026-07-06) добавило канон-ориентированный обход, базисы из git и обновление входящих рёбер при supersede; обсуждение паттерна «сверка на записи» в том же kb-проекте (2026-07-07, концепт §7.4) добавило протокол противоречия и команду `memory_impact`.
- Workflow-блок: `workflow/active/memory-alignment-hardening/` (3 task packets).
- Блок выравнивания ADR-0030 уже исполнен и заархивирован; этот блок закрывает найденные после исполнения пробелы и не меняет принятую архитектуру.

## Цель

Закрыть девять пробелов, найденных после исполнения блока выравнивания (ревью целевой архитектуры, кросс-ревью и обсуждение паттернов параллельного kb-проекта):

1. LF-нормализация канона (`.gitattributes`, запись с LF, хэши по LF-содержимому) — без нее git-autocrlf на Windows дает ложные hash mismatch.
2. Гарантия отсутствия петли `needs-reconcile` (пометка страницы reconciler'ом не должна выглядеть новым изменением).
3. Синглтон `memory_reconcile` (advisory lock на PostgreSQL, lock-файл на dev SQLite).
4. Инструкция эмиссии `relations:` для семантического прохода — сейчас словарь и материализатор есть, а производителя рёбер нет.
5. Golden set `MemoryEvalCase` (не менее 20 кейсов) + периодический `memory_eval` — иначе критерии возврата профилей и графа никогда не сработают.
6. Матрица восстановления в `docs/deployment/MEMORY_DEPLOYMENT.md` (что резервируется, что пересобирается).
7. Остатки ретроспективы: чистка `ranking_profiles` в `contracts/ai/memory_profiles.json`, фильтр очереди в review-UI.
8. Канон-ориентированный обход reconciler и базисы из git (кросс-ревью 2026-07-06): файлы, добавленные/перенесенные/удаленные в обход `memory.remember`, видимы детерминированному проходу (строки проекции создаются, перенос — по `knowledge_id`, пропажа — issue); `--force` пересчитывает гард классификации и очередь из git-истории; дедуп-ключ очереди включает причину; при `supersede` обновляются входящие рёбра.
9. Сверка на записи (концепт §7.4, дополнение ADR-0030 от 2026-07-07): инструкция семантического прохода получает протокол противоречия — конфликт с существующей страницей не правится молча, эмитится ребро `contradicts` с провенансом; команда `memory_impact` выдает детерминированный проверочный набор для прямых правок канона (входящие рёбра, исходящие `relations:`, топ гибридного поиска); golden set пакета 02 читается и как мера recall проверочного набора (кейсы самонахождения). Горячий путь `memory.remember` сверкой не нагружается.

## Не цели

- Не менять архитектуру ADR-0030 и формат frontmatter.
- Не реализовывать data store (5а/5б), graph runtime search, профили ранжирования, фрагментный поиск.
- Не выполнять массовую перезапись существующих файлов знаний (LF-нормализация применяется при следующей записи каждой страницы).
- Не менять права доступа, trust-гейты, secret handles.

## Пакеты

| Packet | Содержание |
|---|---|
| `01-canon-line-endings-and-reconciler-guards` | `.gitattributes`, LF-нормализация записи и хэшей, тест отсутствия петли `needs-reconcile` (+фикс через git-автора `memory-reconciler`, если тест падает), синглтон `memory_reconcile`, тест обеих платформенных веток блокировки; канон-ориентированный обход (новые/перенесенные/пропавшие файлы), базисы из git при `--force`, `kind` в дедуп-ключе очереди |
| `02-eval-golden-set-and-recovery-matrix` | Команда `memory_eval_seed`, черновой golden set (>=20 кейсов) на реальном корпусе (кейсы самонахождения — заодно мера recall сверки на записи), прогон `memory_eval` с отчетом, матрица восстановления в `MEMORY_DEPLOYMENT.md` |
| `03-edge-emission-and-retro-leftovers` | Инструкция эмиссии `relations:` в конфигурации семантического прохода (`knowledge_reflection_worker`) с протоколом противоречия (`contradicts` вместо молчаливой правки), обновление входящих рёбер при `supersede` (хелпер обратного индекса по `MemoryKnowledgeEdge`), команда `memory_impact` (детерминированный проверочный набор — сверка на записи), чистка `ranking_profiles` до одного `balanced`, фикс фильтра очереди review-UI |

Порядок: 01 -> 02 -> 03. Пакеты 02 и 03 независимы и могут идти параллельно после 01.

## Definition of Done блока

- Все три пакета приняты; полный набор проверок зеленый на SQLite и PostgreSQL.
- Файл знания, записанный из тела с CRLF, содержит только LF и дает тот же content-hash, что LF-вариант.
- Файл, добавленный в канон напрямую (мимо `memory.remember`), после reconcile находится поиском; перенос файла при том же `knowledge_id` не создает дубль; пропавший файл дает issue `canon_file_missing`.
- Повторный `memory_reconcile` после пометки страницы сообщает ноль изменений; параллельный второй запуск завершается без работы.
- В `MemoryEvalCase` не менее 20 активных кейсов; отчет `memory_eval` приложен к приемке.
- Инструкция семантического прохода содержит протокол противоречия (`contradicts` с провенансом, молчаливая правка чужой страницы запрещена); `memory_impact` для заданной страницы выводит входящие рёбра, исходящие `relations:` и топ гибридного поиска, не меняя состояния.
- `MEMORY_DEPLOYMENT.md` содержит матрицу восстановления; `MEMORY_MVP_CURRENT_STATE.md` актуализирован.
- Backlog: блок удален после приемки; записи `Later` (профили, graph runtime search) ссылаются на golden set как выполненное предусловие.
