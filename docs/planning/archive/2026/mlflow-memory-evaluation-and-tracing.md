# Active plan: MLflow для оценки качества и трассировки памяти

Статус: черновик.

Дата: 2026-05-26.

Важно: этот план требует детальной проработки перед реализацией. Он фиксирует направление и предварительный порядок работ, но не является готовым исполнительным заданием.

Связанный ADR: `docs/adr/ADR-0014-mlflow-memory-evaluation-and-tracing.md`.

## Цель

Добавить внешний контур оценки качества и трассировки системы памяти на базе MLflow, не заменяя обязательные собственные проверки `memory_eval`.

Нужно получить:

- историю проверочных прогонов памяти;
- сравнение версий поиска, индексов, промптов и моделей;
- трассировку цепочки `memory.search` и ответа агента;
- метрики качества поиска и обоснованности ответа;
- безопасные отчеты без секретов и необезличенных персональных данных.

## Базовый принцип

`memory_eval` остается обязательной быстрой проверкой правил проекта.

MLflow добавляется отдельным слоем:

```text
memory_eval
  -> безопасность, права, trusted sources, архитектурные правила

memory_quality_eval
  -> качество поиска, трассировка, сравнение прогонов через MLflow
```

MLflow не должен участвовать в горячем пути ответа пользователю.

## Предварительный объем работ

Входит:

- добавить настройки MLflow tracking URI и режима безопасной записи трасс;
- добавить команду `memory_quality_eval`;
- связать `MemoryEvalCase` с прогоном качества;
- записывать в MLflow параметры поиска, версии индексов, найденные `document_id` и безопасные метрики;
- добавить трассировку шагов поиска и сборки контекста;
- сохранить краткий JSON-итог в `data/memory/eval/`;
- добавить проверки, что секреты и сырые чувствительные данные не уходят в MLflow;
- подготовить безопасный набор тестовых сценариев качества.

Не входит в первый черновой этап:

- постоянная трассировка всех боевых запросов;
- запись сырых писем, документов, API-ответов или полных чувствительных контекстов;
- подключение Ragas/DeepEval как обязательной зависимости;
- промышленная настройка PostgreSQL и отдельного хранилища артефактов;
- UI в портале для просмотра отчетов.

## Предварительная архитектура

```text
MemoryEvalCase
  -> memory_quality_eval
  -> memory.search
  -> retrieval trace
  -> optional agent answer
  -> MLflow run and trace
  -> scorers
  -> data/memory/eval/<report>.json
```

Минимальные сущности связи:

- `request_id`;
- `mlflow_run_id`;
- `mlflow_trace_id`;
- `eval_case_code`;
- `suite`;
- `engine`;
- `created_at`.

Вопрос о том, хранить ли `mlflow_run_id` и `mlflow_trace_id` прямо в `MemoryAccessAudit` или в отдельной таблице результатов оценки, требует отдельной проработки.

## Черновой порядок реализации

1. Проверить совместимость версии MLflow с текущим Python и зависимостями проекта.
2. Добавить dependency-gate: проект должен работать без MLflow, если контур качества выключен.
3. Добавить настройки:
   - `MLFLOW_TRACKING_URI`;
   - режим записи безопасного текста;
   - ограничение размера трассы;
   - срок хранения локальных артефактов.
4. Добавить thin adapter для MLflow:
   - включение/выключение;
   - создание run;
   - запись параметров;
   - запись метрик;
   - безопасная запись артефактов.
5. Добавить команду `memory_quality_eval`.
6. Подключить собственные числовые метрики:
   - `expected_document_recall_at_k`;
   - `forbidden_document_hits`;
   - `trusted_source_violations`;
   - `scope_violations`;
   - `latency_ms`;
   - `context_item_count`;
   - `context_token_estimate`.
7. Добавить минимальную трассировку `memory.search` внутри quality-run без изменения обычного пользовательского пути.
8. Добавить безопасные `MemoryEvalCase` для качества поиска.
9. Добавить unit-тесты адаптера и команды.
10. Добавить e2e-проверку: тестовый набор знаний -> `memory_quality_eval` -> JSON-отчет и MLflow run.
11. Обновить эксплуатационную документацию.

## Вопросы для детальной проработки

- Где хранить связь `MemoryAccessAudit` и MLflow: в существующей модели, отдельной модели или только в JSON-отчете?
- Какая часть текста знания может попадать в MLflow при локальном запуске?
- Нужен ли отдельный профиль для проверки боевых данных без записи текста?
- Какая минимальная версия MLflow нужна проекту?
- Должен ли MLflow быть обязательной зависимостью или опциональным extra-пакетом?
- Нужен ли отдельный контейнер/сервис для MLflow в docker-compose MVP?
- Какие RAG-оценщики включать первыми: только собственные метрики или сразу MLflow RAG scorers?
- Какой срок хранения трасс и артефактов нужен для локального MVP?

## Предварительные критерии приемки

- `memory_eval` продолжает работать без MLflow.
- Если MLflow выключен, основное приложение и память работают без ошибок.
- `memory_quality_eval --engine mlflow --suite smoke` создает проверочный прогон.
- В отчет попадают `document_id`, метрики и технические параметры, но не секреты.
- Сырые письма, документы, API-ответы и необезличенные персональные данные не пишутся в MLflow.
- В `data/memory/eval/` создается краткий JSON-отчет.
- Есть unit-тесты на безопасную запись и запрет чувствительных данных.
- Есть e2e-проверка основного сценария quality-run.
