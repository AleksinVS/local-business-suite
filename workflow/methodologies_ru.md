## 1. Обзор методик

Этот каталог описывает паттерны воркфлоу для multi-agent delivery, не привязанные к конкретному репозиторию.

### Доступные методики

- `Vertical slices`
- `Spec -> scaffold -> harden`
- `Review-driven loop`
- `Contract-first / schema-first`
- `TDD`

### Когда что использовать

`Vertical slices` эффективны, когда команде важно рано показывать бизнес-видимый прогресс, а фичу можно разрезать на относительно независимые пользовательские инкременты.

`Spec -> scaffold -> harden` эффективна, когда нужен быстрый рабочий базис, а строгость (тесты, валидация, крайние случаи) можно усилить на следующем проходе без потери архитектурного контроля.

`Review-driven loop` эффективна, когда требования нестабильны, качество дизайна критично, или короткий цикл review-feedback безопаснее, чем один большой проход реализации.

`Contract-first / schema-first` эффективна, когда нескольким агентам или сервисам нужно интегрироваться по одному стабильному контракту, и цена рассинхронизации высока.

`TDD` эффективна, когда поведение хорошо определено, регрессии дороги, и самый быстрый путь к уверенности это зафиксировать границы поведения тестами.

### Сквозные use cases

- Предпочитать `Vertical slices` для продуктовых фич с ясными границами UI, workflow и хранения данных.
- Предпочитать `Spec -> scaffold -> harden` для новых модулей, миграций от ad hoc к структурированному коду и платформенных добавлений, где нужен быстрый, но безопасный плацдарм.
- Предпочитать `Review-driven loop` для UX-heavy задач, неоднозначных требований или legacy-кода со скрытыми ограничениями.
- Предпочитать `Contract-first / schema-first` для API, AI tool contracts, event flows и границ между командами.
- Предпочитать `TDD` для багфиксов, детерминированной доменной логики и рискованных рефакторингов, где важнее исполнимые доказательства, чем narrative-план.

### Типовые misuse-паттерны

- `Vertical slices` плохи как дефолт, когда каждый срез вынужден многократно трогать один и тот же хрупкий shared layer.
- `Spec -> scaffold -> harden` плоха, когда сам scaffold может привести к необратимым последствиям (например, drift публичного API или опасный rollout схемы).
- `Review-driven loop` плоха, когда поверхность review слишком дорогая для повторов, а требования уже стабильны.
- `Contract-first / schema-first` избыточна, когда задача локальная, краткоживущая и без интеграционных границ.
- `TDD` слаба как дефолт, когда основная неопределенность это исследовательский дизайн, а не поведение.

## 2. Vertical Slices

### Цель

Доставлять бизнес-видимые инкременты, которые проходят через несколько слоев системы, при этом каждый срез остается ограниченным и отдельно ревьюируемым.

### Роли агентов

- `Architect (L1)` задает границы срезов, инварианты и поверхность приемки.
- `Orchestrator` маршрутизирует срезы по зависимостям и держит состояние явно.
- `Executor (L3)` реализует один ограниченный срез end-to-end.
- `PM (L2)` выполняет block-level acceptance после завершения нужных срезов.

### Общий план

1. Определить один user-visible блок.
2. Разрезать блок на тонкие вертикальные срезы с явными границами владения.
3. Завершать каждый срез кодом, тестами и task-level self-acceptance от исполнителя.
4. Делать block-level acceptance только после завершения обязательных срезов блока.

### Общие принципы

- Один срез должен иметь одну понятную поверхность приемки.
- Работа в shared layer должна быть минимальной и выноситься вперед только при реальной необходимости.
- Границы срезов должны снижать неоднозначность оркестрации.
- Исполнитель не должен “додумывать” архитектуру внутри среза.

### Воркфлоу

1. Архитектор выбирает vertical slicing и объясняет, почему методика подходит блоку.
2. Архитектор задает порядок срезов, allowed files, forbidden moves и тесты.
3. Оркестратор материализует task packets без перефразирования.
4. Исполнители отдают срезы по одному, записывая артефакты реализации и task acceptance.
5. Оркестратор держит состояние, безопасно ретраит и делает artifact probe.
6. PM выполняет block acceptance после завершения блока.

### Use cases

- Бизнес-фича с понятными touchpoints: UI, сервисы, данные.
- Доставка, где стейкхолдерам важен видимый прогресс после каждой единицы реализации.
- Монолит или модульная система, где ценнее end-to-end завершение фичи, чем “по слоям”.

### Misuse cases

- Весь блок доминируется shared инфраструктурой.
- Один и тот же нестабильный контракт приходится переписывать в каждом срезе.
- Срез нельзя валидировать без почти полной готовности остальных срезов.

## 3. Критерии выбора Vertical Slices

### Выбирать Vertical Slices, когда

- блок можно разбить на пользовательские инкременты;
- у каждого инкремента есть локальная поверхность приемки;
- зависимости между инкрементами реальны, но управляемы;
- ранняя частичная доставка дает продуктовую ценность;
- прогресс должен быть понятен не только архитекторам.

### Избегать Vertical Slices, когда

- работу “съедает” shared platform слой;
- архитектура слишком нестабильна, чтобы фиксировать границы срезов;
- каждый срез будет многократно менять один и тот же набор хрупких файлов;
- главный риск в контрактах/инфре, а не в сборке фичи;
- приемка невозможна до почти полного завершения блока.

### Сильные сигналы

- Есть ясное утверждение “этот срез работает”.
- Владение файлами можно ограничить на уровне среза.
- Тесты можно сопоставить срезу без изобретения новой архитектуры.

### Тревожные сигналы

- “Потом подчистим shared код” встречается в каждом срезе.
- Несколько срезов нуждаются в одном фундаментальном примитиве и планируют временно дублировать его.
- Оркестратору приходится переинтерпретировать требования, чтобы маршрутизировать работу.

## 4. Spec -> Scaffold -> Harden

### Цель

Прийти к безопасной и ревьюируемой реализации в три осознанных прохода: задать форму, собрать минимально рабочую структуру, затем ужесточить корректность и сопровождаемость.

### Роли агентов

- `Architect (L1)` определяет целевую форму и guardrails.
- `Executor A` делает scaffold.
- `Executor B` делает harden: поведение, тесты, крайние случаи.
- `PM (L2)` принимает блок только после harden.

### Общий план

1. Зафиксировать требуемую форму и границы.
2. Построить минимальный жизнеспособный каркас.
3. Усилить тесты, валидацию, контракты и документацию.

### Общие принципы

- Scaffold намеренно неполон, но структурно корректен.
- Harden это не “косметическая чистка”, а доказательство корректности.
- Scaffold не должен вносить публичный drift, который потом нельзя безопасно исправить.

### Воркфлоу

1. Архитектор выбирает метод, когда форма известна, но сразу “в прод-качество” слишком замедлит старт.
2. Первый проход создает файлы, роуты, модели, placeholders и тонкое happy-path поведение.
3. Второй проход закрывает пробелы, добавляет тесты, усиливает валидацию, убирает временные упрощения.
4. PM принимает только hardened результат, не scaffold.

### Use cases

- Новый модуль внутри существующей кодовой базы.
- Миграция от proof-of-concept к production-ready структуре.
- Фича, где архитектура известна, но полная корректность требует второго прохода.

### Misuse cases

- Внешние контракты уже публичные и не терпят “временного каркаса”.
- Scaffold создает опасные данные или необратимые сайд-эффекты.
- Нет дисциплины гарантированно довести harden-фазу.

## 5. Review-Driven Loop

### Цель

Короткими циклами implementation-review-correction сходиться к качеству, когда требования неполны, дизайн-решения чувствительны, или вероятны скрытые ограничения.

### Роли агентов

- `Architect (L1)` задает границы и поверхность review.
- `Executor (L3)` реализует ограниченную задачу.
- `Reviewer` или `PM (L2)` возвращает decision packet: пробелы или acceptance.
- `Orchestrator` управляет циклом и предотвращает тихий drift.

### Общий план

1. Реализовать ограниченное изменение.
2. Быстро и конкретно проревьюить.
3. Вернуть только целевые корректировки, а не “переписать все”.
4. Повторять, пока блок не готов к приемке.

### Общие принципы

- Review должен быть узким, исполнимым и по возможности подкрепленным тестами.
- Rework должен быть ограниченным, а не бесконечным.
- Цикл нужен, чтобы снижать неоднозначность, а не заменять архитектуру.

### Воркфлоу

1. Архитектор определяет задачу и критерии review.
2. Исполнитель отдает код и локальную верификацию.
3. Reviewer возвращает `accepted` или `returned` с явным списком gaps.
4. Оркестратор маршрутизирует rework обратно исполнителю.
5. PM выполняет финальный block acceptance, когда цикл закрыт.

### Use cases

- UX / дизайн-heavy задачи.
- Legacy-код со скрытыми ограничениями.
- Работа, где риск это тонкие регрессии, а не отсутствие scaffolding.

### Misuse cases

- Стабильная, детерминированная реализация, которую можно сделать в один дисциплинированный проход.
- Большие архитектурные сдвиги, которые нужно решать до начала кодинга.
- Команды, которые возвращают расплывчатые narrative review вместо actionable corrections.

## 6. Contract-First / Schema-First

### Цель

Стабилизировать внешний или внутренний контракт до реализации, чтобы несколько агентов могли работать параллельно без семантического дрейфа.

### Роли агентов

- `Architect (L1)` определяет контракт и acceptance rules.
- `Executor(s)` реализуют producer/consumer/UI по замороженному контракту.
- `PM (L2)` валидирует соответствие контракту на block acceptance.

### Общий план

1. Сначала определить контракт.
2. Заморозить или версионировать его.
3. Реализовать код по контракту.
4. Валидировать соответствие и крайние случаи.

### Общие принципы

- Контракт это поверхность координации.
- Реализация вторична по отношению к совместимости.
- Изменение контракта требует явного review, а не тихого дрейфа.

### Воркфлоу

1. Архитектор публикует schema/API shape/event format/state machine.
2. Оркестратор маршрутизирует исполнителей “против” этого контракта.
3. Исполнители реализуют ограниченные ответственности.
4. PM проверяет conformance, negative cases и совместимость.

### Use cases

- Проектирование API.
- AI tool contracts.
- Межсервисные события.
- Общие data-contracts для нескольких экранов/клиентов.

### Misuse cases

- Одноразовые локальные фичи без интеграционной границы.
- Исследовательская продуктовая работа, где контракт ожидаемо часто меняется.
- Сценарии, где схема добавляет больше оверхеда, чем сама реализация.

## 7. TDD

### Цель

Сначала определить поведение исполнимыми ожиданиями (тестами) и затем реализовать код под них.

### Роли агентов

- `Architect (L1)` при необходимости задает scope и границы поведения.
- `Executor A` сначала пишет/уточняет тесты.
- `Executor B` реализует, пока тесты не станут зелеными (или один исполнитель делает оба шага).
- `PM (L2)` валидирует финальное поведение и regression coverage на block level.

### Общий план

1. Определить ожидаемое поведение в тестах.
2. Реализовать минимальный код для прохождения тестов.
3. Рефакторить, сохраняя green.

### Общие принципы

- Тесты должны фиксировать поведение, а не внутренние детали реализации.
- Red-green-refactor имеет смысл только если тесты содержательные и ограниченные.
- TDD не отменяет архитектуру; оно уточняет поведение внутри нее.

### Воркфлоу

1. Написать падающие тесты, выражающие требуемое поведение.
2. Реализовать минимально корректное решение.
3. Рефакторить для ясности и сопровождаемости.
4. Расширить покрытие на регрессии и negative cases.
5. PM принимает блок при достаточной корректности и покрытии.

### Use cases

- Багфиксы с ясным воспроизведением.
- Доменная логика с детерминированными правилами.
- Рефакторинги, где главный риск это регрессии.

### Misuse cases

- Ранний exploratory UX.
- Задачи, где архитектура еще не определена.
- Дорогие/флейковые тесты, не отражающие реальное поведение.

## 8. Изменения в skills для поддержки методик

### Назначение

Нужно, чтобы skills поддерживали более одной методики и не предполагали vertical slices по умолчанию.

### Требуемые изменения

- Skill архитектора должен явно выбирать методику разработки для каждого блока из:
  - `Vertical slices`
  - `Spec -> scaffold -> harden`
  - `Review-driven loop`
  - `Contract-first / schema-first`
  - `TDD`
- Контракт плана архитектора должен хранить:
  - `development_method`
  - `method_selection_rationale`
  - `acceptance_model`
- Skill оркестратора должен маршрутизировать выполнение согласно выбранной методике, а не предполагать per-slice PM acceptance.
- Skill исполнителя должен владеть `task_acceptance` и писать выделенный артефакт task acceptance по фиксированному шаблону.
- Skill PM должен владеть `block_acceptance`, а не per-task acceptance, если архитектор явно не описал исключение.
- Общие workflow contracts должны фиксировать immutable per-task артефакты и canonical latest артефакты.
- Скрипты оркестратора должны поддерживать:
  - генерацию prompt-файлов по режимам;
  - post-run artifact probing;
  - heartbeat и idle detection для CLI workers;
  - генерацию workflow health summary;
  - генерацию block retrospective.

### Терминология

- Нестандартные термины нужно избегать, если есть общепринятый аналог.
- Если нестандартный термин все же используется, при первом появлении нужно пояснять значение в скобках.
- Пример: `provider-aware retry policy (правила ретрая, которые по-разному реагируют на capacity limits, tool hangs, активные подпроцессы проверки и уже записанные артефакты)`.

### Acceptance model

- `task_acceptance` выполняет исполнитель как локальную верификацию и self-review.
- `block_acceptance` выполняет PM после реализации всего блока.
- PM review на этапе планирования допускается как `plan_review` и отличается от `block_acceptance`.

## 9. Spec -> Scaffold -> Harden vs Review-Driven Loop

### Ключевое различие

`Spec -> scaffold -> harden` это стадийная стратегия построения. Предполагается, что целевая форма в основном известна, и выгодно быстро собрать структурированный базис, а затем ужесточить качество.

`Review-driven loop` это итеративная стратегия коррекции. Предполагается, что основная неопределенность не в структуре, а в качестве решений, которое раскрывается при реальном review.

### Что оптимизирует каждая методика

`Spec -> scaffold -> harden` оптимизирует скорость появления структурированного прогресса с запланированным вторым проходом на строгость.

`Review-driven loop` оптимизирует безопасную сходимость, когда первый проход, вероятно, будет неполным, “визуально мимо” или ограниченным неизвестными условиями.

### Типичная последовательность

`Spec -> scaffold -> harden`

1. Зафиксировать целевую форму.
2. Построить scaffold.
3. Harden: валидация, тесты, крайние случаи, документация.

`Review-driven loop`

1. Реализовать ограниченную задачу.
2. Проревьюить.
3. Вернуть целевые корректировки.
4. Повторять до acceptance.

### Лучшие use cases

Использовать `Spec -> scaffold -> harden`, когда архитектура известна, но “полный прод-финиш” слишком замедлит первый проход.

Использовать `Review-driven loop`, когда скрытые ограничения, UX-чувствительность или legacy-поведение делают ранний review важнее, чем стадийный scaffold.

### Основной failure mode

Failure mode `Spec -> scaffold -> harden`: команда никогда не доходит до harden-фазы.

Failure mode `Review-driven loop`: бесконечная итерация низкого качества из-за расплывчатых review или нестабильных границ.
